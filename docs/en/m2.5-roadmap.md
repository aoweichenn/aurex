# M2.5 Roadmap

## Stage Position

M2.1 closed the main semantic-safety gaps, performance hot spots, diagnostic
budgets, and the current no-std language-core baseline. M2.5 does not reopen the
discarded std track or widen the language surface first. It moves the frontend
from a sequential compiler toward an engineering base that is queryable,
reusable, and consumable by IDE tooling.

The order matters:

1. Turn existing semantic results into stable, explicit, cacheable data.
2. Split parse, module graph, signatures, bodies, generics, and diagnostics into
   queries.
3. Add lossless syntax, incremental parsing, and LSP-native consumption paths.

That leaves one hard constraint in place: future query work must not preserve
metadata that depends on prose, display strings, or hidden contextual inference.

## Comparison With The Regex Expressiveness Audit

`examples/libs/regex` exposes a second fact: the M2.1 language core can write a
systems library, but it lacks enough abstraction and resource semantics to write
one cleanly. Text/bytes paths, manual storage, callbacks, iterators, buffer
writers, and cross-module private state all turn into boilerplate. This should
not be mistaken for reopening the discarded std track, and it must not displace
the M2.5 query / lossless / IDE foundation.

M2.5 therefore has two differently ranked tracks:

| Track | Original M2.5 Position | Adjustment From The Regex Audit |
|:------|:-----------------------|:---------------------------------|
| Frontend foundation | Query keys, dependency tracking, lossless syntax, and IDE-native diagnostics | This remains the main M2.5 line and the core acceptance target; std or surface-syntax expansion cannot replace it |
| Language / library expressiveness | The original route explicitly avoids widening the language surface first | Add a design-freeze and admission gate: accept only P0/P1 features that remove systems-library duplication, repair resource invariants, or support query boundaries |
| Standard library rebuild | Explicit non-goal | Still a non-goal; `Vec`, `Box`, `Result`, writers, and similar pieces are recorded as minimal language contracts and shapes, not as a resurrection of old std |
| Resources / traits / closures | Full resources, async, and macros must not be pulled into the query migration | Keep that constraint; still, resources, traits, and closures need M2.5-level semantic designs or regex-class systems libraries will stay awkward |

In short: **query/lossless/IDE is the engineering line for M2.5; regex P0/P1 is
the language-boundary line that must be frozen before the M2.6-style library
work.** If an expressiveness feature affects stable keys, query invalidation,
diagnostic protocol, or module visibility, it has to be designed under the
frontend-foundation rules before implementation.

## Why Item 1 Comes First

Diagnostics already have `DiagnosticCategory` and `DiagnosticCode`. If sema still
infers them from message prefixes or substrings, three later goals are weakened
at once:

- Rewording text changes a machine protocol, so CLI, JSON, and LSP cannot reuse
  the data safely.
- A diagnostics query can cache only strings instead of explicit semantic events.
- Localization, IDE rendering, and future fix-its all have to duplicate brittle
  classification rules.

M2.5 therefore starts by separating what a diagnostic **is** from how it is
worded: kind/category/code are structured semantics; message text is only the
presentation layer.

## Target Architecture

### A. Stable-ID-driven Queries

- Query keys use stable definition identities, file fingerprints, module
  identities, and local body identities.
- Parse, module graph, item signature, function-body typecheck, generic
  instance, match check, and diagnostics are separate queries.
- Query reuse must not depend on display names, transient node ids, or emitted
  strings.

### B. Lossless Syntax + AST Lowering

- Parsing first produces a lossless CST that preserves trivia and tokens.
- The semantic AST lowers from the CST while keeping the current typed AST /
  checked-overlay advantages.
- Syntax nodes need stable identity so local edits rebuild only affected ranges.

### C. IDE-native Diagnostics

- CLI text output, `--diagnostics=json`, and future LSP support consume the same
  diagnostic event stream.
- Diagnostics queries are keyed by semantic events, not by final strings.
- Ordering, deduplication, note/help relationships, and future fix-its remain
  structured fields.

### D. Language / Library Expressiveness Gate

M2.5 does not treat the regex library pain as a request to rebuild std, but it
does turn that pain into language and frontend constraints:

- New features must have stable semantic identity; types, instances, query keys,
  and ABI identity cannot depend on display strings, module-private convention,
  or C symbol spelling.
- Any resource feature must first define checked move/copy/drop semantics, IR
  lowering invariants, and diagnostic boundaries instead of relying on
  `defer destroy(...)` conventions.
- Any trait/protocol feature must serve static dispatch, generic body checking,
  `where` diagnostics, and query invalidation; dynamic objects and implicit
  runtime lookup are not the first step.
- Module visibility must allow internal library splits without exposing public
  ABI; this has to be designed together with stable module identity for lossless
  syntax and queries.

## Expressiveness P0

These items come from the regex library audit. They are the highest-priority
expressiveness work that M2.5 should freeze or begin. Their relationship to the
original M2.5 plan is supportive: they make future systems libraries and query
boundaries viable, but they do not replace the query/lossless line.

| P0 Item | Regex Pain | Comparison / Constraint Against Original M2.5 | M2.5 Delivery Boundary |
|:--------|:-----------|:----------------------------------------------|:-----------------------|
| Generic backend closure | `alloc_zeroed[T]` still needs an explicit `sizeof[T]`; checked can pass while full lowering/backend is not complete | The original plan already needs stable generic instance keys; this closes generic body lowering so checked and IR semantics do not diverge | `sizeof[T]` / `alignof[T]` lower through IR/LLVM inside generic functions; method-local generics get parser/sema/ABI design; minimal const generics are specified even if implemented in stages |
| Package visibility / module fragments | `vm/engine.ax` is almost 4000 lines, but splitting it forces internal helpers and fields to become `pub` | Query keys need stable module identity; this adds the missing boundary for same-module multi-file implementation and package-private visibility | Design `pub(package regex)` / `pub(crate)` or equivalent; design `module ... part ...` or equivalent fragments; internal fields remain visible to the library but outside public API |
| User traits / protocols with static dispatch | Text/bytes, callbacks, input cursors, and writers are duplicated or runtime-mode based | Original M2.5 deferred trait expansion; regex shows traits are the core way to remove systems-library duplication | First support static traits without trait objects: methods, associated types, associated consts, default methods, `where T: Trait` body checking, and instantiation diagnostics |
| Resource-semantics skeleton | `Regex`, `RegexSet`, `Captures`, and `Workspace` rely on manual destroy and early-return cleanup | Original M2.5 must not import a full borrow checker; this freezes move/copy/drop only, not lifetimes | Design move-only structs, `Copy`, `Drop`/destructor timing, partial-move restrictions, double-free diagnostics, and IR drop lowering; implementation may follow the query line, but design must be fixed in M2.5 |
| Minimal owned containers / writer shapes | Dynamic program arrays, workspaces, replacement/escape/database writers are hand-written boilerplate | M2.5 still does not rebuild std; this only records minimal no-std contracts needed by the language | Specify dependencies and ABI/Drop constraints for `Vec[T]`, `Box[T]`, `Result[T,E]`, `Option[T]`, and `BufferWriter`; implementation depends on the P0 resource/generic closure |

## Expressiveness P1

P1 items have high leverage but must not outrank P0. They are documented in M2.5
to prevent future library APIs from growing through boilerplate.

| P1 Item | Regex Pain | Comparison / Constraint Against Original M2.5 | Admission Condition |
|:--------|:-----------|:----------------------------------------------|:--------------------|
| Closures / lambdas | Replacement callbacks have to be top-level function pointers, cannot capture state, and split across text/bytes callback types | Original M2.5 avoids high-level expansion; start with non-capturing lambdas and do not pull in async/macros | Non-capturing lambdas lower to function pointers; capturing lambdas are designed as environment structs plus call methods; `Fn` / `FnMut` / `FnOnce` depend on P0 traits/resources |
| Iterator protocol | `find_next`, `captures_next`, `split_next`, and `stream_next` all hand-sync state | Current `for` is range-only; this extends only to trait-backed iterators, not generators as P0 | Design `Iterator` with associated `Item`, `next(&mut self) -> Option[Item]`, and `for item in iter` lowering; generator/yield stays later |
| `pub use` / re-export | `api.ax` contains many type aliases and thin wrappers | This does not replace the query line but must integrate with module graph/export cache | Support `pub use module::{name}` and aliases; export queries remember re-export origins for diagnostics and definition lookup |
| Named/default arguments | `replace_all`, `replace_first`, `replace_n`, and append variants cause API combinatorics | Do not introduce overload resolution that destabilizes lookup/keying; only explicit named arguments and compile-time defaults | Defaults enter function signature stable keys; call-site lowering keeps ABI deterministic; runtime overload selection is rejected |
| Safe slice / pointer-boundary helpers | Raw pointer indexing spreads `unsafe` through business logic | M2.5 already tightened unsafe boundaries; this adds a safe wrapper path after the unsafe construction point | Building a slice from raw parts remains unsafe; slice indexing/copy/fill centralizes bounds or unchecked strategy behind safe APIs |
| Derive / default impls | `RegexOptions`, result structs, and status enums need repetitive constructors and boilerplate | Only derive for fully specified capabilities; do not synthesize resource behavior implicitly | First design `Default` / `Copy` / `Eq` / `Debug`; `Drop` is not derived and stays explicit |

## Non-goals

- Rebuilding the standard library in M2.5.
- Pulling full borrow checking, async, or macros into the query migration.
- Making the full resource system a blocker for the M2.5 query line; M2.5 only
  freezes move/copy/drop semantics and the smallest implementation slices.
- Introducing trait objects, dynamic dispatch, full closure-capture ABI,
  generators, macros, or a package manager as M2.5 P0.
- Keeping a second parallel frontend just to appear incremental.
- Treating the LSP protocol adapter as the query or syntax architecture itself.

## Acceptance

1. Diagnostic metadata is no longer derived from message text; rewording does
   not change category/code.
2. Every query has explicit inputs, outputs, stable keys, dependencies, and
   invalidation boundaries.
3. Syntax preserves trivia and local edits only recompute affected parse / sema
   regions.
4. CLI, JSON, and LSP reuse one diagnostic data source with deterministic order.
5. Existing stress, coverage, and regression gates stay in place during the
   architecture migration.
6. Expressiveness P0 designs do not break query keys, module graph, diagnostic
   protocol, or checked/IR layering; `sizeof[T]` generic lowering, package
   visibility/module fragments, static traits, and move/copy/drop each have
   semantic docs, positive/negative test plans, and IR/ABI constraints.
7. Expressiveness P1 implementation starts only after the relevant P0
   constraints are verifiable; every new syntax feature must explain how it
   reduces regex-class systems-library duplication rather than only improving
   surface spelling.
