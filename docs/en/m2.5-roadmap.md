# M2.5 Roadmap

## Stage Position

M2.1 closed the main semantic-safety gaps, performance hot spots, diagnostic
budgets, and the current no-std language-core baseline. M2.5 does not reopen the
discarded standard-library track and does not widen the language surface. Its
job is to move the frontend from a sequential compiler toward an engineering
base that is queryable, reusable, and consumable by IDE tooling.

The stage boundary is now fixed:

1. Freeze query-key design and turn existing semantic results into stable,
   explicit, cacheable data.
2. Wire parse, module graph, signatures, bodies, generics, and diagnostics into
   query records and dependency tracking.
3. Finish the current lossless syntax, structured diagnostics, and IDE-native
   snapshot entry points.
4. Preserve the stable identity and query boundaries needed by M3, but do not
   implement new high-level language features in M2.5.

The hard constraint remains: future query work must not preserve metadata that
depends on prose, display strings, transient node ids, or hidden contextual
inference.

## Why Item 1 Comes First

Diagnostics already have `DiagnosticCategory` and `DiagnosticCode`. If sema still
infers them from message prefixes or substrings, three later goals are weakened
at once:

- Rewording text changes a machine protocol, so CLI, JSON, and LSP cannot reuse
  the data safely.
- A diagnostics query can cache only strings instead of explicit semantic events.
- Localization, IDE rendering, and future fix-its all have to duplicate brittle
  classification rules.

M2.5 therefore starts by separating what a diagnostic is from how it is worded:
kind/category/code are structured semantics; message text is only the
presentation layer.

## Completed Foundation: Query Keys

The first M2.5 query-key infrastructure batch is complete. The design uses
Stable Semantic Query Keys + Session Fast Handles + Red/Green Fingerprint
QueryContext:

- A query key says what to compute; a fingerprint says whether the input or
  result changed.
- `TypeHandle`, `ExprId`, `StmtId`, vector indexes, and source offsets are
  session-fast handles only. They are not persistent query keys.
- `CanonicalTypeKey`, `GenericInstanceKey`, `ModuleKey`, and `ModulePartKey` are
  prerequisites for M3 module and generic work. M2.5 does not expand them into
  trait, resource, closure, or iterator semantics.
- Diagnostics are structured query events. CLI, JSON, and future LSP output only
  render them.

See the Chinese canonical [M2.5 Query Key Design](../zh/m2.5-query-key-design.md)
for the detailed design and implementation status. Future M3 module and generic
work must reuse this path and state its stable identity, invalidation,
diagnostic, and module-identity effects before implementation.

## Target Architecture

### A. Stable-ID-driven Queries

- Query keys use stable definition identities, file fingerprints, module
  identities, and local body identities.
- Parse, module graph, item signature, function-body typecheck, generic
  instance, match check, and diagnostics are separate queries.
- Query reuse must not depend on display names, transient node ids, or emitted
  strings.

### B. Lossless Syntax + AST Lowering

- Parsing first produces a lossless CST that preserves trivia and tokens.
- The semantic AST can lower from the lossless CST facade into the existing
  parser while keeping the current typed AST / checked-overlay advantages.
- Syntax nodes now have stable identity from kind/range/token span/depth; the
  IDE snapshot uses that identity for offset queries and edit-impact node
  selection, while full local subtree reparse remains a follow-up optimization.

### C. IDE-native Diagnostics

- CLI text output, `--diagnostics=json`, and future LSP support consume the same
  diagnostic event stream.
- Diagnostics queries are keyed by semantic events, not by final strings.
- Ordering, deduplication, note/help relationships, and future fix-its remain
  structured fields.

### D. M3 Handoff

The handoff from M2.5 to M3 is not "start all advanced features." It is:

- Upgrade the module system from file-level concatenation to logical modules and
  module parts.
- Upgrade generics from the current monomorphization path to query-backed
  template signatures, instance signatures, and instance bodies.
- Keep generic ABI, generic instance identity, and module identity stable without
  relying on display strings or session-only handles.

See the [M3 Roadmap](m3-roadmap.md) for the next stage.

## Deferred Features

The following features are no longer part of M2.5 acceptance and do not block the
M3 module/generic stage:

- User traits / protocols, associated types, associated consts, and trait
  objects.
- `Copy` / `Drop`, move-only structs, destructor lowering, borrow checking,
  lifetimes, and RAII.
- Closure / lambda capture, `Fn` / `FnMut` / `FnOnce`, generators, and async.
- General iterator protocols, `for item in container`, derive/default impls, and
  complex named/default arguments.
- Standard-library rebuilds, package managers, version solving, and dependency
  solving.

These areas may keep research notes and reserved query-key tags, but they need
fresh feature designs before implementation and must build on the module and
generic boundaries stabilized after M3.

## Non-goals

- Rebuilding the standard library in M2.5.
- Pulling full borrow checking, resource/drop, traits, closures, iterators,
  async, or macros into the query migration.
- Treating M3 module or generic work as leftover M2.5 acceptance.
- Keeping a second parallel frontend just to appear incremental.
- Treating the LSP protocol adapter as the query or syntax architecture itself.

## Acceptance

1. Diagnostic metadata is no longer derived from message text; rewording does
   not change category/code.
2. Every query has explicit inputs, outputs, stable keys, dependencies, and
   invalidation boundaries.
3. Syntax preserves trivia and local edits only recompute affected parse / sema
   regions.
4. CLI, JSON, and LSP reuse one diagnostic data source with deterministic order.
5. Existing stress, coverage, and regression gates stay in place during the
   architecture migration.
6. The M3 entry point is documented clearly: the next stage is modules and
   generics completion only; RAII, traits, closures, and iterators no longer
   hang off the M2.5 acceptance tail.
