# Aurex M3.1 Generics Completion Plan

## Purpose

This document is the execution entry point for M3.1 generics work. Future work
should read this file plus the active work package instead of rebuilding the
entire M2 / M2.5 / M3.0 context every time.

The [M3 Roadmap](m3-roadmap.md) explains why M3.1 exists. This document explains
how to execute it: what to read, what invariants must hold, which files are in
scope, and what proves completion.

## Baseline

- Branch: `m3.1`
- Baseline commit: `af4ed87 Start M3.1 generic ABI stabilization`
- M3.0 module-system closure is complete.
- M3.1 is the current highest priority.
- Already started: generic struct / enum / function ABI suffixes now derive
  from `GenericInstanceKey` / stable fingerprints instead of session-only
  `TypeHandle.value`.
- Current closure state: method-local generics are implemented, and the WP-7
  audit confirms that ABI names, stable ids, query subjects, checked metadata,
  IR/LLVM/native paths use `GenericInstanceKey` / `GenericInstanceIdentity` as
  the stable identity boundary. `TypeHandle.value` remains only a session-local
  cache / lookup fast key.
- Remaining risk: future query-authority deepening and later traits, resource
  semantics, or const generics must not bypass the current generic identity,
  body, side-table, and lowering closure.

## Goals

M3.1 does not widen the language surface. It turns the existing generic
implementation into a stable compiler subsystem:

1. Generic instance identity derives from `GenericInstanceKey` / canonical type
   identity.
2. Generic ABI naming is stable across compiler sessions.
3. `GenericTemplateSignature`, `GenericInstanceSignature`, and
   `GenericInstanceBody` become the query-backed authority boundaries.
4. Retained typed bodies, generic side tables, IR lowering, LLVM lowering, and
   native execution consume the same instance-body view.
5. Generic function builtin type operands work through sema, IR, and LLVM:
   `sizeof[T]` / `alignof[T]`, `ptrat[*T]`, `ptrcast[*T]`, and `bitcast[*T]`
   consume the same checked side-table facts.
6. Method-local generics move from unsupported into implementation after
   identity, ABI, and query boundaries are stable.
7. Tests, coverage, generic stress, query pruning, and native execution do not
   regress.

## Non-goals

- User traits, protocols, associated types, associated consts, trait objects, or
  dynamic dispatch.
- Const generics.
- RAII, `Drop`, `Copy`, move-only structs, borrow checking, lifetimes, or
  automatic cleanup.
- Closure capture, `Fn` / `FnMut` / `FnOnce`, generators, async, or iterator
  protocols.
- Standard library rebuilds, package management, version solving, or dependency
  resolution.
- A new generic syntax family. M3.1 keeps `Name[T]`, `fn id[T](value: T)`, and
  built-in capability `where` clauses.

## Global Invariants

- `TypeHandle.value` is a session-local fast handle. It must not enter stable
  ABI, stable ids, incremental keys, query keys, or cross-session observable
  behavior.
- Display strings, diagnostic text, checked-dump text, and C ABI names are
  outputs. They are not semantic identity inputs.
- Generic instance identity comes from `GenericInstanceKey`; type arguments come
  from `CanonicalTypeKey`; constraint environments come from stable
  `ParamEnvKey`.
- `GenericInstanceIdentity` is the sema bridge between stable keys, fingerprint
  text, stable definition ids, and ABI suffixes.
- `GenericTemplateSignature`, `GenericInstanceSignature`, and
  `GenericInstanceBody` must become query provider authorities. Eager sema may
  materialize or consume query results, but it should not remain the only source
  of truth.
- Checked metadata must carry enough structured identity for dumps, lookup,
  incremental cache, IR lowering, and tooling to avoid rebuilding generic
  identity independently.
- IR and LLVM lowering consume checked sema / generic instance body / side table
  results. They must not recover generic semantics from raw AST guesses.
- Method-local generic parameters must be distinct from owner generic
  parameters in scope, lookup, inference, ABI, and query keys.
- Resource capabilities, user traits, and const generics do not enter M3.1
  through side paths.

## WP-1A: Generic ABI Stabilization

Completed first slice:

- `generic_instance_abi_suffix` takes `query::GenericInstanceKey`.
- Generic struct / enum / function instantiate paths compute
  `GenericInstanceIdentity` before forming ABI suffixes.
- White-box tests cover two sessions with different `TypeHandle.value`
  assignments but equal canonical instance keys.

Closure audit result:

- Owner-generic methods and method-local generic methods build
  `GenericInstanceIdentity` before deriving stable ids, ABI suffixes, and
  checked signature identity. Method instances use `GenericInstanceKey` with
  `DefNamespace::member`; type arguments are ordered as `owner_type + ordered
  generic args`.
- Generic type aliases remain transparent aliases and do not create independent
  runtime ABI types. `GenericTypeAliasInstanceInfo` still records
  `GenericInstanceKey`, the resolved target type, the stable id, and a
  target-sensitive instance-signature incremental key, so query/cache-visible
  identity is structured.
- `generic_instance_key_suffix` / `append_generic_instance_key_suffix` still use
  `TypeHandle.value`, but only as session-local lookup/cache keys. White-box
  coverage explicitly checks that this suffix can differ across sessions while
  `GenericInstanceKey`, stable fingerprints, and ABI suffixes stay equal.
- Checked dumps, IR dumps, and diagnostics use display strings / c_names only as
  output text. Incremental-cache generic signature/body rows collect
  `GenericInstanceKey` and authority structures from checked metadata instead
  of deriving identity from display/c_name text.
- LLVM symbols and native artifacts consume checked `c_name` values written by
  sema. The generic suffix embedded in those names already derives from the
  stable `GenericInstanceKey`.

Completion requires focused gtests, full ctest, coverage, and explicit notes for
struct / enum / function / owner-generic method / method-local generic method /
type alias identity paths.

## WP-1B: Generic Instance Identity Propagation

Goal: every generic item category carries one structured identity through
checked metadata.

Current state:

- `FunctionSignature` stores `GenericInstanceKey`; generic function and
  owner-generic method instantiation write the same identity in retained and
  non-retained side-table paths.
- `EnumCaseInfo` stores `GenericInstanceKey`; lightweight
  `GenericEnumInstanceInfo` records the query-facing identity of generic enum
  instances without introducing a full `EnumInfo` model for ordinary enums.
- Generic type alias instances are no longer only session-local resolved cache
  entries. `GenericTypeAliasInstanceInfo` records `GenericInstanceKey`,
  resolved target type, stable id, and a target-type-sensitive
  instance-signature incremental key. Type aliases remain transparent aliases
  and do not create a distinct runtime ABI type.
- `CheckedModule` clone / copy / move preserves function, enum-case, generic
  enum instance, and generic type-alias instance identity.
- Incremental-cache generic instance signature subjects are collected from
  structured checked metadata, deduplicated by `GenericInstanceKey`, and invalid
  key / incremental-key pairs are skipped.
- White-box coverage checks function signature identity, the discard-side-table
  path, generic enum case identity, generic type alias instance identity,
  checked module copy/move preservation, and driver generic instance cache rows.

- Does enum/type alias/function/method metadata need `GenericInstanceKey` fields

- Does `FunctionSignature` need to store a `GenericInstanceKey`, or are stable
  ids plus generic args enough?
- Should generic enum identity live on existing `TypeInfo` / `EnumCaseInfo`, or
  does the model need a dedicated enum metadata record?
- Should type alias instances be query-visible identities or purely resolved
  target-type aliases?
- Should checked dumps and lowerers consume stored generic instance identity?

## WP-2: Generic Query Authority

Goal: make generic query records provider authorities:

- `GenericTemplateSignature` owns template params, constraints, visibility,
  module/package/part identity.
- `GenericInstanceSignature` owns return types, parameter types, method flags,
  unsafe/variadic state, capability environment, and signature fingerprint.
- `GenericInstanceBody` owns retained typed body, side-table layout, body
  fingerprint, and lowering inputs.

Read:

- generic query headers and sources under `include/aurex/infrastructure/query/` and `src/query/`
- `src/driver/incremental_cache/subjects/semantic.cpp`
- `src/driver/incremental_cache/subjects/ordering.cpp`
- `src/sema/internal/sema_generic_analyzer.cpp`
- `tests/gtest/query/query_key_tests.cpp`

Completion requires provider input/output tests, record fingerprint tests, edge
verifier coverage, replay/pruning coverage, and sema paths that clearly consume
or materialize provider-authoritative results.

Current state:

- The query layer now has `GenericTemplateSignatureAuthority`,
  `GenericInstanceSignatureAuthority`, and `GenericInstanceBodyAuthority`.
  Provider inputs no longer carry only a bare `IncrementalKey` / body
  fingerprint.
- `GenericTemplateSignatureAuthority` records the signature incremental key,
  `ModulePartKey`, namespace, visibility rank, template parameter count, and
  constraint count. `GenericTemplateSignatureInfo` now also carries
  `constraint_count` so checked metadata can feed the provider without
  re-reading syntax.
- `GenericInstanceSignatureAuthority` records instance kind, visibility,
  type/const argument counts, param-env predicate count, value parameter count,
  generic argument count, return/receiver/unsafe/variadic/definition flags, and
  the signature incremental key. Provider results are derived from these
  structured fields instead of the bare incremental key.
- Generic struct / enum instance signature fingerprints are now sensitive to
  the resolved shape. Struct instances mix field stable keys, field visibility,
  and canonical field types; enum instances mix the underlying/payload layout,
  case stable keys, discriminants, and canonical payload types. This prevents
  cache reuse from treating field or payload changes under the same
  `GenericInstanceKey` as unchanged.
- `GenericInstanceBodyAuthority` records checked-body fingerprint, instance
  signature result, generic side-table span/count data, sparse fallback count,
  and retained/local-dense/sparse flags. Body query results are sensitive to the
  signature result, checked body, and side-table layout.
- `src/driver/incremental_cache/subjects/semantic.cpp` builds authorities from
  checked metadata and module records. `subjects/ordering.cpp` uses the same
  query helpers to reproduce record results, so provider execution,
  provider-skip replay, and pruning share one fingerprint rule.
- Query tests cover authority valid/invalid paths, template constraint changes,
  instance method/receiver changes, body side-table count changes, dependency
  edges, fallback behavior, and cycles for generic signature/body queries.
- Sema whitebox tests cover generic struct field-shape changes and generic enum
  payload-shape changes: the signature fingerprint must change even when the
  `GenericInstanceKey` stays the same.
- Driver/incremental-cache tests cover generic instance signature/body rows,
  query-pruning reuse, and malformed graph/identity repair paths, proving the
  cache fast path can explain the new generic query records.

## WP-4: Generic Builtin Operand Closure

Goal: support generic function builtin operands through sema, IR, LLVM, and
diagnostics while keeping sema as the authority for substituted types.

Read:

- `src/sema/internal/sema_builtin_expression_analyzer.cpp`
- `src/sema/internal/sema_type_services.cpp`
- `src/sema/internal/sema_generic_analyzer.cpp`
- `src/ir/lower_ast_expr.cpp`
- `src/ir/ir_dump.cpp`
- `src/backend/llvm/emission/llvm_backend_value.cpp`

Decisions:

- Generic params do not implicitly satisfy `Sized`; `sizeof[T]` /
  `alignof[T]` require `where T: Sized`.
- Void, opaque, unsized-like, and invalid operands keep using the sema builtin
  checker's structured diagnostics.
- Builtin type operands belong to body semantics, not
  `GenericInstanceSignature`; the body fingerprint and retained side tables are
  the lowering authority.
- Top-level `cast[T](value)` remains governed by the existing scalar-cast rule.
  M3.1 does not add a new `Scalar` / `Cast` capability for arbitrary `T`.

Current state:

- Generic function and method instance-body `GenericAnalysisScope` now caches
  syntax type handles in retained and non-retained side-table paths.
- `sizeof[T]` / `alignof[T]` lower after instantiation with sema-substituted
  concrete targets such as `sizeof i32` / `alignof i32`.
- `ptrat[*const T]`, `ptrcast[*const T]`, and `bitcast[*const T]` close through
  the same retained `syntax_type_handles`.
- `ptraddr`, `sliceptr`, `slicelen`, `strptr`, `strblen`, `strvalid`,
  `strfromutf8`, and `strraw` are covered in retained generic bodies through
  expression side tables.

Completion requires positive and negative samples, IR dump assertions, and
native execution coverage.

## WP-5: Method-local Generics

Goal: implement method-local generic parameters as a real language capability.

Current implementation:

- Parser-owned ordering stays stable: owner params from `impl[T] Owner[T]` are
  prepended before method-local params in the method item's `generic_params`.
- Sema registration accepts method-local generic params. Owner params are the
  `GenericParamIdentity` values referenced by `impl_type_pattern`; remaining
  params are method-local.
- Explicit `value.method[T](arg)` / `Type.method[T](arg)` type args bind only
  method-local params. Owner params are inferred from the receiver or associated
  owner type.
- Normal method calls infer method-local params from call arguments only when a
  method actually has local params. Owner-only generic methods keep the owner
  type inference path, so receiver / argument coercions are not mistaken for
  generic unification failures.
- Generic method instances use `GenericInstanceKey` with
  `DefNamespace::member`; identity args are `owner_type + ordered generic args`.
  Generated semantic keys and ABI suffixes distinguish repeated instances such
  as `method[i32]` and `method[bool]` on the same owner.
- Instantiated generic methods are not indexed into ordinary method-name
  lookup; call resolution returns to the template bucket and reuses or creates
  the exact instance for the current type args.
- IR lowering recognizes explicit generic method callees shaped as
  `generic_apply(field(...))` and still inserts the receiver argument.
- Diagnostics cover method-local arity, inference failure, unsatisfied
  capability constraints, and explicit type args on non-generic methods.

Design before implementation:

- Scope order between owner generic params and method-local params.
- Explicit type-argument syntax for method calls.
- Method lookup interaction with owner type, method name, explicit args, and
  inferred args.
- `GenericParamIdentity` encoding for method-local params.
- ABI suffix composition from owner instance identity and method-local args.
- `GenericInstanceKey` namespace and type-arg ordering.
- Diagnostics for owner constraint failure, method-local constraint failure, and
  inference failure.

Read:

- `src/sema/internal/sema_generic_analyzer.cpp`
- `src/sema/sema_call.cpp`
- `src/sema/sema_lookup.cpp`
- `src/sema/internal/sema_lookup_resolver.cpp`
- `src/sema/internal/sema_lookup_indexer.cpp`
- `include/aurex/frontend/sema/checked_module.hpp`
- `src/sema/generic_instance_identity.cpp`
- `tests/gtest/integration/regression_tests.cpp`

Completion coverage:

- Positive samples cover owner-generic methods, method-local-only methods, and
  mixed owner + method-local methods.
- Negative samples cover inference failure, arity errors, unsatisfied
  constraints, and explicit type args on a non-generic method. Private /
  ambiguous method coverage continues through the existing method lookup tests.
- Checked dumps, LLVM IR positive samples, and native execution are covered.
  `GenericInstanceKey` continues through the existing generic instance identity
  and query-subject path; finer provider-level tests remain a follow-up for the
  next query-authority deepening pass.

## WP-7: Generic Closure Audit And Release Baseline

Goal: close M3.1 generics as an engineering baseline without widening the
language surface:

- Audit every generic identity entry point and confirm that stable identity
  derives only from `GenericInstanceKey` / `GenericInstanceIdentity`.
- Audit ABI names, stable ids, incremental keys, query subjects, checked dumps,
  IR/LLVM/native output for any remaining path where `TypeHandle.value`, display
  strings, or c_names are used as reverse identity inputs.
- Record the M3.1 completion / release baseline so later work packages do not
  reinterpret the accepted boundary.
- Add a formal sample for mixed owner-generic and method-local generic method
  instances.

Audit matrix:

| Surface | Current authority | Audit result |
|:--|:--|:--|
| Generic struct | `StructInfo::generic_instance_key` | ABI suffixes, stable ids, and signature fingerprints derive from `GenericInstanceIdentity` |
| Generic enum | `GenericEnumInstanceInfo::generic_instance_key` + `EnumCaseInfo::generic_instance_key` | Enum instances and cases store one structured key; payload shape affects the signature fingerprint |
| Generic type alias | `GenericTypeAliasInstanceInfo::generic_instance_key` | Transparent aliases do not create runtime ABI types, but query/cache-visible identity and resolved target type are recorded |
| Generic function | `GenericFunctionInstanceInfo::generic_instance_key` + `FunctionSignature::generic_instance_key` | Retained and non-retained paths write the same key into checked function signatures |
| Owner-generic method | `FunctionSignature::generic_instance_key` with `DefNamespace::member` | Owner type participates in method identity args; ABI suffixes derive from stable keys, not owner display names |
| Method-local generic method | `GenericInstanceKey` for `owner_type + ordered generic args` | Explicit/inferred method-local args are separate from owner args; lookup returns to the template bucket and does not pollute ordinary method lookup |
| Session-local lookup cache | `generic_instance_key_suffix(args)` | May use `TypeHandle.value` as a session-local fast key only; it must not enter ABI/query/stable identity |
| Checked dumps / diagnostics | Display strings / c_names as output | Presentation only; never an identity input |
| Incremental cache | Generic signature/body/lower IR query subjects | Collects `GenericInstanceKey` from checked metadata and authority structures, deduplicates by key, and skips invalid keys |
| IR / LLVM / native | Checked `c_name` + retained body view | Lowering consumes `GenericFunctionInstanceBodyView` and sema side tables; LLVM/native consume stable IR symbols |

Release baseline:

- Stable identity / ABI / checked metadata / query subjects are supported for
  generic structs, enums, type aliases, functions, owner-generic methods, and
  method-local generic methods.
- Generic function/method bodies, retained side tables, IR lowering, LLVM
  lowering, and native execution consume the same body view.
- `sizeof[T]` / `alignof[T]`, unsafe pointer type operands, and slice/string
  value-only builtins close inside generic bodies.
- Method-local generic explicit type args, argument inference, arity /
  inference / `where` / non-generic misuse diagnostics are supported.
- User traits, associated types, const generics, resource capabilities, RAII,
  closures, async/generators/iterators, and standard-library rebuilds remain out
  of M3.1.

New coverage:

- `tests/samples/positive/generics/method_local_identity_closure_m3_1.ax`
  covers owner-only methods, method-local methods, and identical method-local
  type args across different owner instances.
- `SampleSuite_PositiveRuntime_method_local_identity_closure_generics` runs that
  sample as a formal runtime test. `SampleSuite_PositiveSamples` continues to
  compile every positive sample through LLVM IR emission.
