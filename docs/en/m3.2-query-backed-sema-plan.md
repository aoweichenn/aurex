# Aurex M3.2 Query-backed Sema Design And Execution Plan

## Purpose

This document is the M3.2 design entry point. M3.0 closed the module system,
and M3.1 closed generic identity / ABI / body lowering. M3.2 does not rush into
traits, RAII, const generics, or standard-library rebuilds. It moves semantic
analysis toward a modern query-backed compiler architecture.

M3.2 keywords: stable identity, query authority, reusable checked facts, explicit
invalidation, IDE/LSP consumption.

## Baseline

- Branch: `m3.2`
- Parent baseline: `m3` / `m3.1` at `59a2ddf Complete M3.1 generic closure audit`
- Stable foundations:
  - R5 driver/session/pipeline/stage metadata.
  - M3.0 `ModuleKey` / `ModulePartKey` / module graph/export boundaries.
  - M3.1 `GenericInstanceKey` / `GenericInstanceIdentity` / generic signature/body authority.
  - `aurex_tooling::IdeSnapshot`, lossless syntax, structured diagnostics, and query graph persistence.

## Goals

M3.2 does not add language syntax. It turns sema from "one eager analyzer
produces a checked module" into "query-backed semantic authority":

1. `ItemSignature`, `GenericTemplateSignature`, `GenericInstanceSignature`,
   `BodySyntax`, `TypeCheckBody`, and `GenericInstanceBody` have unified query
   authority entry points.
2. Eager sema may materialize query results, but it should not remain the only
   source of truth.
3. Checked metadata clearly separates durable facts, session-local caches, and
   lowering-only side tables.
4. Incremental cache / query pruning / provider-skip replay can explain semantic
   result reuse, not only file-level reuse.
5. IDE/tooling can consume query-backed semantic facts, so a later LSP adapter
   does not bypass parser/sema/query.
6. The M3.1 generic identity, ABI, IR/LLVM/native baseline does not regress.

## Non-goals

- User traits, protocols, associated types, associated consts, trait objects, or
  dynamic dispatch.
- Const generics.
- RAII, `Drop`, `Copy`, move-only structs, borrow checking, lifetimes, or
  automatic resource cleanup.
- Closures, generators, async, or iterator protocols.
- Standard-library rebuilds, package management, or dependency resolution.
- Broad parser/IR/backend rewrites. M3.2 changes sema/query/tooling boundaries.

## Compiler References And Adaptation

M3.2 adopts mature compiler ideas in a form that fits Aurex's current size:

- **rustc query model**: stable keys, providers, dependency edges, and red-green
  reuse. M3.2 does not introduce a full HIR/THIR/MIR split or trait solver.
- **Salsa-style incremental**: input queries, derived queries, result
  fingerprints, and dependency graphs. M3.2 stays single-threaded first to keep
  determinism and invalidation easy to verify.
- **Clang Sema / ASTContext**: keep arena/interner performance and centralized
  checked-fact ownership, but avoid turning all behavior into global mutable
  context or overly fine virtual interfaces.
- **LLVM pass/analysis manager**: every sema query provider should declare its
  input authority, output facts, and invalidation conditions.

## Global Invariants

- Stable query keys come only from `ModuleKey`, `ModulePartKey`, `StableDefId`,
  `StableMemberKey`, `GenericInstanceKey`, `BodyKey`, and canonical type identity.
- `TypeHandle.value`, `IdentId.value`, vector indexes, display names,
  diagnostic text, checked dumps, and c_names must not become cross-session
  query identity.
- `CheckedModule` remains the aggregate output for one compilation, but M3.2
  must be able to point every durable checked fact back to a query authority.
- Sema providers must not recover facts from raw AST guesses when an upstream
  query already owns them. AST access must go through `ModulePartKey` and
  item/body source ranges.
- IR lowering consumes checked sema / retained bodies / side tables. It does not
  participate in sema query computation.
- Diagnostic category/code/message layering remains unchanged. Query-backed
  diagnostics must emit the same structured protocol.
- Every work package preserves the M3.1 generic stress, query pruning, coverage,
  and native execution baseline.

## WP-1: Sema Query Authority Inventory

Goal: do the structural inventory first, without changing production behavior:

- List durable facts, session-local caches, and lowering-only side tables in
  `CheckedModule`.
- Assign each durable fact to a query owner: item signature, generic template
  signature, generic instance signature, body syntax, type-check body, generic
  instance body, or diagnostics.
- Identify each provider's stable key, input authority, output fingerprint,
  dependency edges, and invalidation conditions.

Completion requires an authority matrix in both design documents, no identity
gaps in existing query records / checked metadata, and at least `git diff --check`
for documentation-only work.

Completed authority matrix:

| Checked fact / cache | Storage class | Stable key | Provider authority | Result fingerprint | Dependency and invalidation boundary |
|:--|:--|:--|:--|:--|:--|
| file content | durable input fact | `FileKey` | `FileContentProviderInput` source fingerprint | `QueryResultFingerprint` | Invalidated by source path/package/role or content changes |
| lex file | durable derived fact | `LexFileKey` | file-content result + lex config | `QueryResultFingerprint` | Depends on `file_content`; invalidated by content/config changes |
| parse file | durable derived fact | `ParseFileKey` | lex result + parser config | `QueryResultFingerprint` | Depends on `lex_file`; invalidated by token/config changes |
| module part | durable module syntax fact | `ModulePartKey` | parse result + part topology | `QueryResultFingerprint` | Depends on `parse_file`; invalidated by part path/name/index/module changes |
| module graph | durable module fact | `ModuleKey` | import/package/topology graph fingerprint | `QueryResultFingerprint` | Invalidated by import target package, source-root topology, or import surface changes |
| item list | durable module fact | `ModuleKey` | visible item identity list | `QueryResultFingerprint` | Depends on `module_graph`; invalidated by item identity set changes |
| module exports | durable module fact | `ModuleKey` | public/package export signature | `QueryResultFingerprint` | Depends on `item_list` and re-exported `module_exports` |
| item signature | durable sema fact | `DefKey` | `ItemSignatureAuthority` | `item_signature_result_fingerprint` | Depends on `module_exports`; invalidated by definition kind, visibility, module part, shape, or signature key changes |
| function body syntax | durable sema body fact | `BodyKey` | `FunctionBodySyntaxAuthority` | `function_body_syntax_result_fingerprint` | Invalidated by body owner/slot/ordinal, module part, or syntax fingerprint changes; absolute source range is provenance/validation only, so offset shifts outside the body do not invalidate it |
| type-check body | durable sema body fact | `BodyKey` | `TypeCheckBodyAuthority` | `type_check_body_result_fingerprint` | Depends on `function_body_syntax` and `item_signature`; invalidated by checked body, signature result, or side-table summary changes |
| generic template signature | durable generic sema fact | `DefKey` | `GenericTemplateSignatureAuthority` | `generic_template_signature_result_fingerprint` | Depends on `item_list`; invalidated by template namespace, visibility, parameters, or constraints |
| generic instance signature | durable generic sema fact | `GenericInstanceKey` | `GenericInstanceSignatureAuthority` | `generic_instance_signature_result_fingerprint` | Depends on template signature; invalidated by type/const args, parameter environment, shape, or signature key changes |
| generic instance body | durable generic body fact | `GenericInstanceKey` | `GenericInstanceBodyAuthority` | `generic_instance_body_result_fingerprint` | Depends on generic instance signature; invalidated by retained body or side-table layout/fallback changes |
| lower function IR | lowering-only derived fact | `BodyKey` / `GenericInstanceKey` | lowerer output fingerprint | `QueryResultFingerprint` | Depends on type-check body or generic instance body; does not feed sema facts backward |
| diagnostics | durable reporting fact | producer `QueryKey` | `DiagnosticsProviderInput` | `QueryResultFingerprint` | Depends on the producer query; message text is not the source of sema identity |

`CheckedModule` durable metadata now provides materialization inputs through
stable ids, incremental keys, module ids, part indices, retained body ids, and
generic side-table summaries. Session-local lookup/cache fast keys only serve
the current analysis run. Expression, pattern, type, and statement side tables
are explicitly type-check/lowering consumption surfaces, not cross-session query
identity.

## WP-2: Item And Body Provider Boundary

Goal: bring non-generic item and function-body queries up to the level of the
M3.1 generic queries:

- `ItemSignatureAuthority` records visibility, module part, definition kind,
  and parameter/field/payload shape.
- `BodySyntaxAuthority` records body source ranges, syntax fingerprints, and
  owner def keys.
- `TypeCheckBodyAuthority` records signature results, diagnostics results,
  checked-body fingerprints, and side-table layout summaries.

Forbidden shortcuts:

- Do not infer body identity from display strings or c_names.
- Do not make the lowerer compute type-check query facts.
- Do not use diagnostic message text as the only result-fingerprint input.

Completed provider boundary:

- Added `ItemSignatureAuthority`, carrying signature incremental key,
  `ModulePartKey`, namespace, `DefKind`, visibility rank, value/generic
  parameter counts, return/receiver/unsafe/variadic flags, and definition flag.
- Added `FunctionBodySyntaxAuthority`, carrying syntax fingerprint, owner
  `DefKey`, `ModulePartKey`, body source range, body slot, and ordinal.
- Added `TypeCheckBodyAuthority`, carrying checked body fingerprint, body
  syntax result, item signature result, expr/pattern/type/stmt side-table
  counts, coercion count, retained-side-table flag, and diagnostics flag.
- Non-generic item/body provider inputs no longer accept a bare
  `IncrementalKey` or bare body fingerprint. Provider outputs and incremental
  cache `query_record_for_subject` use the same authority result helpers.
- `FunctionBodySyntaxAuthority` retains the absolute body source range for
  provenance and range-order validation. The result fingerprint does not mix
  absolute offsets, so comments or trivia inserted outside a body do not force
  all body queries to recompute.
- No new query kind was added. The existing graph remains
  `item_signature -> module_exports`,
  `type_check_body -> function_body_syntax`, and
  `type_check_body -> item_signature`; generic authority boundaries keep the
  M3.1 shape.

## WP-3: Checked Fact Materialization

Goal: keep eager sema compiling complete programs while marking durable checked
facts as materialized from query authorities:

- `CheckedModule` remains the aggregate result and gains or reuses provenance metadata.
- Incremental-cache subjects collect directly from authority results instead of
  rebuilding approximate fingerprints.
- Provider-skip replay can explain item, body, and generic semantic results.

Completed materialization path:

- `src/driver/incremental_cache/subjects/semantic.cpp` constructs authorities
  from `CheckedModule` stable ids, incremental keys, module ids, part indices,
  function-body AST ranges, retained generic body views, and side-table
  summaries.
- `src/driver/incremental_cache/subjects/ordering.cpp` builds current query
  records through the same authority result helpers that provider defaults use,
  avoiding divergence between provider replay and cache writing.
- Provider-skip replay seed/evaluate can explain item signature, function body
  syntax, type-check body, generic template signature, generic instance
  signature, and generic instance body through the same `QueryRecord` /
  dependency edge model.
- `CheckedModule` does not duplicate query records yet. It reuses stable source
  metadata as materialization input, avoiding two fact stores before the WP-4
  service split. Durable cross-session facts live in query records/cache;
  side tables needed by the current lowering run remain in the checked
  aggregate.

## WP-4: Sema Service Boundary Split

Goal: reduce `SemanticAnalyzerCore` coupling without rewriting algorithms:

- `LookupService`: module/import/re-export/type/value/method lookup.
- `TypeService`: canonical types, storage/layout eligibility, coercion.
- `GenericService`: template/instance identity, constraints/capabilities, instance bodies.
- `BodyCheckService`: function bodies, expressions, statements, and patterns.

Services receive explicit context and authority inputs. They should not
implicitly own the entire analyzer state.

Completed service boundary:

- Added `src/sema/internal/sema_services.hpp/.cpp`, collecting today's eager
  sema entry points behind four internal facades:
  `SemanticLookupService`, `SemanticTypeService`, `SemanticGenericService`, and
  `SemanticBodyCheckService`.
- Each service exposes `SemanticServiceBoundary`, declaring its domain, service
  name, consumable authority query kinds, and `owns_analyzer_state == false`.
  This stage delegates to the existing analyzers/resolvers instead of rewriting
  algorithms or copying analyzer state.
- `SemanticAnalysisPipeline::run_function_body_phases()` now reaches generic
  function definitions through `GenericService` and ordinary function bodies
  through `BodyCheckService`; layout validation enters the ABI checker through
  `TypeService`.
- White-box coverage fixes the service boundaries, authority-kind maps,
  explicit `SemaContext` exposure, and generic/body-check routing, so later work
  has a concrete boundary instead of adding more direct `SemanticAnalyzerCore`
  aggregation.

## WP-5: Tooling Semantic Query Surface

Goal: make `IdeSnapshot` consume M3.2 query-backed facts:

- Hover / definition / references can point to stable def/member/generic
  instance keys.
- Diagnostic owner stages and query dependency edges can be returned together.
- Reserve semantic fact pagination / filtering for a later LSP adapter without
  implementing full LSP in M3.2.

Completed tooling semantic surface:

- `IdeQuerySnapshot` now contains `semantic_facts`. `IdeSemanticFact` carries
  fact kind, owner `QueryKey`, stable `DefKey` / `MemberKey` / `BodyKey` /
  `GenericInstanceKey`, source range, part index, and checked flag.
- `IdeDefinition` continues to expose the stable `DefKey` and now reserves
  member-key / generic-instance-key fields, so a later LSP adapter does not
  need to bypass the tooling snapshot to inspect checked internals.
- After lossless parse, AST lowering, and checked sema, `build_ide_snapshot(...)`
  materializes the IDE semantic query surface. `module_graph`, `module_part`,
  `item_list`, `module_exports`, `item_signature`,
  `generic_template_signature`, `function_body_syntax`, and `type_check_body`
  records enter the same `snapshot.query.records` vector.
- The same snapshot now returns semantic dependency edges:
  `module_part -> parse_file`, `item_list -> module_graph`,
  `module_exports -> item_list`, `item_signature -> module_exports`,
  `type_check_body -> function_body_syntax`, and
  `type_check_body -> item_signature`.
- Tooling semantic module identity now matches sema stable module identity,
  avoiding a split between IDE `ModuleKey` values and checked stable def keys.
- This stage only provides the in-memory semantic fact surface. LSP pagination
  and filtering remain reserved for a later protocol adapter.

## WP-6: Incremental Reuse And Quality Gates

WP-4/5/6 closure acceptance gate:

```sh
tools/format_check.py
git diff --check
cmake --build --preset full-llvm -j 8
ctest --test-dir build/full-llvm --output-on-failure
tools/check_coverage.sh
tools/query_pruning_gate.py --skip-build
make perf-stress-threshold
```

When changing query graphs or invalidation, also run:

```sh
tools/query_graph_fuzz.py --skip-build --rounds 4 --functions 8 --function-step 8
```

When changing generic providers or side tables, also run:

```sh
AUREX_STRESS_BUILD_DIR=build/full-llvm tools/generic_stress.py --skip-build --counts 200,500,1000,2000
```

WP-6 means all existing tests must continue to pass, and query pruning, query
graph fuzzing, coverage, stress, and native execution must not regress. Future
M3.2 follow-up work keeps this command set as the default quality gate.

## Execution Protocol

When asked to continue an M3.2 work package:

1. Read this document's goals, invariants, and active work package.
2. Check `git status --short --branch` and confirm branch `m3.2`.
3. Read only the package's source entries plus direct callers/callees.
4. Implement or update docs.
5. Run the package's quality gate.
6. Update this document's status.
7. Commit and push.
