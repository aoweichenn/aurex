# Aurex M3.3 Tooling Session And Incremental Sema Plan

## Document Role

M3.3 starts after M3.2 Query-backed Sema is closed. M3.2 made semantic facts
query-backed and visible through `aurex_tooling::IdeSnapshot`; M3.3 turns that
snapshot into a long-lived tooling and LSP-consumable architecture.

M3.3 is not a language-feature stage. It is the bridge from "build one IDE
snapshot for one in-memory buffer" to "serve editor requests from a versioned
workspace state while reusing query-backed semantic facts".

## Current Baseline

- Branch: `m3.3`
- Parent baseline: `m3` / `m3.2` at
  `7f35b52 Complete M3.2 sema service and tooling query surface`
- Stable inputs:
  - R5 driver/session/pipeline/stage metadata.

  - M3.2 item/body/generic sema query authority and `IdeSnapshot::query`

## External Design References

M3.3 adopts these mature IDE/compiler ideas in Aurex-sized form:

- [Language Server Protocol 3.17](https://microsoft.github.io/language-server-protocol/specifications/lsp/3.17/specification/):
  use the standard initialize/shutdown lifecycle, text-document sync, requests,
  notifications, cancellation, diagnostics, hover, definition, references, and
  semantic-token capability shape. Do not let protocol DTOs become compiler
  internals.
- [clangd design](https://clangd.llvm.org/design/): keep open files and unsaved
  changes as first-class state, continuously produce diagnostics, answer editor
  requests from a parsed/checked per-file view, and add an index when cross-file
  knowledge is needed. Aurex will start single-threaded and in-process instead
  of cloning clangd's `TUScheduler` and background-index scale immediately.
- [rust-analyzer architecture](https://rust-analyzer.github.io/book/contributing/architecture.html):
  make client-supplied files and project structure the input state, derive
  semantic state lazily, keep syntax independent from LSP/semantic state, expose
  IDE-facing POD-like APIs, and keep LSP serialization in an adapter layer.

## Goals

1. Add a durable tooling session boundary above `IdeSnapshot`.
2. Introduce a small LSP adapter shell that consumes tooling API outputs instead
   of reaching into parser, sema, query, or driver internals.
3. Make open-buffer edits versioned and deterministic.
4. Reuse M3.2 semantic query records and dependency edges to explain what an edit
   invalidates.
5. Add cross-file IDE feature groundwork without building a full background
   index first.
6. Keep all existing compile, query, coverage, and stress gates green.

## Non-goals

- No VS Code extension, editor package, or platform installation story.
- No full completion engine, rename engine, code actions, formatting, inlay
  hints, or semantic tokens in the first implementation slice.
- No multi-threaded scheduler in the first implementation slice.
- No remote index, LSIF index, or persistent workspace database.
- No user traits, resource semantics, RAII, closures, const generics, package
  manager, or standard-library rebuild.
- No bypass around `IdeSnapshot`, `PipelineStage`, or query-backed sema facts.

## Core Architecture

```text
LSP JSON-RPC transport
  |
  +-- LspServer / LspDispatcher
        |
        +-- ToolingSession
              |
              +-- VersionedDocumentStore
              +-- ToolingProjectConfig
              +-- SnapshotCache
              +-- SemanticFactIndex
              +-- IncrementalReusePlanner
                    |
                    +-- aurex_tooling::build_ide_snapshot(...)
                    +-- M3.2 QueryContext records / dependency edges
```

The central rule is that LSP is only an adapter. The compiler-facing boundary is
`ToolingSession`, not LSP protocol structs. `ToolingSession` owns open document
state, project configuration, snapshot cache, and query-reuse summaries. It
returns IDE-native value types; an LSP adapter converts those values to protocol
JSON.

## Global Invariants

- A tooling request must read from one immutable snapshot version.
- A document change must produce a new versioned document state before any
  diagnostics or semantic query runs.
- LSP URI, filesystem path, package identity, module identity, and
  `SourceRole` must be normalized before building a query key.
- LSP DTOs are not semantic identity. Stable identity still comes from
  `PackageKey`, `FileKey`, `ModuleKey`, `ModulePartKey`, `DefKey`,
  `MemberKey`, `BodyKey`, `GenericInstanceKey`, and source ranges.
- Diagnostics must continue to use `IdeDiagnostic.owner_stages` and the existing
  structured diagnostic category/code contract.
- Syntax trees remain syntax data. Do not store semantic state in the lossless
  tree.
- Body-local edits must not invalidate unrelated item signatures or unrelated
  function bodies when query dependencies prove they are reusable.
- The first implementation stays deterministic and single-threaded; cancellation
  is represented by generation checks, not preemptive thread cancellation.

## Current Implementation Status (2026-05-29)

WP-1 through WP-6 are closed for the current acceptance boundary:

- Added `aurex_tooling::ToolingSession` above `IdeSnapshot`. It owns versioned
  open-document state, package/source-role configuration, URI/path
  normalization, stale client-version rejection, and snapshot caching. The
  session API remains protocol-neutral and does not expose LSP JSON or protocol
  DTOs.
- Added `build_ide_snapshot_into(...)` so cached `IdeSnapshot` objects are built
  in place. This avoids stale internal views after a returned snapshot is moved
  or copied.
- Added `aurex_tooling::LspServer` with deterministic JSON-RPC framing,
  initialize/shutdown/exit lifecycle, full-document text sync, diagnostics
  publishing, hover, definition, references, and documentSymbol support. The LSP
  layer calls only `ToolingSession`; it does not read parser, sema, query, or
  driver internals.
- Completed protocol-neutral IDE projection for diagnostics, hover, definition,
  references, and document symbols. Document symbols prefer M3.2 checked
  `semantic_facts` and fall back to top-level AST item projection when checked
  facts are absent.
- Added `src/tooling/reuse.cpp` for edit-impact planning and reuse summaries.
- Added `src/tooling/workspace_index.cpp` for a small open-file semantic index
  keyed by definition, member, body, and generic-instance identity.
- Added `tests/gtest/tooling/session_lsp_tooling_tests.cpp` covering session
  lifecycle, cache reuse, URI/path normalization, error paths, JSON-RPC framing,
  LSP lifecycle/text sync, diagnostics, hover, definition, references,

  workspace indexing.

The next batch should not start completion, rename, formatting, or semantic

## WP-1: Tooling Session And VFS Boundary

### Scope

Create a protocol-neutral session layer:

- `ToolingDocumentId`: normalized URI/path identity plus package/source role.
- `ToolingDocumentVersion`: monotonically increasing client version when
  available, otherwise session-local generation.
- `ToolingDocumentStore`: open/close/update full text first; incremental text
  edits are a later optimization.
- `ToolingProjectConfig`: package identity, import paths, root path, and default
  source role.
- `ToolingSession`: build or reuse `IdeSnapshot` for a document version.
- `ToolingSnapshotHandle`: immutable view used by hover/definition/references
  calls.

### Acceptance

- Unit tests cover open/change/close, stale version rejection, path/URI
  normalization, package identity propagation, and snapshot reuse.
- No LSP JSON appears in the compiler/tooling session API.
- Existing `build_ide_snapshot(...)` behavior remains unchanged for direct API
  callers.

## WP-2: LSP Protocol Shell

### Scope

Add the smallest useful protocol adapter:

- JSON-RPC header/content framing.
- `initialize`, `initialized`, `shutdown`, `exit`.
- `textDocument/didOpen`, `textDocument/didChange`, `textDocument/didClose`.
- `textDocument/hover`, `textDocument/definition`,
  `textDocument/references`.
- `textDocument/publishDiagnostics`.
- Static capability response for supported features.
- Request cancellation recognized by request id and generation checks.

### Deliberate Limits

- Start with full-document sync only. Incremental text-edit application can be
  added after the document store has focused tests.
- No completion, rename, code actions, formatting, or semantic tokens yet.
- No editor-specific extension fields.

## WP-3: IDE Feature Projection

### Scope

Make the protocol-neutral IDE value layer complete enough for LSP conversion:

- Hover: stable label, detail, source range, owner stages, semantic fact key.
- Definition: `DefKey`, `MemberKey`, `GenericInstanceKey`, range, part index.
- References: local references first, then same-open-file checked references.
- Document symbols: top-level item list and checked symbol kinds.
- Diagnostics: structured owner stages and module-part context.

This package should keep public API structs serializable in shape without
depending on a JSON library or LSP types.

## WP-4: Incremental Reuse Planner

### Scope

Use M3.2 query records to explain reuse:

- Convert `IdeEditImpact` into a candidate invalidation root.
- Compare old/new semantic `QueryRecord` and dependency edges.
- Report which facts are unchanged, recomputed, or invalidated.
- Preserve body-local edit behavior: a body edit should prefer
  `function_body_syntax` and `type_check_body` recomputation over global
  signature recomputation.
- Surface reuse summaries in tests and optional debug API.

The first slice can still call `build_ide_snapshot(...)`; the planner explains
reuse rather than replacing all computation with a demand-driven executor.

## WP-5: Workspace Semantic Index

### Scope

Add a small dynamic index for open files and package-local facts:

- Index definitions by `DefKey`.
- Index members by `MemberKey`.
- Index function bodies by `BodyKey`.
- Index generic instances by `GenericInstanceKey`.
- Merge same-file `IdeSnapshot` facts with open-buffer facts from sibling
  module parts.

This is not a background index. It is the minimum structure needed so LSP
definition/references can grow beyond one snapshot without scanning unrelated
state.

## WP-6: Quality Gates

For code-changing packages, run:

```sh
tools/format_check.py
git diff --check
cmake --build --preset full-llvm -j 8
ctest --test-dir build/full-llvm --output-on-failure
tools/check_coverage.sh
tools/query_pruning_gate.py --skip-build
tools/query_graph_fuzz.py --skip-build --rounds 4 --functions 8 --function-step 8
make perf-stress-threshold
```

When a package changes generic/type-check reuse, also run:

```sh
AUREX_STRESS_BUILD_DIR=build/full-llvm tools/generic_stress.py --skip-build --counts 200,500,1000,2000
```

LSP adapter packages must also include deterministic JSON-RPC fixture tests for
message framing, lifecycle, document sync, diagnostics, hover, definition, and
references.
