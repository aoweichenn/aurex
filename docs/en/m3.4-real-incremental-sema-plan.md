# Aurex M3.4 Real Incremental Sema Execution Plan

## Document Position

M3.4 starts after M3.3 Tooling Session And Incremental Sema is closed and
merged back to `m3`. M3.2 made semantic facts query-backed. M3.3 added a
long-lived `ToolingSession`, snapshot cache, LSP adapter boundary, edit-impact
reuse planning, and a small workspace semantic index.

M3.4 closes the next compiler-architecture gap: reuse must move from
explanation to execution. The compiler should not only report that a body-local
edit leaves most query facts unchanged; the next snapshot should actually reuse
those facts and recompute only the invalidated semantic roots.

## Current Baseline

- Branch: `m3.4`
- Parent baseline: `m3` after M3.3 at
  `50a6720 tooling: close M3.3 WP-4-6`
- Stable foundations:
  - R5 driver/session/pipeline/stage metadata.
  - M3.0 module/package/part identity.
  - M3.1 generic identity, ABI suffix, and generic query authority.
  - M3.2 item/body/generic sema query authority and `IdeSnapshot::query`
    facts.
  - M3.3 `ToolingSession`, versioned document store, snapshot cache, LSP
    adapter, reuse planner, and workspace semantic index.

## External Design References

M3.4 adapts mature compiler and IDE ideas to the current Aurex architecture:

- rustc-style red/green incremental compilation: reuse is valid only when the
  input keys and all dependency fingerprints prove that a query result remains
  unchanged.
- rust-analyzer/salsa-style derived state: source text and project structure are
  inputs; semantic facts are derived values that can be recomputed on demand.
- clangd-style edit responsiveness: opened buffers are first-class state, and
  editor requests should observe one immutable snapshot version.
- LLVM new pass-manager discipline: preserved analyses and invalidation must be
  explicit instead of assumed by convention.

Aurex should adopt the dependency and invalidation discipline, but not copy the
full rustc database, clangd scheduler, or LLVM pass infrastructure. The current
project needs a deterministic single-threaded executor first.

## Goals

1. Add an explicit previous-snapshot / previous-query context input to tooling
   snapshot construction.
2. Make semantic query reuse executable for unchanged records instead of only
   explainable after a full rebuild.
3. Preserve fine-grained invalidation boundaries for body-local, signature,
   generic, module-export, and module-graph edits.
4. Update workspace semantic indexes by affected fact identity where possible.
5. Keep CLI behavior, diagnostics JSON, profile JSON, incremental-cache format,
   and LSP protocol shape stable unless a work package explicitly updates them.
6. Keep full tests, focused tooling tests, query pruning, coverage, and stress
   gates green.

## Non-goals

- No completion, rename, semantic tokens, formatting, code actions, inlay hints,
  or editor extension work.
- No multi-threaded scheduler or background index.
- No package manager, dependency resolver, lockfile, or version solver.
- No user traits, associated types, const generics, RAII, resource semantics,
  closures, async, or standard-library rebuild.
- No second compiler frontend or parser/sema bypass for tooling.
- No query execution path that treats LSP DTOs as compiler identity.

## Core Architecture

```text
ToolingSession
  |
  +-- VersionedDocumentStore
  +-- SnapshotCache
  +-- IncrementalSnapshotInput
        |
        +-- current document text/version/config
        +-- optional previous IdeSnapshot
        +-- optional previous query snapshot
        +-- optional reuse plan from edit impact
  |
  +-- build_incremental_ide_snapshot(...)
        |
        +-- parse/module/sema pipeline
        +-- QueryReuseExecutor
        +-- SemanticFactReuseStore
        +-- WorkspaceSemanticIndex incremental update
```

The M3.4 implementation keeps the eager checked aggregate materialized, but the
semantic query layer now executes green seeding from the previous query snapshot:
when the current query key, stable identity, result fingerprint, and previous
dependency record match, the provider is skipped through `QueryContext` cache
reuse. The resulting reuse plan and workspace-index update are observable
through protocol-neutral tooling value types.

## Global Invariants

- One tooling request observes one immutable snapshot version.
- Source text, package identity, source role, module identity, part identity,
  target configuration, and relevant compiler options must be normalized before
  query keys are constructed.
- Stable semantic identity comes only from `PackageKey`, `FileKey`,
  `ModuleKey`, `ModulePartKey`, `DefKey`, `MemberKey`, `BodyKey`,
  `GenericInstanceKey`, and source ranges.
- A reused query result must prove that its key and dependency fingerprints are
  unchanged.
- A malformed or missing previous reuse decision is never trusted silently; it
  must become a recompute decision or a structured malformed-reuse diagnostic in
  debug/test APIs.
- Body-local edits must not invalidate unrelated signatures, module exports, or
  generic instance identities unless dependency edges prove a wider impact.
- Signature edits must invalidate downstream body and generic facts that depend
  on that signature.
- Module-surface edits must invalidate module export, import, lookup, and
  affected downstream facts without rebuilding unrelated modules.
- The eager `CheckedModule` may remain a materialized aggregate, but durable
  semantic facts must continue to identify their query authority.
- LSP JSON remains outside compiler and tooling semantic APIs.

## WP-1: Incremental Snapshot Build Input

Status: completed for the current acceptance boundary.

### Scope

Add an explicit input object for incremental IDE snapshot construction:

- Current document identity, version, source text, package config, source role,
  import paths, and root path.
- Optional previous `IdeSnapshot` for the same normalized document/config.
- Optional previous query snapshot containing query records and dependency
  edges.
- Optional edit impact and reuse plan produced by M3.3 reuse planning.
- A deterministic policy for when previous context is rejected.

### Acceptance

- `ToolingSession` can request a snapshot with previous context when the cached
  previous version matches the normalized document/config lineage. Completed.
- Missing, stale, mismatched, or malformed previous context falls back to clean
  recompute without crashing. Completed.
- Existing `build_ide_snapshot(...)` behavior remains available for one-shot
  callers. Completed.
- Tests cover cache hit, cache miss, stale version, mismatched context, and
  malformed previous context. Completed.

### Implementation Notes

- Added `ToolingIncrementalSnapshotInput`,
  `ToolingIncrementalSnapshotResult`, and
  `ToolingIncrementalSnapshotStatus`.
- `ToolingSnapshotHandle` now carries the incremental build result for the
  snapshot it exposes.
- `ToolingSession::change_document(...)` preserves the previous cached snapshot
  when the previous version was materialized, then `snapshot(...)` threads that
  context into the next build summary.
- `tooling_incremental_snapshot_result(...)` classifies clean builds, cached
  snapshots, accepted previous context, mismatched previous context, stale
  previous context, and malformed previous context.
- Existing one-shot `build_ide_snapshot(...)` and `build_ide_snapshot_into(...)`
  callers are unchanged.

## WP-2: Query Record Reuse Execution

Status: completed for M3.4.

### Scope

Convert reuse summaries into executable decisions:

- Classify query records as reusable, recomputed, invalidated, or malformed.
- Reuse unchanged query results for item signatures, function body syntax,
  type-check bodies, generic template signatures, generic instance signatures,
  and generic instance bodies where the current authority result can be proven
  equivalent.
- Keep dependency fingerprints and result fingerprints in the decision record.
- Expose a protocol-neutral debug summary for tests and future profile viewers.

### Acceptance

- Body-local edits reuse unrelated item signatures and unrelated bodies.
- Signature edits invalidate dependent body/type-check/generic facts through
  deterministic invalidated fact reporting.
- Generic template and module-surface edits remain bounded by stable query keys
  and dependency records; broader project-graph persistence stays in M3.6.
- Malformed decisions never assert; they recompute and surface a deterministic
  status.

### Implementation Notes

- Added `IdeIncrementalSnapshotInput` so snapshot construction can receive the
  previous `IdeQuerySnapshot` without changing one-shot callers.
- `build_ide_snapshot_into(...)` now seeds reusable file/lex/parse/diagnostics,
  module-surface, item-signature, generic-template-signature, function-body
  syntax, and type-check-body query records into `QueryContext` before provider
  evaluation.
- `ToolingIncrementalSnapshotResult` now exposes the executed
  `ToolingReusePlan` plus `ToolingQueryReuseExecutionSummary`, including reused,
  recomputed, invalidated, malformed, and body-local counters.

## WP-3: Semantic Fact Stability

Status: completed for M3.4.

### Scope

Make fact identity stable across incremental snapshot generations:

- Preserve `DefKey`, `MemberKey`, `BodyKey`, and `GenericInstanceKey` for
  unaffected facts.
- Preserve checked/unchecked fact classification and source ranges when syntax
  identity is unchanged.
- Keep source range updates local when an edit shifts following text but does
  not change semantic identity.
- Ensure materialized `CheckedModule` facts and tooling `semantic_facts` agree
  on reused identities.

### Acceptance

- Tests compare old/new semantic fact sets after body-local, signature,
  generic-use, module-part, and re-export edits.
- Reused facts remain addressable through hover, definition, references, and
  document symbols.
- Recomputed facts get new decision status without corrupting stable identity
  for unaffected facts.

### Implementation Notes

- Repeated body-local edit tests verify unchanged `add` item signatures and
  body syntax facts keep the same `DefKey`/`BodyKey` stable strings across
  snapshot generations.
- Body-local and removed-definition tests validate unchanged, recomputed, and
  invalidated semantic fact status at the tooling boundary.
- Generic template body-edit tests verify template signatures remain unchanged
  while the edited function body facts are recomputed.

## WP-4: Workspace Index Incremental Update

Status: completed for M3.4.

### Scope

Use M3.4 fact decisions to update the M3.3 workspace semantic index:

- Remove entries for invalidated facts.
- Replace entries for recomputed facts.
- Retain entries for reusable facts.
- Record per-document version and per-fact generation for debugging.
- Keep close-document cleanup deterministic.

### Acceptance

- Index updates after body-local edits do not drop unrelated definitions,
  members, bodies, or generic instances.
- Index updates report retained, replaced, removed, and inserted fact counts by
  stable fact identity.
- References/definitions/document symbols continue to read from the same
  protocol-neutral tooling APIs.
- Changed documents remove externally visible stale entries immediately while a
  private previous-fact snapshot is retained for the next incremental update
  comparison.

### Implementation Notes

- Added `ToolingWorkspaceIndexUpdateStats` and
  `ToolingWorkspaceSemanticIndex::update_snapshot(...)`.
- Workspace updates compare previous and current document facts by stable query,
  member, body, definition, generic instance, or fallback range identity.
- `ToolingSession` stores pending workspace facts across document changes so old
  versions are not externally visible but update stats still compare against
  the previous snapshot.

## WP-5: Performance And Stress Gates

Status: completed for M3.4.

### Scope

Add validation that M3.4 is measurably incremental and robust:

- Repeated body-local edit loop.
- Repeated signature edit loop.
- Multi-file module-part edit loop.
- Generic instance edit loop.
- Malformed reuse decision fixtures.
- Profile summary checks for reused/recomputed/invalidated counts.
- Coverage for new production files at the project threshold.

### Implementation Notes

- Added focused body-local edit-loop coverage, removed-definition invalidation
  checks, generic body-edit reuse checks, malformed reuse-plan checks, and
  workspace-index stale-version checks.
- The release validation set for this phase is `ctest`,
  `tools/check_coverage.sh`, `tools/query_pruning_gate.py`,
  `tools/query_graph_fuzz.py`, and `tools/generic_stress.py`.

### Default Commands

```sh
ctest --test-dir cmake-build-release --output-on-failure -j4
./tools/check_coverage.sh -j4
```

When a change touches query pruning, generic type checking, or graph behavior,
also run the relevant focused gates:

```sh
tools/query_pruning_gate.py --skip-build
tools/query_graph_fuzz.py --skip-build --rounds 4 --functions 8 --function-step 8
AUREX_STRESS_BUILD_DIR=cmake-build-release tools/generic_stress.py --skip-build --counts 200,500,1000,2000
```

## WP-6: Documentation And Release Closure

Status: completed for M3.4.

### Scope

- Update English and Chinese roadmap, next-steps, progress, and API docs.
- Record M3.4 acceptance and any remaining explicit non-goals.
- Remove stale M3.3 references that imply M3.3 is still active.
- Merge `m3.4` back to `m3` only after tests and coverage pass.

### Closure Notes

- M3.4 is closed as a deterministic, single-threaded incremental semantic-query
  execution layer. It does not introduce background scheduling, persistent
  workspace databases, or subtree reparsing; those remain assigned to M3.5 and
  M3.6.
- LSP DTOs remain outside compiler/query/sema internals.
- The next architectural stage is M3.5 Incremental Syntax And Stable AST
  Identity.
