# Aurex M3.5 Incremental Syntax And Stable AST Identity Plan

## Status

M3.5 is complete on the `m3.5` branch for the current deterministic
tooling/syntax boundary.

This phase adds no language syntax and does not introduce a second parser. It
moves the incremental architecture proven in M3.4 down into lossless syntax and
AST projection so repeated IDE edits can explain which syntax subtrees, AST
item/body identities, and semantic facts still line up.

## Delivered

1. **Range-based document edits**
   - `ToolingDocumentTextEdit` describes `begin`, `removed_length`, and
     `inserted_text`.
   - `ToolingSession::change_document_range(...)` applies small edits to the
     versioned document store.
   - `ToolingSession::change_document_range_with_reuse_plan(...)` returns the
     new version, applied edit, precise `IdeEditImpact`, reuse plan, and
     incremental snapshot result through one entry point.
   - Out-of-range edits return `invalid_source` without corrupting the current
     open document state.

2. **Position-independent lossless syntax stable keys**
   - `LosslessNodeStableKey` does not use absolute source ranges or token
     indexes.
   - The key is built from node kind, identifier-anchor hash, subtree token
     fingerprint, token/semantic-token/child shape, and depth.
   - Unchanged subtrees can keep the same stable key after prefix or local
     edits, while the existing `LosslessNodeKey` remains available for precise
     range/token debug identity.

3. **Lossless subtree reuse stats**
   - `compare_lossless_stable_nodes(previous, current)` compares immutable
     lossless trees by stable-key multisets.
   - `LosslessSyntaxReuseStats` reports previous/current node counts, reused
     nodes, recomputed nodes, invalidated nodes, and stable-key collisions.
   - `ToolingIncrementalSnapshotResult::syntax_reuse` exposes those counters to
     tooling/session callers.

4. **Stable AST item/body projection**
   - `IdeAstNodeInfo` and `ToolingAstNode` provide protocol-neutral projection
     from offsets to AST items or function bodies.
   - Item projections expose stable `DefKey` strings; function body projections
     expose stable `DefKey` plus `BodyKey` strings.
   - Offset-to-token, syntax-node, AST-node, and semantic-fact projections now
     align within the same snapshot/result surface.

5. **Normal tests**
   - `LosslessStableNodeKeySurvivesPrefixEdit` verifies stable keys for an
     unchanged function subtree after a prefix edit.
   - `ToolingSessionRangeEditReportsSyntaxReuseAndStableAstBody` verifies range
     edit application, syntax reuse stats, and stable AST body keys.
   - `ToolingSessionAppliesRangeEditWithoutReusePlan` verifies the plain range
     edit path also produces reusable previous context.

## Boundaries

- M3.5 still uses eager aggregate snapshot construction: lexer, lossless tree,
  AST lowering, and sema aggregate construction still materialize the current
  snapshot.
- Syntax reuse in this phase is immutable-subtree reuse explanation through
  stable keys and tooling results; old node objects are not physically grafted
  into the new tree.
- Full subtree reparse, physical green-tree sharing, background scheduling, and
  a persistent project database remain later work.
- M3.6 handles project graph and persistent query DB. M3.7 adds high-level IDE
  features such as completion, rename, and semantic tokens after the snapshot
  and reuse layers are stable.

## Acceptance

- Range edits and full replacement coexist while preserving stale-version
  checks.
- Unchanged lossless subtrees can be matched with `LosslessNodeStableKey` after
  prefix/local edits.
- `ToolingIncrementalSnapshotResult` reports sema query reuse, syntax reuse, and
  workspace-index update counters together.
- AST item/body projection stable keys match the `DefKey`/`BodyKey` shape used
  by semantic facts.
- Existing build, gtest, coverage, query pruning, query graph fuzz, and generic
  stress gates stay green.
