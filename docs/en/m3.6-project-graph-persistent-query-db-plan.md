# Aurex M3.6 Project Graph And Persistent Query DB Plan

Status: 2026-05-30. M3.6 is complete on the `m3.6` branch for the current
project graph and persistent query DB boundary.

## Stage Position

M3.6 does not add language syntax. It lifts the module, generic,
query-backed sema, tooling-session, and stable-syntax work from M3.0-M3.5 into
a project-level execution model: the same project/workspace input shape should
drive CLI `--check`, incremental cache, query graph execution, and tooling
sessions instead of letting the driver, module loader, and IDE layer assemble
roots independently.

Modern compiler references adapted for Aurex:

- Clang/LLVM treats compilation databases, target settings, driver options, and
  dependency/module files as reproducible build inputs.
- rustc incremental compilation separates crate graph, session options, dep
  graph nodes, and on-disk query cache records.
- rust-analyzer uses a VFS/open-file model and crate graph as the long-lived
  workspace input for query facts.
- Swift/Roslyn keep solution/workspace/project/document layers, where open
  buffers explicitly override file-system state.

Aurex adopts the useful boundary discipline without expanding scope: M3.6 adds
project model, project graph query, cache keys, and profile/invalidation
explanations. It does not add a package manager, dependency resolver, lockfile,
registry protocol, or version solver.

## Core Design

### ProjectModel / WorkspaceModel

M3.6 adds the `aurex_project` target. Core types live in
`include/aurex/infrastructure/project/project_model.hpp`:

- `ProjectModelInput`: session kind, package root, source root, package
  identity, `PackageKey`, import roots, target config, command options, and
  open buffers.
- `ProjectModel`: one normalized project input, with stable `ProjectKey`,
  identity fingerprint, and non-zero global id.
- `WorkspaceModel`: a stable identity over one or more projects.
- `ProjectTargetConfig`: emit kind, optimization level, clang path, clang args.
- `ProjectCommandOptions`: diagnostic format and query-pruning flag.
- `ProjectOpenBuffer`: path, uri, package identity, virtual buffer identity,
  source role, text fingerprint, text size, and generation.

Normalization rules:

- Package root, source root, import root, and open-buffer path use
  canonical-or-absolute paths.
- Import roots and open buffers are sorted before identity construction so call
  order does not affect stable keys.
- Open-buffer text contributes through `project_open_buffer_fingerprint(...)`.

### Driver / Tooling Adapters

The driver adds `include/aurex/application/driver/project_model.hpp`:

- `project_model_from_invocation(...)`
- `workspace_model_from_invocation(...)`

CLI input maps manifest root/source-root, `-I` import roots, package identity,
emit kind, optimization level, clang options, diagnostic format, and query
pruning into the project model.

`ToolingSession` now owns a `project::WorkspaceModel` and refreshes it on
document open/change/close:

- `ToolingProjectConfig` has `source_root`.
- Open buffers contribute virtual source role, uri, package identity, buffer
  identity, and text fingerprint.
- Tooling import roots use a `tooling-import-root:` package-identity prefix.

### Query Graph

The query layer adds:

- `QueryKind::project_graph`
- `ProjectKey`
- `project_graph_query_key(...)`
- `ProjectGraphProviderInput` / `ProjectGraphProviderOutput`
- `provide_project_graph_query(...)`
- `project_graph_query_record(...)`

`module_graph` is no longer an isolated root query. It explicitly depends on:

- the current project's `project_graph` query.
- the `module_part` queries composing that module.

This makes module-graph checking an explicit query-graph schedule decision
instead of an incidental result of recursive module-loader order.

### Persistent Query DB

Incremental cache schema moves from 1 to 2 and records project inputs in the
header:

- `project_identity`
- `package_root`
- `source_root`
- `target_config`
- `command_options`
- `open_buffers`

Cache query rows now support the `project_graph` kind. Query execution stats,
query-pruning profile output, provider-evaluation profile output, and subject
ordering all count project graph facts. The `module_graph` provider dependency
list is persisted and validates `module_graph -> project_graph` and
`module_graph -> module_part` edges.

### Invalidation / Profile

`try_reuse_incremental_check_cache(...)` now compares project inputs before
source-fingerprint or query-pruning reuse.

New profile sub-event:

```text
incremental_cache.project_inputs
```

Detail shape:

```text
result=reuse,changed_inputs=none
result=reject,changed_inputs=schema|project_identity|package_root|source_root|target_config|command_options|open_buffers
```

Reported changed inputs can include schema, compiler, mode, root, project
identity, package root, source root, target config, command options, open
buffers, package, import paths, and import packages.

## Non-Goals

M3.6 explicitly does not implement:

- package manager.
- dependency resolver.
- lockfile.
- registry protocol.
- version solver.
- multithreaded scheduler.
- physical syntax subtree grafting.
- IDE completion/rename/semantic tokens. Those move to M3.7.

## Acceptance

M3.6 acceptance surface:

- CLI `--check` and tooling sessions consume the same project/workspace input
  shape.
- Source root, package identity, target config, driver options, module graph,
  and open buffers affect stable cache keys.
- Persistent cache can write/read `project_graph` query rows and dependency
  edges.
- Module graph depends on project graph and module parts, and the query edge
  verifier rejects malformed kind/identity edges.
- Profile output explains project input reuse/reject and changed inputs.
- Existing tests, coverage, query pruning, query graph fuzz, and stress gates
  remain green.
