# Aurex M3.7 IDE Semantic Features Plan And Closure Record

## Stage Position

M3.7 builds on the M3.3 `ToolingSession`, M3.4 executable query reuse, M3.5 stable syntax identity, and
M3.6 project/workspace graph work. It does not add language semantics or change CLI compilation output. The goal is
to project existing query-backed semantic facts into protocol-neutral IDE features, with LSP remaining a thin adapter.

Core rules:

- `aurex_tooling` is the internal IDE semantic boundary. LSP DTOs must not enter parser, sema, query, or session code.
- Completion, rename, semantic tokens, inlay hints, code actions, and workspace symbols all read the same
  `IdeSnapshot` / `ToolingWorkspaceSemanticIndex` facts.
- LSP request handlers must guard against stale generations before publishing document-scoped results.
- Quick fixes are generated only from structured diagnostics data, not from arbitrary primary error text guessing.

## Completed Work Packages

### WP-1 Completion

- Added `IdeCompletionItem` and `ToolingCompletionItem`.
- Completion context now distinguishes item, expression, member, and module-path positions.
- Current-file completion merges checked globals, parameters, locals, fields, enum cases, methods, and keywords.
- The session layer also merges open-workspace semantic facts and deduplicates by stable definition/member/generic keys.
- Local symbols are visible only inside their scope range and after their declaration point.

### WP-2 Rename

- Added `ToolingRenamePlan`, `ToolingRenameEdit`, and `ToolingRenameConflict`.
- Rename targets come from `definition_at_offset` symbol identity, not raw text matching.
- Edit planning reuses references and includes definition ranges plus cross-file definition facts known to the workspace index.
- New names are checked for Aurex identifier shape, reserved keywords, and conflicts with visible symbols.
- The output is a protocol-neutral workspace edit plan; LSP only serializes it as `WorkspaceEdit`.

### WP-3 Semantic Tokens And Inlay Hints

- Added `IdeSemanticToken` and `ToolingSemanticToken`.
- Semantic tokens first classify syntax token kinds, then refine identifiers using checked symbol facts:
  function, method, parameter, variable, type, enum, enum member, property, keyword, number, string, comment,
  operator, punctuation, and related types.
- Token modifiers cover declaration, definition, and readonly.
- Added `IdeInlayHint` and `ToolingInlayHint`; the first slice emits type hints for checked locals without explicit type annotations.

### WP-4 Code Actions

- `ToolingDiagnostic` now preserves diagnostic children.
- Added `ToolingCodeAction` and `ToolingCodeActionEdit`.
- The current quick fix supports lookup suggestions: a help diagnostic shaped as `did you mean \`name\`?` produces a
  `quickfix` replacing that structured range.
- The implementation consumes structured severity/range/code/message data and does not infer fixes from plain primary
  error text.

### WP-5 Workspace Symbols And Cross-file Surface

- Added `ToolingWorkspaceSymbol`.
- `workspace_symbols(query)` materializes all open document snapshots before reading the workspace index.
- It returns stable symbols for item signatures, generic template signatures, and member facts, with query filtering.
- Completion, references, definition, and rename share the workspace semantic index stable keys.

### WP-6 LSP Projection

The LSP adapter now advertises and handles:

- `textDocument/completion`
- `textDocument/rename`
- `textDocument/semanticTokens/full`
- `textDocument/codeAction`
- `workspace/symbol`
- `textDocument/inlayHint`

The server remains a synchronous lightweight adapter, but document-scoped handlers record the starting
`ToolingDocumentVersion` and call `is_generation_current(...)` before returning a result.

## Current Limits

- Completion gets visible module exports from open-workspace semantic facts; it is not yet backed by a closed-world
  package index.
- Rename can plan edits across open-workspace definition facts, but it does not scan unopened files and does not yet own
  a full reference database.
- Member completion uses indexed member facts and does not yet perform full receiver type-directed completion.
- Quick fixes currently cover lookup suggestions only. Module/source-part fixes, import insertion, and type conversion
  fixes are deferred.
- Inlay hints currently cover local type hints only. Parameter hints, generic argument hints, and implicit conversion
  hints are deferred.
- The LSP server remains a small JSON adapter and does not yet implement a background scheduler, cancellation tokens, or
  concurrent request execution.

These are stage boundaries, not API-shape limitations. New IDE features should continue to follow the
`IdeSnapshot -> ToolingSession -> LSP adapter` projection chain.

## Next Stage Entry

M3.8 should push the query architecture below sema into lowering, IR, and backend work:

- Function and generic-instance lowering query authority.
- Type layout, enum layout, ABI symbol, and lower-generic-IR query facts.
- IR analysis preservation connected to query invalidation.
- Explicit boundaries between LLVM emission units and target-independent IR units.

M3.8 should not reopen the M3.7 LSP DTO boundary. IDE-facing code should continue to consume tooling value types only.
