# Aurex M3.8 Query-backed Lowering / Backend Reuse Plan And Closure Record

## Status

M3.8 is closed. This stage pushes the query-backed sema authority from M3.2-M3.4 down into lowering, the IR pass pipeline, and backend emission boundaries.

## Design Rules

- `lower_function_ir` is no longer backed by a sema/type-check placeholder fingerprint. Incremental cache writes lower IR query rows only after real lowering and the IR pass pipeline have completed.
- Ordinary function bodies and generic function instances keep stable `BodyKey` / `GenericInstanceKey` identities.
- Target-independent IR units, layout/ABI facts, and LLVM emission units are distinct fingerprint boundaries. Backend profile metadata can observe LLVM unit summaries, but IDE/tooling does not consume backend DTOs.
- The IR pass-manager summary explicitly exposes scheduled/executed passes, changed state, preserved analyses, and invalidated analyses.
- `check`, `typed`, and `checked` emit modes still stop at sema/checked output and do not write lower IR rows. `ir`, `llvm-ir`, and native emit modes write the incremental cache after lowering/pass execution, with real IR-unit facts.

## Delivered Work

1. Function-body lowering query authority
   - `lower_function_ir` result fingerprints now mix the type-check result, the real function IR unit fingerprint, and the LLVM emission unit fingerprint.
   - The dependency edge still points to the matching `type_check_body`, and the result is also dependency-sensitive so dependent rows are not incorrectly reused when the dependency changes.

2. Generic-instance lowering query authority
   - Generic function instance bodies map from `GenericInstanceKey` to the lowered IR function symbol.
   - `lower_function_ir` stable keys remain compatible with `GenericInstanceKey`, with dependency edges pointing to `generic_instance_body`.

3. Type-layout / enum-layout / ABI-symbol facts
   - Added `ir::layout_abi_fingerprint(...)`, covering record layout, payload enum record layout, global constants, and function ABI signatures.
   - Function IR unit fingerprints mix the layout/ABI fingerprint, so layout, enum payload storage, and ABI symbol changes invalidate the affected lower/backend unit facts.

4. IR pass preservation / invalidation
   - `PassPipelineRunSummary` now carries `invalidated_analyses`.
   - `analysis_id_name(...)` is the stable test/profile/debug name boundary.
   - `LoweringPipeline` records `ir.pass_pipeline` profile detail with scheduled/executed/changed/preserved/invalidated summary fields.

5. LLVM emission unit boundary
   - Added `ir::llvm_emission_unit_fingerprint(...)` and `ir::function_ir_unit_fingerprints(...)`.
   - `BackendPipeline` records `llvm.emit_ir` profile detail with function unit count and the layout/ABI fingerprint global id.
   - The backend still consumes optimized Aurex IR and does not turn LLVM text back into a query key.

## Tests

- `tests/gtest/ir/ir_fingerprint_tests.cpp` covers separation between function IR units, layout/ABI facts, and LLVM emission units.
- `tests/gtest/ir/pass_pipeline_tests.cpp` covers analysis names and preserved/invalidated analysis summary.
- `tests/gtest/driver/cli_driver_tests.cpp` covers typed emit skipping lower IR rows, IR emit writing lower IR rows, O0/O1 optimized IR unit fingerprint changes, query schedule order, dependency edges, and incremental-cache write-failure propagation at each emit stop point.

## Next

M3.9 should be release-baseline hardening: documentation alignment, dead/stale path cleanup, public API audit, full gates, and coverage stability without expanding the language-feature scope.
