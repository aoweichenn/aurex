# Aurex M3.9 M3 Release Baseline And Authority Audit

## Status

M3.9 is the final hardening and release-baseline stage for M3. It does not add language features. It closes the M3.0-M3.8 work on modules, generics, query-backed sema, tooling sessions, incremental syntax, project graph, IDE semantic features, and lowering/IR/backend reuse into the engineering baseline used before any M4 work starts.

The completed M3 line is:

- M3.0: language-level modules, module parts, package visibility, selective re-export, and module query boundaries.
- M3.1: generic identity, generic query authority, generic body/lowering, generic builtins, and method-local generics.
- M3.2: query-backed sema authority and semantic facts consumable by IDE/tooling.
- M3.3: protocol-neutral tooling session, LSP adapter boundary, reuse planner, and workspace semantic index.
- M3.4: real incremental sema execution.
- M3.5: range edits, lossless syntax stable keys, syntax reuse stats, and AST projection.
- M3.6: project graph, workspace model, and persistent query database.
- M3.7: completion, rename, semantic tokens, inlay hints, code actions, workspace symbols, and LSP projection.
- M3.8: query-backed lowering, IR unit fingerprints, pass invalidation summaries, and backend emission unit boundaries.

## Authority Boundary Audit

M3.9 fixes the following authority boundaries. Later stages may extend behavior inside these boundaries, but they must not bypass a boundary by reading another layer's private state or creating a parallel compilation path.

| Layer | Authority Input | Authority Output | Forbidden Bypass |
|:------|:----------------|:-----------------|:-----------------|
| Source / Lex | `SourceManager`, source identity, open-buffer text | tokens and lossless trivia-preserving source facts | sema reading files directly or lexing again |
| Parse / Syntax | token stream and parser recovery policy | AST, lossless tree, stable syntax keys, source ranges | sema/tooling inferring AST shape from tokens |
| Module / Project | `ProjectModel`, `WorkspaceModel`, `ModuleRecord`, `ModulePartKey` | module graph, module exports, module package exports, project graph query facts | import lookup bypassing loader/project graph |
| Sema | AST, module records, query authority input | `CheckedModule` durable facts, query records, semantic facts, diagnostics | tooling/LSP calling sema internals or reading session-local caches |
| Generic | `GenericInstanceKey`, canonical type identity, generic authority records | stable id, ABI suffix, generic signature/body facts, retained body view | using display strings, C names, or session-only `TypeHandle.value` as cross-session identity |
| Query / Incremental Cache | stable keys, authority records, dependency edges, project inputs | query records, query edges, reuse plan, provider-skip replay | cache reuse based only on timestamps or coarse source hashes |
| Tooling | `ToolingSession`, `IdeSnapshot`, workspace semantic index | diagnostics, hover, definition, references, completion, rename, tokens, hints/actions | LSP DTOs leaking into compiler core or LSP handlers bypassing tooling value types |
| Lowering / IR | AST + checked facts + retained side tables | verified Aurex IR, IR pass summary, IR unit fingerprints | backend reading AST/sema private tables, or lower IR rows written before real lowering |
| Backend | optimized Aurex IR, target config, LLVM emitter | LLVM IR/native artifacts, emission-unit profile facts | backend text becoming frontend/query authority |

## Public API Baseline

M3.9 keeps the following public APIs as the entry points for later work:

- `aurex::driver::CompilerInvocation`, `Compiler`, and the compilation pipeline public facade.
- `aurex::driver::pipeline_stage_record(...)`, `PipelineStageMetadata`, and profile phase classification.
- `aurex::query::*Key`, query authority/result helpers, `QueryContext`, `QueryExecutor`, and `QueryProviderSet`.
- `aurex::tooling::ToolingSession`, `IdeSnapshot`, protocol-neutral IDE value types, and the `LspServer` adapter.
- `aurex::project::ProjectModel` / `WorkspaceModel`.
- `aurex::ir::ModulePassManager`, `ModuleAnalysisManager`, verifier gates, and IR fingerprint APIs.

Any later trait/resource/closure/iterator/package-manager design must first explain where it plugs into these APIs instead of creating a new driver, module loader, sema path, or backend path.

## Final Gate Set

The M3 release baseline gate set is:

```sh
cmake --build cmake-build-release -j4
ctest --test-dir cmake-build-release --output-on-failure -j4
./tools/check_coverage.sh -j4
tools/query_pruning_gate.py --skip-build
tools/query_graph_fuzz.py --skip-build --rounds 4 --functions 8 --function-step 8
AUREX_STRESS_BUILD_DIR=cmake-build-release tools/generic_stress.py --skip-build --counts 200,500,1000,2000
tools/format_check.py $(git diff --name-only -- '*.cpp' '*.hpp') $(git ls-files --others --exclude-standard -- '*.cpp' '*.hpp')
git diff --check
```

Coverage keeps the existing 95% minimum for source lines, functions, and regions. Later production-code changes that push region coverage below the gate must add normal repository tests, not lower the threshold or rely on temporary tests.

## Residual Risks

M3.9 records these items explicitly and leaves them outside M3:

- Physical green-tree grafting, full subtree reparse, and a background multi-threaded scheduler are not on the main path yet.
- The workspace database remains the current project/query cache boundary, not a complete package manager or dependency resolver.
- User traits, associated types, const generics, resource capabilities, RAII, closures, async/generators/iterators, and standard-library rebuilds remain post-M3 design work.
- Backend emission units are observable, but there is not yet a full multi-target codegen-unit scheduler.
- Region coverage is a hard gate, so later changes need to preserve test margin early.

## Closure

M3 has moved Aurex from the M2.5 frontend foundation into a modern compiler architecture line: module/generic identity is stable, sema/query/tooling/lowering/backend boundaries are observable, and IDE plus incremental-cache behavior share the same authority facts. The M3 branch should not keep expanding language surface area; M4-level trait/resource/package/language-expressiveness work should start from this baseline as a separate design stream.
