# M1 进度报告

日期：2026-05-07

范围：M1 验收样例、标准库 OS 能力、subprocess stdout capture baseline、file metadata / mtime baseline、集成测试覆盖、测试耗时基线。
结论：M1 还没有完整完成，但已经从路线图推进到 active tree 中的可运行验收基线。当前项目已经能用 Aurex 写出两个最小系统级样例：一个 compiler frontend slice，以及一个 typed build-tool slice；两者都有 checked dump、IR dump 和 native run 覆盖。typed build-tool 已经开始使用 captured stdout 和真实 source/stamp mtime 比较，而不只是检查子进程 exit code 或 stamp 是否存在。

## 1. 本轮目标

这轮工作的目标不是继续补零散语法点，而是验证 M1 的核心问题：

- Aurex 是否已经能自然表达编译器前端中的 source manager、diagnostic、lexer、token stream 和 parser 子集。
- Aurex 是否已经能自然表达 typed build graph、target、dependency、custom command、incremental check、build、clean、run 和 test 流程。
- 构建工具能否使用 typed argv subprocess，而不是继续依赖 shell 字符串拼接。
- 新增能力是否能进入 integration tests，避免只靠手动 smoke。
- 新增验收用例是否会明显拖慢已有测试入口。

## 2. 已落地内容

### 2.1 标准库 subprocess baseline

新增 `std.sys.process::Command`，作为 M1 build-tool 的 typed subprocess 基线：

- 文件：`std/sys/process.ax`
- 新增结构：`Command`
- 新增 API：`command()`、`command_arg()`、`command_run()`、`command_run_capture()`、`command_destroy()`
- 新增 method API：`Command.arg()`、`Command.run()`、`Command.run_capture()`、`Command.destroy()`
- 新增拥有型输出：`ProcessOutput`
- 新增输出 API：`ProcessOutput.exit_status()`、`ProcessOutput.stdout()`、`ProcessOutput.stdout_c_data()`、`ProcessOutput.destroy()`
- 返回模型：`Result<i32, i32>`，成功时返回子进程 exit status。

底层 host-c support 新增：

- 文件：`std/ffi/c/support/host_c.c`
- 新增 ABI：`aurex_std_v0_run_process`
- 新增 ABI：`aurex_std_v0_run_process_capture`
- 新增 ABI：`aurex_std_v0_free_process_output_data`
- 实现方式：`fork` / `execvp` / `waitpid`
- stdout capture 通过 pipe + `dup2(STDOUT_FILENO)` 实现，返回的 buffer 由 `ProcessOutput.destroy()` 释放。
- capture FFI 使用调用方传 `HostProcessOutput*` 的形状，避免大 struct by-value return 在 arm64 C ABI 上出现不稳定约定。
- 失败约定：
  - program 为空、参数非法：返回 `127`
  - fork、malloc、waitpid 等 host failure：返回 `126`
  - 子进程被 signal 终止：返回 `128 + signal`
- `waitpid` 已处理 `EINTR`，避免信号打断导致误报失败。

当前还未完成：

- cwd 设置。
- env 设置。
- stderr capture。
- stdin/stdout/stderr pipe。
- timeout / kill。
- 更结构化的 process error enum。

### 2.2 标准库 file metadata baseline

新增 `std.fs.file::FileMetadata`，作为 M1 build-tool incremental check 的文件状态基线：

- 文件：`std/fs/file.ax`
- 新增结构：`FileMetadata`
- 新增 API：`metadata(path) -> Result<FileMetadata, i32>`
- 新增 method API：`FileMetadata.exists()`、`FileMetadata.is_file()`、`FileMetadata.is_dir()`、`FileMetadata.size()`、`FileMetadata.modified_time_ns()`
- `file_exists()` 已改为复用 `metadata()`，避免继续用 `fopen` 间接判断文件存在。

底层 host-c support 新增：

- 文件：`std/ffi/c/support/host_c.c`
- 新增 ABI：`aurex_std_v0_file_metadata`
- 实现方式：POSIX `stat`
- 返回内容：exists、is_file、is_dir、size、modified_ns
- FFI 形状：调用方传 `HostFileMetadata*`，避免 struct by-value return ABI 风险。

当前还未完成：

- 目录遍历。
- symlink metadata / lstat 区分。
- 权限、owner、mode 等更完整 metadata。
- path normalization。

## 4. 当前测试耗时基线

本轮验证结果：

```text
cmake --build build -j
通过

build/bin/aurex_tests --gtest_color=no '--gtest_filter=AurexIntegrationTest.M1Examples*'
2/2 通过
M1ExamplesCompileAndRun: 约 1229 ms
M1ExamplesExposeAcceptanceFeatureSet: 约 75 ms
合计: 约 1304 ms

tools/run_tests.sh
14/14 通过
Total Test time (real): 约 6.12 sec
```

判断：

- 新增 M1 验收测试没有把完整测试入口拉到不可接受范围。
- 当前 M1 native compile/run 的主要耗时仍来自真实编译和链接，而不是测试断言本身。
- checked/IR feature-surface 测试很快，适合继续扩展。
- 后续如果 M1 样例继续变大，应优先把更多断言放在 checked/IR 层；native run 保持少量端到端 smoke。

## 5. 当前 M1 完整度判断

可以认为已经完成：

- M1 两个验收程序的最小 active-tree baseline。
- typed subprocess 的最小标准库入口。
- stdout capture 的最小标准库入口。
- file metadata / mtime 的最小标准库入口。
- frontend slice 的 source manager / diagnostics / lexer / token stream / parser summary。
- build-tool slice 的 project / target / dependency / command / captured stdout / source-stamp mtime incremental / run / test 主流程。
- checked / IR / native 三层测试保护。
- 文档和 examples README 的入口说明。

不能认为已经完成：

- 完整自举前端。
- 完整 typed build tool。
- 完整 OS/process 标准库。
- 完整资源管理模型。
- 完整泛型约束、trait、class/object model。
- 完整 pattern matching 工业级诊断。
- 完整 AIR 后端契约。

一句话结论：

M1 已经从“规划中的功能集合”推进到了“可运行、可测试、可继续扩展的验收骨架”。下一步不是重新开路线，而是在这个骨架上继续补真实工程能力。

## 6. 下一步建议

优先级建议如下：

1. 继续增强 `std.sys.process::Command`
   - 加 cwd/env。
   - 加 stderr capture。
   - 加 stdin/stdout/stderr pipe。
   - 加 timeout。
   - 把状态码升级为更结构化的 process result。

2. 补目录和更完整 metadata API
   - 当前已有最小 metadata、mtime、is_file、is_dir。
   - 下一步补目录遍历、lstat/symlink、权限/mode 和 path normalization。

3. 补 `Map<K, V>` 或最小 string-key map
   - frontend 需要 symbol table。
   - axbuild 需要 target name -> id lookup。
   - 这是两个 M1 样例继续变真实的共同瓶颈。

4. 扩展 frontend 样例
   - 从 summary parser 推进到 AST node list。
   - 加 source span diagnostic。
   - 加 parser error recovery。
   - 加 module/import token 解析。

5. 扩展 axbuild 样例
   - target graph topological walk。
   - cycle diagnostics。
   - 真实 compile/link command modeling。
   - build/test 命令 stderr 捕获和结构化报告。

6. 保持测试性能纪律
   - checked/IR 断言覆盖大部分 feature surface。
   - native run 只保留关键端到端 smoke。
   - 大样例进入慢测分组前必须有明确价值。
   - 每轮新增测试后记录 focused test 和 `tools/run_tests.sh` 的 real time。
