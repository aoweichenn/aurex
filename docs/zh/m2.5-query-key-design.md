# M2.5 Query Key 设计

## 优先级

本设计是 M2.5 当前第一优先级。后续 QueryContext、依赖追踪、lossless syntax、
IDE-native diagnostics、generic instance 复用、trait/resource 设计冻结，都必须先遵守
这里定义的 key 边界。

M2.5 不先实现完整 query engine，也不先扩张高级语言特性。第一步是把 query key
设计冻结，并为现有 sema / driver / cache 逐步迁移提供稳定接口。

## 设计结论

Aurex 采用 **Stable Semantic Query Key + Session Fast Handle + Red/Green Fingerprint
QueryContext** 的混合架构。

- Stable Semantic Query Key：跨 session、跨进程 cache、IDE 编辑和未来并行 query
  共享的持久语义身份。
- Session Fast Handle：当前编译进程内部继续使用 `TypeHandle`、`ExprId`、`StmtId`、
  `IdentId`、interned id 和 arena index 保持热路径性能。
- Red/Green Fingerprint QueryContext：query key 表示“要计算什么”；fingerprint 表示
  “输入或结果有没有语义变化”。依赖变红后重算，结果 fingerprint 不变则上层依赖继续标绿。

这个选择借鉴 rustc / Salsa 的 demand-driven query 和 red-green change pruning、
Swift RequestEvaluator 的渐进迁移和 cycle detection、Bazel Skyframe 的 immutable
key/value dependency graph、Roslyn / rust-analyzer 的 lossless syntax 经验。Aurex
只采用适合当前工程阶段的部分：先稳定 key，再做 QueryContext，不一次性重写前端。

## 当前代码约束

现有实现已经具备稳定身份的种子：

- `include/aurex/sema/identifier.hpp` 已有 `StableFingerprint128`、
  `StableModuleId`、`StableDefId`、`StableMemberKey`、`IncrementalKey`。
- sema 诊断已经显式携带 semantic kind、category、code，message 不再承担机器语义。
- incremental cache 已落盘 source fingerprint、module row 和 checked def row。

但这些结构还不能直接视为完整 query key 系统：

- 稳定 ID 仍在 `sema` 层，后续 parse、driver、syntax、IDE 都要消费，应上移到
  `aurex::query` 或 `aurex::base` 可复用层。
- `TypeHandle` 是 `TypeTable` 的 session-local 下标，不能持久化为 semantic type key。
- `ExprId` / `StmtId` 是 compact AST 下标，不能作为跨 session 的 body/local syntax key。
- generic instance 当前仍有基于 `TypeHandle.value` 拼接 string key 的路径，只能保留为
  session lookup / ABI display 的过渡实现。
- 当前 `--incremental-cache` 仍是 coarse cache：所有 root/import source fingerprint
  匹配时 `--check` 直接命中；还不是细粒度 query cache。
- lexer 当前跳过 trivia，lossless syntax 尚未落地；因此 syntax node key 只能预留，
  不能反过来污染语义 key。

## 核心原则

1. Query key 表示“计算目标”，fingerprint 表示“内容是否变化”。不要把内容 hash
   塞进 identity。
2. 跨 session key 必须结构化、可序列化、确定性、与 display string 无关。
3. session-local handle 只用于当前进程内性能，不进入落盘 key、diagnostic protocol
   或 IDE semantic identity。
4. diagnostics 是 query 结果，不是全局 side-effect string sink。
5. generic/type/resource/trait identity 必须先 canonicalize，再参与 query key。
6. lossless syntax 支持 IDE 和局部解析，但 sema key 不能依赖临时 syntax offset。
7. 新高级特性进入实现前，必须先说明如何映射到 `DefKey`、`MemberKey`、
   `BodyKey`、`CanonicalTypeKey`、`GenericInstanceKey` 和 diagnostics query。

## Key 分层

### `CompilerConfigKey`

只包含影响语义、layout、ABI、target、feature gate、unsafe 策略的配置。

不进入该 key 的内容：

- 文本诊断格式。
- 颜色输出。
- profiling 开关。
- dump 展示选项。

这些展示配置不能导致 sema query 失效。

### `FileKey`

文件身份，不包含文件内容。建议字段：

- workspace / package identity。
- canonical path hash。
- virtual buffer identity。
- source role。

文件内容变化由 `FileContent(FileKey)` 的 result fingerprint 表达。如果 content hash
直接进入 `FileKey`，每次编辑都会制造一个新文件身份，IDE 增量复用会失效。

### `PackageKey`

当前阶段可以先由 root、import path set 和 package root 生成稳定 hash。未来升级为
manifest name、version、source root、dependency source 的结构化身份。

`PackageKey` 是后续 `pub(crate)`、`pub(package ...)`、module fragments 和包内可见性
的地基。

### `ModuleKey`

逻辑模块身份，建议字段：

- `PackageKey`。
- logical module path。
- module kind。

不要用物理文件路径直接代表模块。后续同一逻辑模块可能由多个文件 fragment 组成。

### `ModulePartKey`

同一模块多文件拆分的预留身份：

- `ModuleKey`。
- `FileKey`。
- part role / part name / stable part index。

局部编辑只应让对应 part 的 parse/body query 变红，但 module exports 仍归并到
同一个 `ModuleKey`。

### `DefKey`

稳定定义身份，建议字段：

- `ModuleKey`。
- `DefNamespace`。
- `StableDefPath`。
- `DefKind`。

`StableDefPath` 是语义路径，不是 display string。顶层函数、类型、const、enum case、
field、method、generic template 都应能映射到 `DefKey` 或 `MemberKey`。

重复定义场景不应靠不稳定 disambiguator 随机制造新 key。首选做法是同一语义位置产生
冲突诊断，由 diagnostics query 解释冲突。

### `MemberKey`

成员身份，建议字段：

- owner `DefKey`。
- member name atom。
- member kind。
- stable ordinal，仅用于 tuple field、synthetic payload 等没有源名的成员。

struct field、enum case、trait method、associated type、associated const、future
drop glue 都应能放进这个模型。

### `BodyKey`

函数体或表达式体身份，建议字段：

- owner `DefKey`。
- `BodySlot`。

`BodySlot` 至少预留：

- function body。
- const initializer。
- global initializer。
- default argument expression。
- trait default method body。
- destructor/drop body。
- future closure body。

不要用 `ExprId` 或 `StmtId` 做 `BodyKey`。

### `GenericParamKey`

泛型参数身份，建议字段：

- owner `DefKey`。
- parameter index。
- parameter kind。

参数名用于诊断和 debug print，可以参与 signature fingerprint，但不能单独作为唯一身份。
这样改名不会不必要地改变泛型实例身份。

### `CanonicalTypeKey`

这是 query-key 系统里最关键的类型层。它不能保存 `TypeHandle.value`。

建议做成结构化 variant：

- builtin。
- pointer。
- reference。
- array。
- slice。
- tuple。
- function。
- nominal type：`DefKey + type args`。
- generic param：`GenericParamKey`。
- future const arg。
- future associated type projection。
- future trait object，M2.5 不实现但保留 tag 空间。

`TypeHandle` 继续保留为当前 session 的高速查表结果；`CanonicalTypeKey` 用于
query/cache/generic instance/diagnostic protocol。

### `GenericInstanceKey`

泛型实例身份，建议字段：

- template `DefKey`。
- canonical type args。
- canonical const args。
- `ParamEnvKey`。

`ParamEnvKey` 预留 where clause、trait bound、resource capability、associated type
约束和 future specialization 环境。

`TargetLayoutKey` 不应默认进入通用 semantic instance key。layout 和 lowering 相关
差异应放进 `TypeLayout(...)`、`LowerFunctionIR(...)` 等 query 的 config key。

### `LocalSyntaxKey`

lossless CST 落地前，只允许作为临时辅助身份：

- `BodyKey`。
- local node path hash。
- node kind。

它不能进入跨 session sema key。lossless CST / GreenTree 落地后，再升级成稳定 syntax
node identity。

### `DiagnosticKey`

诊断身份，建议字段：

- producer `QueryKey`。
- `DiagnosticKind`。
- primary stable subject。
- ordinal role。

诊断输出顺序、去重、note/help 关联和 future fix-it 都要基于结构化事件。CLI 文本、
JSON 和 LSP 只负责渲染，不重新推断语义。

## QueryKey 表达

统一概念是：

```cpp
struct QueryKey {
    QueryKind kind;
    QuerySchemaVersion schema;
    QueryPayload payload;
};
```

实现时不建议在所有热路径复制一个巨大的 `std::variant`。更好的形状是：

- 每个 query kind 定义自己的 typed key。
- 每个 typed key 提供 stable serialize、stable hash、debug print、equality。
- `QueryInterner` 把 typed key intern 成小的 `QueryNodeId`。
- 内存热路径传 `QueryNodeId`。
- 落盘 cache 写 `QueryKind + schema + stable key bytes/hash + result fingerprint`。

## 第一批 Query

M2.5 初期只做能明显改善架构边界、又不会推翻整条 pipeline 的 query。

| Query | Key | 输出 | 初期目的 |
|:------|:----|:-----|:---------|
| `FileContent` | `FileKey` | source snapshot + content fingerprint | 统一文件和 IDE buffer 输入 |
| `LexFile` | `FileKey + LexConfigKey` | token stream | 为 lossless lexer 预留边界 |
| `ParseFile` | `FileKey + ParserConfigKey` | compact AST 或 future CST root | 拆出 parse 入口 |
| `ModuleGraph` | root key + import set key | module/import DAG + cycle diagnostics | 模块依赖 query 化 |
| `ModuleExports` | `ModuleKey` | exports/re-exports/import-visible surface | 支撑模块级失效 |
| `ItemList` | `ModuleKey` | top-level item skeleton | 分离 item discovery 和 body checking |
| `ItemSignature` | `DefKey` | checked signature + signature fingerprint | body 修改不污染签名依赖 |
| `FunctionBodySyntax` | `BodyKey` | body syntax handle | body query 的输入层 |
| `TypeCheckBody` | `BodyKey` | checked body facts + local diagnostics | 函数体级增量 |
| `GenericTemplateSignature` | `DefKey` | generic params + where/capability | 泛型模板稳定身份 |
| `GenericInstanceSignature` | `GenericInstanceKey` | substituted signature | 泛型签名复用 |
| `GenericInstanceBody` | `GenericInstanceKey` | instantiated body facts | 泛型 body / lowering 复用 |
| `Diagnostics` | producer `QueryKey` | diagnostic event stream | CLI/JSON/LSP 统一来源 |
| `LowerFunctionIR` | `BodyKey` 或 `GenericInstanceKey` + `IrConfigKey` | Aurex IR | 后端增量边界 |

第一批真实 query 的迁移从 `ItemSignature(DefKey)` 开始，并已经扩展到 source-stage、
module-stage、body-stage、generic-stage、lowering-stage 和 diagnostics-stage 的
query-key-compatible rows。后续仍以 provider 边界和 red-green graph 为准，不再回退到
driver 侧手写 query row。

## 失效模型

M2.5 采用 red-green change pruning：

1. 输入变化时，对应 input query 标红。
2. 依赖它的 query 被标脏，但不立即全量重算。
3. 访问 query 时按依赖顺序重算必要节点。
4. 如果 query 新旧 result fingerprint 相同，该 query 标绿，上层依赖可以继续复用。
5. 如果 result fingerprint 不同，继续向上扩散。

典型收益：

- 改注释或空白：`FileContent` 变，lossy sema signature 不变，上层大部分 query 绿。
- 改函数体但不改签名：`TypeCheckBody` 变，`ItemSignature` 和调用方 lookup 绿。
- 改 private helper：当前 module 内受影响，public `ModuleExports` 可能保持绿。
- 改 public signature：`ItemSignature` 变，依赖模块相应 query 变红。
- 改 generic body：已实例化 body query 按 `GenericInstanceKey` 局部失效。

## 性能规则

- Key 必须小且不可变。热路径传 interned id，不反复复制 vector/string。
- stable hash 和 equality 分离。hash 只做快速定位和落盘索引，最终仍用 typed equality
  确认。
- display string 延迟生成。dump、diagnostic message、ABI suffix 才生成展示文本。
- `CanonicalTypeKey` 需要 interner。常见小 children 应支持 inline storage 或 arena
  storage，避免泛型实例大量 heap allocation。
- result fingerprint 必须结构化。signature fingerprint 只包含对外语义，不包含诊断文案、
  source range、dump 顺序或 profiling 信息。
- query provider 必须显式声明依赖。不能从隐式全局 mutable state 偷读 source、module、
  checked side table。
- 先单线程正确，再并行。QueryContext 先有 `in_progress`、cycle stack、done/failed
  状态，为后续并行做边界。

## 明确禁止

- 禁止把 `ExprId`、`StmtId`、`TypeHandle.value`、vector index、source offset 作为
  持久 query key。
- 禁止把 display name、ABI C symbol、dump 文本或 diagnostic message 当 semantic key。
- 禁止把完整 AST hash 当作唯一增量边界。
- 禁止先实现高级语言特性，再补它们的 stable identity。
- 禁止为 LSP 另建一套与 CLI/JSON 不同的 diagnostics 数据源。
- 禁止为了“看起来增量”保留第二套并行前端。

## 落地顺序

1. 新增 `include/aurex/query` 和 `src/query` 基础模块：stable hash、typed query key、
   canonical type key、generic instance key、debug print、stable serialize。
2. 将现有 `StableFingerprint128`、`StableModuleId`、`StableDefId`、`StableMemberKey`、
   `IncrementalKey` 的语义归属从 sema 上移到 query/base 层，sema 保留 adapter。
3. 实现从 `TypeTable + TypeHandle` 到 `CanonicalTypeKey` 的 builder，先覆盖现有
   builtin、pointer、reference、array、slice、tuple、function、nominal、generic param。
4. 将 generic instance 的稳定身份改为 `GenericInstanceKey` 的结构化 fingerprint。
   现有 string key 可暂时保留为 session lookup 和 ABI/display 过渡层。
5. 扩展 incremental cache，先写 query-key-compatible rows，不改变 coarse cache 行为。
6. 接入第一个真实 query：`ItemSignature(DefKey)`。
7. 接入 `ModuleExports(ModuleKey)` 和 `Diagnostics(QueryKey)`。
8. 接入 `TypeCheckBody(BodyKey)`、`GenericInstanceBody(GenericInstanceKey)`。
9. 接入 `LowerFunctionIR(BodyKey / GenericInstanceKey)` 的 query row、provider dependency
   和 pruning gate。
10. 补齐 `FileContent(FileKey)`、`LexFile(FileKey + LexConfigKey)`、
    `ParseFile(FileKey + ParserConfigKey)` 的 query 层 provider/context 边界，并接入
    driver cache subject collection。
11. 在 current-record green cut 已落地的基础上推进 demand-driven provider、lossless CST /
    GreenTree、局部增量 parse、IDE hover/definition/references。

## 当前落地状态

截至当前 M2.5 query-key 主线，Aurex 已经把 query-key-compatible cache row 的写入边界
收敛到 typed input/provider 模型：

- `ItemSignature(DefKey)` 已有最小真实 provider 边界：query 层提供
  `ItemSignatureProviderInput`、`ItemSignatureProviderOutput` 和
  `provide_item_signature_query`，输入是 `DefKey + signature IncrementalKey`，输出是
  query record、result fingerprint 和显式依赖列表。
- `FileContent(FileKey)`、`LexFile(LexFileKey)` 和 `ParseFile(ParseFileKey)` 已补上
  query 层 provider/context skeleton，并已接入 driver cache subject collection。
  `LexConfigKey` 会进入 `LexFileKey` 的 stable
  identity；`ParserConfigKey` 内含其消费的 `LexConfigKey`，因此 `ParseFile` 的 key 从
  第一版开始就能表达“同一文件、不同 lexer/parser 配置”的不同 cache 身份。
  provider dependency 形状已经固定为 `LexFile -> FileContent`、
  `ParseFile -> LexFile`。driver 现在会从 `SourceManager` 为每个源码文件采集
  source-stage subject：`FileContent` result fingerprint 绑定 raw source text；
  `LexFile` result fingerprint 由真实 lexer token kind + token text 生成，故注释和
  空白不会直接改变 token stream result；`ParseFile` result fingerprint 暂以
  parser config + `LexFile` result 表达 parse skeleton。这一步仍未改变真实 driver 的
  module read/lex/parse 执行方式，也不跳过前端，只把 source-stage rows 和
  dependency edges 落盘。
- query 层已有最小 `QueryContext` skeleton：支持 query node 状态
  in-progress/done/failed、`FileContent` / `LexFile` / `ParseFile` / `ModuleGraph` /
  `ItemList` / `ModuleExports` / `ItemSignature` / `GenericTemplateSignature` /
  `GenericInstanceSignature` provider 注册/调用入口、缓存命中复用、re-entrant cycle 观测、
  dependency recording、
  dependency edge table、cache replay seed 和 `completed_records()` cache record emission
  边界。`QueryContext::completed_records()` 仍按 query kind、key id、result id 和
  stable key bytes 给通用调用者提供确定性排序；driver cache writer 则按已经排好的
  query subject dependency schedule 回收 done record，避免把 cache row emission 退回
  `QueryKind` enum / unordered map 顺序。
- `StableDefId -> DefKey` 的过渡 adapter 已上移到 query 层；driver 侧只负责从
  `CheckedModule` 采集 `ItemSignatureQuerySubject`，并通过 `QueryContext` 按 subject
  schedule 回收已完成 record 写出 item signature query rows，不再自己拼 provider 输出。
  sema 侧的 generic instance identity 也统一通过这个 adapter 生成 template
  `DefKey`，避免 `GenericInstanceKey.template_def` 和 driver 写出的
  `GenericTemplateSignature(DefKey)` 使用两套不同的 module/definition identity。
- `GenericInstanceSignature(GenericInstanceKey)` 已有 provider 边界，并接入
  `QueryContext`；driver 侧只负责采集泛型实例 subject，不直接生成泛型 query record。
  provider input 和 driver subject 对 `GenericInstanceKey` 使用非拥有
  `const GenericInstanceKey*` 边界；当前求值是同步的，key 生命周期由
  `CheckedModule` / 测试 fixture 保证，避免在 cache row emission 热路径复制泛型实参
  vector 和稳定 fingerprint payload。
- `ModuleExports(ModuleKey)` 已接入第一版真实 provider 边界。driver 会从
  `ModuleRecord + CheckedModule` 采集每个逻辑模块的 public function / struct /
  enum case / type alias surface，生成结构化 result fingerprint，并通过
  `QueryContext::evaluate_module_exports` 写出 `module_exports` query row。
- `ModuleGraph(ModuleKey)` 和 `ItemList(ModuleKey)` 已接入第一版真实 provider 边界。
  `ModuleGraph` 的 result fingerprint 当前由 logical module key、模块集合、模块名和
  canonical path 生成；`ItemList` 会从 `CheckedModule` 采集同模块所有 top-level item
  skeleton，包括 generic template、function、struct、enum case 和 type alias，并把
  item kind/name/stable id/incremental key 写进结构化 fingerprint。
- 当前模块级依赖边已经固定为 `ItemList(ModuleKey) -> ModuleGraph(ModuleKey)`，
  `ModuleExports(ModuleKey) -> ItemList(ModuleKey)`。这让 public exports 和 item discovery
  分离，后续 module fragment、import DAG、re-export 和 private item 剪枝都可以沿这条边扩展。
- `GenericTemplateSignature(DefKey)` 已接入第一版 provider 边界。sema 会在注册泛型
  struct / enum / type alias / function / method template 时记录
  `GenericTemplateSignatureInfo`，其 signature fingerprint 包含 item kind、参数名和排序后的
  capability constraints。provider 显式依赖 `ItemList(ModuleKey)`，因此 template signature
  不再直接悬挂在 module exports 上。
- `ItemSignature(DefKey)` provider 继续显式依赖 definition 所在模块的
  `ModuleExports(ModuleKey)`；`GenericInstanceSignature(GenericInstanceKey)` provider
  已改为依赖 template 的 `GenericTemplateSignature(DefKey)`。当前 generic 链路因此变成
  `ItemList -> GenericTemplateSignature -> GenericInstanceSignature`，比直接依赖
  `ModuleExports` 更接近 rustc/Salsa 式 typed query graph，也避免 private generic template
  被错误建模成 public exports 的语义输入。
- `FunctionBodySyntax(BodyKey)` 和 `TypeCheckBody(BodyKey)` 已接入第一版 body 级
  query skeleton。driver 只为有 definition、无 conflict、且能从 `SourceManager`
  range 取回源码片段的普通函数采集 body subject；`FunctionBodySyntax` 的 result
  fingerprint 来自 `BodyKey + 函数定义源码片段`，`TypeCheckBody` 的 result
  fingerprint 来自 `BodyKey + body syntax result + ItemSignature incremental key`。
  这一步仍不跳过 parse/sema，但 cache row 已经能精确区分“签名不变、函数体变化”的
  body 级 recompute。
- `TypeCheckBody(BodyKey)` provider 会显式依赖同一个 `BodyKey` 的
  `FunctionBodySyntax(BodyKey)`，以及 owner `DefKey` 的 `ItemSignature(DefKey)`。
  因此当前 query graph 已经具备 module -> signature -> type-check body 以及
  body syntax -> type-check body 的最小依赖形状。
- `GenericInstanceBody(GenericInstanceKey)` 已接入第一版 provider skeleton。它只为
  retained generic function instance 且有 definition/source range 的实例采集 subject；
  result fingerprint 来自 `GenericInstanceKey + instantiated signature incremental key +
  template body source range`，provider 显式依赖同一个 `GenericInstanceKey` 的
  `GenericInstanceSignature(GenericInstanceKey)`。这一步仍不直接跳过 lowering，但已经
  给泛型 body / future lowering reuse 留下稳定 cache row 和 dependency edge。
- `LowerFunctionIR(BodyKey / GenericInstanceKey)` 已接入第一版 provider skeleton。普通
  函数 lowering query 的 result fingerprint 来自 `BodyKey + TypeCheckBody result`，
  provider 显式依赖同一个 `BodyKey` 的 `TypeCheckBody(BodyKey)`；泛型实例 lowering
  query 的 result fingerprint 来自 `GenericInstanceKey + GenericInstanceBody result`，
  provider 显式依赖同一个 `GenericInstanceKey` 的
  `GenericInstanceBody(GenericInstanceKey)`。这一步仍不跳过真实 `ir::lower_ast`，但已经
  把后端/IR 层的 cache row 和 dependency edge 放进 red-green graph，后续可以在不改
  stable key schema 的前提下接入真正 demand-driven lowering。
- `Diagnostics(QueryKey)` 已接入第一版 producer diagnostics skeleton。driver 会为当前
  已采集的每个非 diagnostics query 生成一个 diagnostics query subject；result
  fingerprint 当前表示“该 producer 的空 diagnostics stream”，provider 显式依赖
  producer `QueryKey`。因此当 producer result 改变时，query plan 会把对应 diagnostics
  标为 dependency-propagated recompute；如果 diagnostics result 仍为空且不变，后续上层
  query 可以继续按 red-green 规则标绿。
- incremental cache 的默认路径仍保持 coarse reuse 语义；未开启实验开关时，
  `try_reuse_incremental_check_cache` 仍要求所有 source fingerprint 完全匹配。
- cache 读取侧已经把 query row 解析为 `ParsedQueryRecord`，校验 schema、query kind、
  payload fingerprint、query global id、result fingerprint 和 stable key bytes，并按
  `QueryKind + stable_key_bytes` 拒绝重复 query identity。
- cache 读取侧也会校验 query dependency edge 的图不变量：edge 的 dependent 和
  dependency 都必须能解析到同一 cache 内的 query row，edge 不能自环或重复，dependency
  的 schedule rank 不能晚于 dependent，并且 dependency kind 必须匹配当前 provider
  合同。例如 `ItemSignature` 只能依赖 `ModuleExports`，`GenericInstanceSignature` 只能依赖
  `GenericTemplateSignature`，`TypeCheckBody` 只能依赖 `FunctionBodySyntax` /
  `ItemSignature`，`LowerFunctionIR` 只能依赖 `TypeCheckBody` / `GenericInstanceBody`，
  `Diagnostics` 只能依赖非 diagnostics producer。cache writer 在发布文件前会对当前
  `QueryCollection` 做同样的 closed、scheduled 和 expected-kind 检查，防止 provider
  产生的断边或错种类边进入可复用 cache。
- cache 读取侧和 writer 发布前现在还会校验 dependency edge 的 stable identity
  关系，而不只看 query kind。`LexFile` 必须指向同一个 `FileKey` 的 `FileContent`；
  `ParseFile` 必须指向同一个 `FileKey` 且同一个 `LexConfigKey` 的 `LexFile`；
  `ItemSignature(DefKey)` 必须指向同一 `ModuleKey` 的 `ModuleExports`；
  `GenericTemplateSignature(DefKey)` 必须指向同一 `ModuleKey` 的 `ItemList`；
  `GenericInstanceSignature(GenericInstanceKey)` 必须指向其 template `DefKey` 的
  `GenericTemplateSignature`；`TypeCheckBody(BodyKey)` 必须指向同一个 `BodyKey` 的
  `FunctionBodySyntax` 和 owner `DefKey` 的 `ItemSignature`；`LowerFunctionIR` 必须指向
  同一个 `BodyKey` 或 `GenericInstanceKey` 的前置 query；`Diagnostics(QueryKey)` 必须
  精确指向其 producer query key。当前实现采用 stable key bytes 的结构化切片投影来完成
  这层校验，等 typed stable-key decoder 落地后再把这段逻辑迁移成显式 typed
  `QueryEdgeVerifier`。
- `QueryContext` 现在也在 provider 完成和 cache replay seed 边界执行 expected-kind
  校验。也就是说，错误依赖不需要等到落盘时才被发现；provider 如果返回
  `ItemSignature -> ItemList` 这类 schedule 上看似可行但语义合同错误的 edge，会直接把
  对应 query 标为 failed。这个边界后续可以迁移到真正 demand-driven provider，而不需要
  重新定义 cache graph 语义。
- 旧 cache 的 parsed query rows 已经可以 seed 进 `QueryContext`，driver 的 query diff
  路径通过 seeded context 查询旧 query record；如果 cache 中存在 `query_edges` /
  `query_edge` 行，seed 时会同步恢复 dependency table，用于
  `QueryReusePlan` 的 recompute propagation。
- `query_record_change_status` 已提供旧 cache query record 与新 query record 的纯函数
  对比状态：missing、unchanged、changed、malformed。query 层已有最小
  `QueryReuseDecision` / `QueryReuseSummary` 边界，把 change status 映射为
  reuse/recompute disposition；同时已有 `QueryReusePlan`，把 decisions 归并为
  reusable、direct recompute roots、dependency-propagated recompute 和最终 recompute
  worklist。driver 侧现在同时暴露 query diff/profile phase：`incremental_cache.query_diff`
  输出 total/missing/unchanged/changed/malformed，`incremental_cache.query_plan`
  输出 reusable/recompute_roots/propagated_recompute/recompute，
  `incremental_cache.source_stage_reuse` 输出 early lookup 的
  result/reason/sources/queries/unchanged/missing/changed/malformed/source_failures。前两者在
  cache write 路径已经驱动 provider-skip；source-stage reuse phase 只在实验 lookup
  路径实际尝试时出现，在默认 cache lookup 路径仍不改变 coarse 命中语义。
- `try_reuse_incremental_check_cache` 已接入第一版 experimental source-stage green reuse。
  开启 `--experimental-query-pruning` 且 raw source fingerprint 不匹配时，driver 会读取
  cache 记录的 source paths，重建当前 `LexFile` / `ParseFile` query record，并与旧 cache
  中同 identity 的 query row 比对；如果所有 source 的 token/parse query 都 unchanged，
  `--check` 可以直接复用旧 semantic-ok cache。默认路径不启用这条捷径；token 变化、
  parse result 变化、缺失 source-stage query row、缺失 source 文件或 malformed cache
  都会拒绝早期复用，并在 profile 中记录首个拒绝原因。当前测试同时覆盖 root source
  与 import source 的 comment-only green case，确认所有 cache source paths 都必须通过
  LexFile/ParseFile green 检查。这一步把“注释/空白变化但 lossy token stream 不变”的收益
  前移到 compile 前的 cache lookup 阶段，但还不是完整 demand-driven sema/lowering reuse。
- driver 已经接入 feature-gated pruning gate：`--experimental-query-pruning`
  默认关闭；开启后，cache write 阶段会先采集轻量 query subject 和候选 query record，
  再消费 `QueryReusePlan`。reusable query 会从旧 cache replay 出的 `QueryContext`
  seed record 和 dependency edges，跳过对应 query provider；recompute /
  propagated recompute query 才进入 provider 求值。`incremental_cache.query_pruning`
  profile 会暴露 enabled/applied/reused/recomputed/fallback，并进一步按
  `file_content`、`lex_file`、`parse_file`、`module_graph`、`module_exports`、`item_list`、
  `item_signature`、`function_body_syntax`、`type_check_body`、
  `generic_template_signature`、`generic_instance_signature`、`generic_instance_body`、
  `lower_function_ir` 和 `diagnostics` 拆出
  reused/recomputed 统计。这个 gate 仍是
  coarse-safe：它本身不跳过 parse/sema；真正 compile 前 early reuse 只由上一条
  experimental source-stage green reuse 处理。
- driver 的 query collection 顺序已经固定为显式 dependency schedule：先求值并写出
  `FileContent`、`LexFile`、`ParseFile`，再进入 `ModuleGraph`、`ItemList`、
  `ModuleExports`、`ItemSignature`、`GenericTemplateSignature`、
  `GenericInstanceSignature`、`FunctionBodySyntax`、`TypeCheckBody`、
  `GenericInstanceBody`、`LowerFunctionIR` 和 `Diagnostics`。真实 cache 文件会按这个
  schedule 持久化 query row，同时持久化 query dependency edge；旧 cache 缺少
  `query_edges` header 时仍按空 edge 表兼容。
- 已有回归测试覆盖：函数体变化但签名不变时 `item_signature` result fingerprint
  不变，同时对应 `function_body_syntax` / `type_check_body` result fingerprint 变化；
  函数签名变化时同一个 `DefKey` 的 result fingerprint 变化。
- incremental cache writer 的 query row emission 已从 `QueryContext`
  completed-record map order 下沉为“subject schedule 求值 + 按 subject schedule 回收
  completed record”。QueryContext 内部也已有最小 dependency edge table；cache 文件已开始
  持久化 `query_edges` / `query_edge` 行，并在 replay 时把 edge table 重建到 cached
  `QueryContext`。当前 experimental pruning 路径已经下沉到 query provider 的
  lazy execution 边界，并有回归测试证明 reusable query 的 cached-only dependency edge
  会被 replay，而不是被重新执行 provider 后覆盖。现在同样的 subject-first /
  provider-lazy 模式已经覆盖 driver source-stage collection，`FileContent` /
  `LexFile` / `ParseFile` rows 会真实落盘；对“注释/空白变化”的预期也已调整为：
  raw `FileContent` 会重算，但 token/parse result 不变时会通过 current-record green
  cut 继续复用，lossy sema/body/lowering query 也继续复用。开启 experimental pruning 的
  `--check` cache lookup 现在还会在 compile 前识别这类 source-stage green case，直接复用
  旧 semantic-ok cache。
- 现在已经补上 query pruning provider-skip 门禁脚本 `tools/query_pruning_gate.py`，
  它现在覆盖三类场景：注释级 cache-bypass 改动下总 query 数为 282，其中
  `FileContent` 直接 changed/evaluate，其余 281 个 query 全部 seed 复用；只改 `main`
  函数体时 source-stage 3 个 query、`function_body_syntax`、`type_check_body` 和
  `lower_function_ir` recompute/evaluate，其余 diagnostics、`ModuleGraph`、`ItemList`、
  `ModuleExports`、item/generic template/generic instance signature 和未变 lowering query
  复用；手动扰动 cached `generic_instance_signature` result 时只有 raw `FileContent`
  和 generic instance signature provider 运行，同时 `LexFile` / `ParseFile`、`ModuleGraph`、
  `ItemList`、`GenericTemplateSignature`、diagnostics 和其余 `lower_function_ir` 全部 seed
  复用。和只看总量相比，这条门禁能确认 reusable query 真的没有执行 provider，并且
  dependency-propagated recompute 分支会被 current-record green cut 截断。
- 这条门禁现在还会解析每个场景最终写出的 `query_edges` / `query_edge` 行，确认 header
  计数、dependency schedule、expected dependency kind 和 stable identity 关系全部一致。
  它把“provider 是否被 skip”和“skip 后 replay 出来的依赖图是否仍满足合同”绑定在同一个
  工业门禁里。
- 当前 green cut 是基于 cache write 前已经采集好的 current query record 做剪枝；
  它足够支撑 M2.5 的 provider-skip gate，但还不是完整的 demand-driven query engine。
  后续要把当前 eager subject fingerprint 逐步下沉到真正按需 provider 中，由 provider
  重算后比较 result、标绿，并复用到 IDE/LSP 的局部增量工作流。

### M2.5 第一批 query-key 完成定义

截至本设计对应实现，M2.5 第一批 query-key 接入已经完成 100%：`FileContent`、`LexFile`、
`ParseFile`、`ModuleGraph`、`ItemList`、`ModuleExports`、`ItemSignature`、
`FunctionBodySyntax`、`TypeCheckBody`、`GenericTemplateSignature`、
`GenericInstanceSignature`、`GenericInstanceBody`、`LowerFunctionIR` 和 `Diagnostics`
都已有 typed input、provider、`QueryContext` 入口、cache row emission、dependency edge、
profile detail、query pruning gate 和回归测试覆盖。

这个 100% 不表示完整 demand-driven engine、lossless CST、IDE/LSP 局部增量或并行 query
已经完成。它表示第一批 query key 的 identity、result fingerprint、provider-skip replay
和 red-green graph 落盘边界已经闭环，后续高级特性必须在这套边界上继续推进。

## 高级特性准入顺序

高级特性必须服从 query-key 主线。推荐顺序：

1. `CanonicalTypeKey` 和 `GenericInstanceKey`。
2. package visibility / module fragments。
3. 静态 trait / protocol。
4. move / copy / drop resource skeleton。
5. 最小 `Vec[T]`、`Box[T]`、`Result[T,E]`、`Option[T]`、`BufferWriter` 约束。
6. closure / lambda。
7. iterator protocol。
8. `pub use` / re-export。
9. named/default arguments。
10. derive/default impl。

任何能力如果会影响 stable key、module graph、diagnostics protocol 或 checked/IR 分层，
必须先更新本设计或对应专题设计，再进入实现。

## 验收清单

- 每个 query 都有 typed key、输入、输出、依赖、result fingerprint 和失效边界。
- 每条 dependency edge 都通过 provider expected-kind、schedule、stable identity、闭包和
  去重校验。
- 所有持久 key 都能 stable serialize / stable hash / debug print / equality。
- `CanonicalTypeKey` 不保存 `TypeHandle.value`。
- `GenericInstanceKey` 不依赖 display string 或 ABI suffix。
- diagnostics query 输出结构化事件，message 只是渲染层。
- incremental cache 能写入 query-key-compatible rows。
- 改函数体不改签名时，signature fingerprint 不变。
- 改 private item 不改 public exports 时，module export fingerprint 不变。
- 现有 coarse cache 行为在迁移期保持兼容。
- 新增 key 模块有单元测试覆盖稳定 hash、序列化、相等性、canonical type 和 generic
  instance 行为。
