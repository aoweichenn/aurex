# M2.5 收尾重构详细设计

## 1. 目标

M2.5 的最后一段工作不是引入新的语言能力，而是把已经完成的前端基础设施收口成一个更清晰、更可插拔、更低耦合的工程骨架。这个阶段只处理现有代码的结构债务，核心目标有四个：

1. 把跨模块的公共能力抽出来，停止在 `cli.cpp`、`compiler.cpp`、`module_loader.cpp`、`incremental_cache.cpp`、`sema.cpp` 里重复长出同类逻辑。
2. 把“编排”和“具体实现”分开，让每个文件只承担一类稳定职责。
3. 保留现有的高性能路径，不引入会在热路径上制造额外分配、复制、虚调用或大对象搬运的抽象。
4. 为后续 M3 模块系统和泛型闭环预留清晰边界，但不提前实现 RAII、trait、closure 或其他 M3 之后的内容。

这个文档只描述 M2.5 收尾重构，不改变 M3 路线图的目标和边界。

## R5 driver / compilation pipeline 收口状态

M2.5 的 diagnostic、module loader、incremental cache、query 和 sema 收尾拆分完成后，最高优先级曾切到
[R5 Compilation Pipeline / Driver Action 重构](r5-compilation-pipeline-refactor.md)。R5 core 现在已经完成；当前最高优先级进入
[M3 模块系统](m3-roadmap.md)。R5 这条线参考
Clang / LLVM 的现代编译器分层，但保持 Aurex 当前轻量 C++20 实现：

- `CompilerInvocation` 只保留纯配置。
- `Compiler` 退回 public facade。
- 一次编译的 `SourceManager`、`DiagnosticSink`、`CompilationProfiler`、cache policy 和 backend emitter
  进入内部 `CompilationSession`。
- source/token/lossless、module/frontend、sema、incremental cache write、IR lowering、LLVM/native backend
  由 `CompilationPipeline` 显式编排。
- diagnostics 渲染继续只走 `diagnostic_renderer`，profile phase 名称、diagnostics JSON/text 协议、
  incremental cache reuse/write 行为和 emit mode 行为保持不变。

这一步是继续现代化重构的主线，不是 M3 语言功能实现。R5 完成后，M3 module part 和
query-backed generics 必须消费这条 pipeline/session/query 主路径，不能另开旁路。

## 2. 现有代码的主要问题

### 2.1 驱动层职责过宽

`src/driver/compiler.cpp` 现在仍然承担完整编译流水线编排，但它也直接依赖诊断渲染结果。`compiler.cpp` 本质上应该只做：

- 构造编译上下文。
- 串起读取、词法、解析、模块加载、语义分析、缓存和后端。
- 根据结果决定停止点。

它不应该知道诊断怎么排版，也不应该负责任何输出协议细节。

`src/driver/cli.cpp` 的问题更明显。这个文件同时包含：

- 命令行选项表。
- 参数解析器。
- 帮助文本生成。
- CLI 退出码策略。
- 失败时的 JSON driver error 输出。

它不是单纯的入口文件，而是“解析器 + 协议适配器 + 文本生成器”的混合体。结果就是 CLI 的错误输出协议和编译器的诊断协议开始分叉。

### 2.2 诊断协议被重复实现

`src/driver/cli.cpp` 里有一套 JSON driver error 打印逻辑，`src/driver/diagnostic_renderer.cpp` 里又有一套诊断事件渲染逻辑。两者都在做：

- JSON 字符串转义。
- `aurex-diagnostics-v1` 信封格式。
- 严格的字段顺序控制。

这会导致两个风险：

- 一旦输出格式改动，就必须同步修改两处。
- 一旦测试覆盖不够，CLI 的错误输出和编译器诊断可能悄悄不一致。

这类问题不算算法复杂，但它会持续侵蚀工程质量分数，因为它直接制造“协议漂移”。

### 2.3 大文件里混着多种语义层级

下面这些文件都已经明显超出“单一职责文件”的舒适区：

- `src/driver/module_loader.cpp`
- `src/driver/incremental_cache.cpp`
- `src/query/query_context.cpp`
- `src/sema/sema.cpp`

这些文件的问题不只是“行数多”，而是内部同时混了不同层级的事：

- 路径解析、文件 IO、模块身份校验、AST 组合、诊断上报。
- 缓存 schema、query 记录、依赖图、序列化、解析、回放、性能统计。
- provider 注册、query 图维护、依赖检查、评估入口、失效传播。
- 语义注册、名称解析、类型检查、泛型、pattern、控制流、diagnostics。

这意味着后续改任何一点，都容易踩到另三点。

### 2.4 公共工具还没有稳定抽层

当前代码里有不少“看上去像工具，实际上属于公共基础设施”的东西，但它们还散落在业务文件里，例如：

- JSON 字符串转义。
- 诊断格式选择。
- 兼容性的错误包裹。
- 模块路径候选构建。
- 一些稳定的字段名、协议常量和格式版本号。

这些东西如果不尽早收口，后面拆模块时会继续被复制。

## 3. 设计原则

### 3.1 只做一层责任

每个模块只解决一类问题：

- driver 负责流程调度，不负责格式细节。
- diagnostics 负责输出协议，不负责业务判断。
- module loader 负责模块加载语义，不负责编译整体流程。
- incremental cache 负责缓存读写，不负责语义检查。
- sema 负责语义，不负责 CLI/driver 协议。
- query 负责依赖图和 query 生命周期，不负责业务层编排。

### 3.2 共享能力必须显式化

如果多个文件都在用同一种能力，它必须进入公共模块，而不是让每个文件各写一遍。最优先抽的就是：

- 诊断渲染。
- JSON 转义。
- driver error 输出。
- 路径/模块候选处理。
- 缓存与 query 的通用记录结构。

### 3.3 保持热路径轻量

这个项目是编译器，不是 GUI 框架。重构不能引入这些负担：

- 热路径上的额外字符串拼接。
- 不必要的 `std::function` 闭包。
- 不必要的中间对象复制。
- 在 sema / query / driver 热路径里为了“抽象好看”而增加动态分发。

能直接写流就直接写流，能用局部小函数就不用虚接口。

### 3.4 目录要表达边界

目录不是装文件的桶，而是表达边界的地图。目标是让人一眼知道：

- 哪些是公共基础设施。
- 哪些是 driver 入口。
- 哪些是 query / sema / syntax 的内部实现。

目录里不应该长期堆放几十个无分组文件。

## 4. 目标结构

下面是建议的收敛方向，不要求一次全做完，但必须按这个方向切。

### 4.1 `src/driver/diagnostics/`

放所有输出协议相关逻辑：

- text diagnostics 渲染。
- JSON diagnostics 渲染。
- driver error 渲染。
- 共享 JSON 转义和 envelope 生成。

对外只暴露少量稳定 API：

- `render_diagnostics(...)`
- `render_driver_error(...)`

`cli.cpp` 和 `compiler.cpp` 只调用这些 API，不再自己拼 JSON。

### 4.2 `src/driver/module/`

把模块加载拆成几块：

- 路径解析与候选构建。
- import 文件解析与校验。
- AST remap / merge。
- 模块记录和模块身份校验。

`ModuleLoader` 可以继续作为总入口，但内部应该变薄。

### 4.3 `src/driver/cache/`

把 incremental cache 拆成：

- schema / record / fingerprint。
- 文件内容快照。
- query graph 和 pruning 记录。
- 读写与落盘。

这个目录里的文件不要再同时承担“格式定义”和“业务流程”。

### 4.4 `src/query/core/` 与 `src/query/providers/`

如果后续继续拆 query，推荐把它分成：

- 核心 query 图和节点管理。
- provider 注册与适配。
- 具体 provider 实现。

这样 `query_context.cpp` 不会一直膨胀成一个“全局大脑”。

### 4.5 `src/sema/phase/` 或按职责拆分

sema 最终应拆成这些自然边界：

- 注册与命名。
- 可见性与模块规则。
- 类型与布局。
- 泛型模板与实例化。
- 表达式/语句/块检查。
- pattern 与 exhaustiveness。
- 函数体分析。

如果现在就要切，建议按 `register / resolve / check / generic / pattern / function-body` 这几个小目标分步拆，而不是一次重写。

## 5. 推荐的第一阶段拆分顺序

### 第一步：统一诊断渲染

先做已经开始的那一刀，并把它收口完成：

- `diagnostic_renderer` 变成唯一的输出协议层。
- `cli.cpp` 不再直接打印 JSON driver error。
- `compiler.cpp` 只调用 renderer，不拼协议。
- 测试只检查渲染结果，不检查内部实现细节。

这是低风险、高收益的一步，因为它能直接减少重复逻辑和格式漂移。

### 第二步：压缩 `cli.cpp`

把 CLI 文件里和“解析命令行”无关的逻辑继续往外搬：

- 错误输出协议移走。
- 如果后面还会长出更多子命令，把子命令分发表单独收纳。

目标是让 `cli.cpp` 只剩下命令解析和入口调度。

### 第三步：拆 `module_loader.cpp`

建议拆成两个聚合度清晰的私有单元，而不是把每个小动作都拆成文件：

- `module_loader_support.cpp`：路径/canonicalization、import 候选、文件读取、lex/parse、
  模块诊断和身份校验。它是加载主流程的私有 support 聚合点，不作为新的公共 API。
- `module_loader_remap.cpp`：IdMap 与 AST remap / append / root fast-path merge。这部分是独立算法域，
  体量和调用方向都适合单独放置。

`module_loader.cpp` 本体保留 `ModuleLoader` 的状态机与编排：加载中集合、已加载文件缓存、
模块名登记、import 递归和最终 append 阶段。这样既避免一个文件继续吃下所有细节，也避免拆成
过多低聚合度的小文件。

### 第四步：拆 `incremental_cache.cpp`

这是最大的一块，必须谨慎切：

- 先把协议常量和 record 结构挪到独立头/源对。
- 再把 I/O 读写、query graph 导出、pruning 统计分开。
- 最后再收回主调度逻辑。

不要一口气动太多，否则风险会指数上升。

### 第五步：sema 与 query 内部收边

当 driver 层干净以后，再把 sema / query 里面的职责边界收紧：

- query 只做依赖与缓存。
- sema 只做语义。
- 任何为了查询而临时拼出来的协议对象都尽量显式化。

sema 的第一刀不是继续把表达式、语句或泛型按小功能点切成更多文件，而是先收缩 public
facade：`include/aurex/frontend/sema/sema.hpp` 只保留 `SemanticOptions` 和 `SemanticAnalyzer`
稳定入口，旧 analyzer 的内部类型、lookup cache、泛型模板信息、表达式 view、pattern
helper 和 phase helper 先移动到 `src/sema/internal/sema_core.hpp`。这样外部入口先变少，
内部算法仍保持原有聚合度，然后再按状态 owner 拆 `NameState`、`TypeState`、`GenericState`、
`FunctionState`、`ModuleState` 和 `FlowState`，避免在状态边界还没清楚时把文件拆得太碎。
当前这组 owner 已经落地，并进一步收进 `SemaState`：name/lookup 索引、类型与 alias 辅助表、
泛型模板/实例缓存、函数/值/函数体状态、模块可见性缓存、当前分析上下文分别有唯一归属。
`SemanticAnalyzerCore` 现在只持有一个 `state_` 聚合入口，白盒测试也通过 `state_.names`、
`state_.types`、`state_.generics` 等路径访问 owner。实现仍放在同一个 internal core 聚合点，
不新增深层目录，也不拆 expr/stmt/generic 算法文件。
随后补上 `SemaContext`：AST module、diagnostic sink 和 semantic options 作为外部会话依赖
统一从 context 进入 core，`SemanticAnalyzerCore` 不再平铺 `module_`、`diagnostics_`、
`options_` 三个字段。这样后续 domain service 可以接收 `SemaContext&` 加必要 state owner，
而不是重新依赖整只 analyzer。
`SemanticAnalyzerCore::analyze()` 也已收成浅层 phase pipeline：session prepare、storage reserve、
declaration phases、function body phases、validation phases 和 finish 分开命名；phase 方法仍留在
`sema.cpp` 这个聚合点，不为每个小阶段新建目录或文件。

## 6. 代码质量评审标准

这次重构的质量门，不只是“能编过”，而是要满足下面这些条件：

1. 每个新增或修改的文件都能用一句话说清楚职责。
2. 任何一个输出协议只允许有一个权威实现。
3. 热路径上不引入明显额外分配或复制。
4. 大文件拆分后，调用方向变得单向清晰，而不是互相拉扯。
5. 新增或重构函数的参数数不超过 6 个；超过时要改用具名 context/value object 或重新切分职责。
6. 测试能覆盖正路径、错误路径和格式边界。
7. 文档能直接回答“为什么这么拆”和“下一刀往哪拆”。

如果这几个条件不满足，就不算完成收尾。

## 7. 本轮已经落地的重构

当前代码已经开始做结构性收口，并且每一步都保持 M2.5 范围内的行为不变：

- `src/driver/diagnostic_renderer.cpp` 已经成为独立的诊断输出模块。
- `compiler.cpp` 的诊断输出调用已经回到公共渲染层。
- `cli.cpp` 的 JSON driver error 逻辑已经并入同一层，不再单独维护第二套转义与 envelope 代码。
- `src/driver/module_loader_remap.cpp` 已经承接 AST ID remap / append / root fast-path merge，
  `module_loader_support.cpp` 聚合承接路径/canonicalization、import 候选、文件读取、
  lex/parse、模块诊断和身份校验；`module_loader.cpp` 现在只保留 `ModuleLoader`
  状态机、模块登记、import 递归和 append 编排。加载中集合的清理也改为局部 RAII scope，
  不再在每个错误分支手动 erase。
- `src/driver/incremental_cache.cpp` 已经降为 public facade，只负责把
  `try_reuse_incremental_check_cache` / `write_incremental_cache` 转发到私有实现层。
- `src/driver/incremental_cache/` 已经承接 incremental cache 的内部模块：
  `format.hpp` 固化 wire format 常量与 parsed record，`io.hpp` 固定内部 I/O API，
  `io.cpp` 聚合负责路径规范化、source fingerprint、module record 排序、cache parser /
  validator、cache row 写出和原子发布，保持 wire format 的读写权威入口不再分叉，
  `query.*` 负责顶层 check/write 编排，`subjects.*` 负责从当前编译产物生成 query subjects，
  `reuse.*` 负责旧 cache query plan 与 pruning gate，`source_stage.*` 负责 source-stage
  快速复用判断，`schedule.*` 负责 query schedule 顺序校验，`profile.*` 负责 profiler 文本，
  `query_stats.*` 负责 per-kind 计数。
- `src/driver/incremental_cache/subjects/` 进一步把 query subject 构造拆成 source、module、
  semantic、ordering 四个内部域：source 域只负责 file/lex/parse fingerprints，module 域只负责
  module graph / exports / item-list signatures，semantic 域只负责 checked module 中的
  item/body/generic/IR subjects 与 definition records，ordering 域只负责 query record
  materialization、diagnostics subjects 和 schedule 排序后的执行分派。
- driver / backend 的 public header 已开始按 facade 边界瘦身：
  `EmitKind`、`DiagnosticOutputFormat`、`ModuleRecord`、`LlvmIrEmitter` 和
  `OptimizationLevel` 拆成窄头文件；`compiler.hpp` 不再暴露 `ir.hpp`，
  `incremental_cache.hpp` 不再包含 `module_loader.hpp` / `checked_module.hpp` / `ast.hpp`，
  `diagnostic_renderer.hpp` 和 `native_toolchain.hpp` 也不再为了一个 enum 依赖完整
  `CompilerInvocation`。这一步不改变运行时行为，只把公共接口上的横向依赖压回具体实现文件。
- query 层也拆出 `query_graph.hpp` / `query_graph.cpp`，承接 `QueryNode`、
  `QueryEvaluationResult`、`QueryDependencyEdge` 和 dependency-kind 规则；
  `query_edge_verifier.hpp`、`query_replay.hpp`、`query_reuse.hpp` 不再因为图结构 API
  直接依赖 provider-heavy 的 `query_context.hpp`。
- query 层继续拆出 `query_provider_set.hpp` / `query_provider_set.cpp`，把 provider
  类型别名、默认 provider wiring 和自定义 provider 回退策略收纳成独立 Strategy 集合；
  `QueryContext` 只保留 query 生命周期、依赖图维护、缓存/失效协调和 provider set 转发，
  不再直接持有十五个 provider 成员或重复组装默认 provider。
- query provider 的外部 wiring 继续收成 `QueryProviderOverrides` 聚合对象：
  `QueryProviderSet` 和 `QueryContext` 不再提供 15 参数构造器，短构造仍兼容常见测试/注入路径，
  扩展 M3 query 时只需要在 overrides 聚合里增加一个 owner 字段，而不是扩散一串 public
  参数列表。
- parser postfix 的 `[]` disambiguation 也按聚合度收口：新增
  `src/parse/bracket_suffix_classifier.hpp` / `src/parse/bracket_suffix_classifier.cpp`，
  只负责 index / slice / generic apply 的分类决策；`parser_postfix.cpp` 继续拥有参数解析、
  recovery 和 AST 构造，避免把表达式后缀拆成一堆小目录或小文件。
- `query_executor.hpp` 也不再包含完整 `query_context.hpp`，只依赖 query graph 结果类型和
  provider input/output 形状；executor 实现文件才回到完整 context，避免使用 batch request
  API 的调用者被动拉入 query 状态图实现。
- sema public facade 已完成第一阶段收缩：`include/aurex/frontend/sema/sema.hpp` 从约 800 行降到
  39 行，只暴露 `SemanticOptions` 和 `SemanticAnalyzer`，`SemanticAnalyzer` 通过私有
  `Impl` 持有内部 core，不再把 analyzer 私有状态、泛型 side table helper、表达式 view、
  lookup key helper 或语句/pattern 分析 helper 暴露给所有 public include 用户。
- 原来的 analyzer 实现收纳为 `src/sema/internal/sema_core.hpp` / `src/sema/sema_facade.cpp`：
  生产调用方继续只包含 `<aurex/frontend/sema/sema.hpp>`，sema 实现文件和内部测试显式包含
  `<sema/internal/sema_core.hpp>`。测试不再通过预处理器开关打开生产类型的私有区。
- 本轮建立 facade/core 边界后，继续完成状态 owner 聚合：
  `NameState` 统一拥有本地 symbol table、typed lookup index、函数/值/enum-case 名称索引；
  `TypeState` 拥有 named type、type alias resolution stack/cache 和 struct side table；
  `GenericState` 拥有模板表、实例缓存、param query key 和 placeholder function；
  `FunctionState` 拥有 global value、definition item 和 body state；
  `ModuleState` 拥有 visible/export module cache；`FlowState` 继续拥有当前分析上下文。sema
  算法仍保持不变，也不急着把 `SemanticAnalyzerCore` 拆成小服务。
- 这些 owner 已进一步由 `SemaState` 聚合，`SemanticAnalyzerCore` 不再直接持有
  `checked_`、`names_`、`types_`、`generics_`、`functions_`、`modules_` 和 `flow_`
  这一组平铺字段；core 只持有 `state_`，可变语义数据从 `state_.checked` /
  `state_.names` / `state_.types` 等窄路径进入，状态所有权更清楚。下一阶段再让
  NameResolver、TypeResolver、GenericEngine、FunctionChecker、ExprChecker、StmtChecker 和
  PatternChecker 通过窄接口拿这些 owner。
- sema 的第一块中等粒度 domain service 已落地为
  `src/sema/internal/name_resolution.hpp` / `src/sema/internal/name_resolution.cpp`：
  `ModuleVisibilityResolver` 统一承接 import alias 解析、visible/export module cache、
  模块路径匹配、public re-export 遍历和 module display name。`SemanticAnalyzerCore`
  保留原有窄方法作为薄 facade，`sema_lookup.cpp` 继续聚合 typed type/function/value/enum
  lookup 与 did-you-mean 逻辑，不把表达式或单个 lookup case 拆成碎片文件。这个 resolver
  只读访问 core 的 context/flow，并且只通过 `ModuleState` 的 mutable cache 写入可见性结果；
  public include 不暴露任何新内部类型。
- `SemaContext` 已补齐 context 层：`ctx_.module`、`ctx_.diagnostics`、`ctx_.options`
  统一承载外部输入和诊断 sink，core 不再直接平铺 `module_`、`diagnostics_`、`options_`。
  `FunctionRegistry` 仍保留自己的 `diagnostics_` 依赖，因为它是 registry 的构造依赖，不属于
  analyzer core 的外部会话状态。
- `SemanticAnalyzerCore::analyze()` 已压成 phase pipeline：`prepare_analysis_session`、
  `reserve_analysis_storage`、`run_declaration_phases`、`run_function_body_phases`、
  `run_validation_phases` 和 `finish_analysis` 只做编排命名，不把算法拆成一堆小文件。
- `FunctionBodyContextScope` 的 7 参数内部构造器已收束为具名 `Config`，继续执行新增或重构函数
  参数不超过 6 个的规则；后续如果遇到类似 `register_function` 这种历史长参数 API，应优先用
  request/config value object 聚合，而不是继续横向加参数。
- 历史长参数 API 已继续收口：`FunctionRegistry::register_function` 改为接收
  `FunctionRegistrationRequest`，函数注册时的 item、owner、semantic key、ABI 名、类型信息和
  incremental identity 被视为同一个语义请求；generic side table 的 sparse layout 也改为
  `GenericSideTableLocalLayoutView`，`configure_local_dense` / `append_generic_side_table_layout`
  不再暴露 8 个并列字段参数。这样既保持外部 API 聚合，也避免内部 helper 继续横向膨胀。

## 8. 不做的事

M2.5 收尾阶段明确不做这些内容：

- 不动 M3 模块系统详细设计。
- 不实现 RAII。
- 不引入 trait / closure / iterator / async。
- 不把泛型扩展成新的高级表达力专题。
- 不为了拆文件而牺牲热路径性能。

## 9. 验收标准

当这轮重构完成时，至少要满足：

- 诊断输出只有一个权威协议实现。
- `cli.cpp` 不再保存单独的 JSON driver error 格式器。
- `compiler.cpp` 只做编排，不做输出格式细节。
- `module_loader.cpp`、`incremental_cache.cpp`、`sema.cpp`、`query_context.cpp` 的职责边界都比当前更清楚。
- 新增文件的目录结构能直接反映模块边界。
- 相关测试通过，并且没有引入明显性能回退。

## 10. 结论

M2.5 的收尾不是“大重写”，而是把已经长出来的工程能力重新归位。先把输出协议和 driver 边界收紧，再逐步拆 module loader、incremental cache、query 和 sema，最后形成一套可继续承接 M3 的松散结构。

这份设计的核心判断很简单：

**越是接近编译器主路径的代码，越应该少一点机灵，多一点清晰。**
