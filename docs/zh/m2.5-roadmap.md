# M2.5 路线图

## 阶段定位

M2.1 已经把语义安全、主要性能爆点、诊断预算和当前 no-std 语言核心收口到可继续演进的基线。M2.5 不再继续补旧 std 路线，也不急着扩大语言面，而是把前端从“顺序编译器”推进到“可查询、可复用、可被 IDE 消费”的工程骨架。

M2.5 的先后顺序是：

1. 先把现有语义结果变成稳定、显式、可缓存的结构化数据。
2. 再把 parse / module graph / signature / body / generic / diagnostics 拆成 query。
3. 最后引入 lossless syntax、增量解析和 LSP-native 消费路径。

这样做的约束很明确：不能为了未来 query 化继续保留依赖文案、display string 或隐式上下文的元数据。

## 当前工作流

| 项目 | 状态 | 说明 |
|:-----|:-----|:-----|
| 1. 语义诊断显式元数据 | 已完成 | sema 在创建诊断时携带稳定 kind/category/code，不再从 message 文本反推分类 |
| 2. Query 输入与 stable key 梳理 | 待处理 | 统一文件、模块、定义、函数体、泛型实例、诊断的 query key |
| 3. QueryContext / 依赖追踪 | 待处理 | 先接 file parse、module graph、item signature，再迁函数体和泛型 |
| 4. Lossless syntax 基线 | 待处理 | 引入保留 trivia 的 CST / GreenTree，AST 从 CST lowering |
| 5. IDE-native 语义入口 | 待处理 | 基于 query 提供 diagnostics、hover、definition、references 等消费路径 |

## 为什么第 1 项先做

诊断已经有 `DiagnosticCategory` 和 `DiagnosticCode`，但如果 sema 仍靠 message 前缀或片段去推断它们，后续会同时伤到三件事：

- 文案调整会改变机器协议，CLI、JSON、LSP 无法稳定复用。
- diagnostics query 只能缓存“字符串”，不能缓存明确的语义事件。
- 本地化、IDE 展示和 future fix-it 都会被迫复制同一套脆弱规则。

因此 M2.5 的第一步不是新增更多诊断，而是固定“诊断是什么”与“诊断怎么说”之间的边界：kind/category/code 是结构化语义，message 只是展示文本。

## 目标架构

### A. Stable-ID-driven Query

- query 以稳定定义身份、文件指纹、模块身份和局部 body identity 为键。
- parse、module graph、item signature、function body typecheck、generic instance、match check、diagnostics 都是独立 query。
- query 结果必须可判定失效，不能靠 display name、临时 NodeId 或输出字符串决定复用。

### B. Lossless Syntax + AST Lowering

- parser 先生成保留空白、注释和 token 的 lossless CST。
- 语义 AST 由 CST 降低得到，继续保留当前 typed AST / checked overlay 优势。
- syntax node 需要 stable identity，支持修改局部后只重建受影响范围。

### C. IDE-native Diagnostics

- CLI 文本输出、`--diagnostics=json`、未来 LSP 都消费同一诊断事件流。
- diagnostics query 的主键是语义事件，不是最终字符串。
- 顺序、去重、note/help 关联和 future fix-it 都必须是结构化字段。

## 非目标

- 不在 M2.5 重新设计标准库。
- 不把完整 borrow checker、resource traits、async 或 macro 提前塞进 query 重构。
- 不为了“看起来增量”保留第二套并行前端。
- 不把 LSP protocol 适配层当成 query / syntax 架构本身。

## 验收

1. 诊断元数据不再由 message 文本派生；改文案不改 category/code。
2. 每个 query 都有明确输入、输出、stable key、依赖和失效边界。
3. 语法树能保留 trivia，并能在局部编辑后只重算受影响 parse / sema 区域。
4. CLI、JSON、LSP 复用同一 diagnostics 数据源，且输出顺序确定。
5. 现有 stress / coverage / regression 质量门继续保留，不以“架构迁移”为由放松。
