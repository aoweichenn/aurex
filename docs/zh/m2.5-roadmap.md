# M2.5 路线图

## 阶段定位

M2.1 已经把语义安全、主要性能爆点、诊断预算和当前 no-std 语言核心收口到可继续演进的基线。M2.5 不再继续补旧 std 路线，也不急着扩大语言面，而是把前端从“顺序编译器”推进到“可查询、可复用、可被 IDE 消费”的工程骨架。

M2.5 的先后顺序是：

1. 先冻结 query key 设计，并把现有语义结果变成稳定、显式、可缓存的结构化数据。
2. 再把 parse / module graph / signature / body / generic / diagnostics 拆成 query。
3. 最后引入 lossless syntax、增量解析和 LSP-native 消费路径。

这样做的约束很明确：不能为了未来 query 化继续保留依赖文案、display string 或隐式上下文的元数据。

## 和 regex 表达力审计的对照

`examples/libs/regex` 暴露出另一个事实：M2.1 的语言核心已经能写系统库，但缺少足够的抽象和资源语义，导致 text/bytes 双路径、手工内存管理、callback、iterator、buffer writer 和跨模块私有状态都要靠样板代码维持。这个问题不能简单塞进旧 std 路线，也不能抢掉 M2.5 的 query / lossless / IDE 主线。

因此 M2.5 现在有两条层级不同的轨道：

| 轨道 | 原有 M2.5 定位 | regex 审计后的调整 |
|:-----|:---------------|:-------------------|
| 前端基础设施主线 | query key、依赖追踪、lossless syntax、IDE-native diagnostics | 仍是 M2.5 的主线和验收核心，不能被 std 或高层语法扩张替代 |
| 语言/库表达力补强 | 原路线明确“不急着扩大语言面” | 作为 M2.5 的设计冻结和准入门槛加入：只接收能消除系统库重复、修复资源不变量或支撑 query 边界的 P0/P1 能力 |
| 标准库重建 | 原路线明确非目标 | 仍然非目标；`Vec` / `Box` / `Result` / writer 等只写成语言约束和最小库形状，不恢复旧 std |
| resource / trait / closure | 原路线要求不把完整资源语义、async、macro 塞进 query 重构 | 保留这个约束；但 resource、trait、closure 必须在 M2.5 留下可实现的语义设计，否则 regex 级系统库无法优雅落地 |

换句话说：**query/lossless/IDE 是 M2.5 的工程主线；regex P0/P1 是 M2.5 后半段和 M2.6 之前必须冻结的语言能力边界。** 如果某项表达力改动会污染 stable key、query invalidation、diagnostics protocol 或 module visibility，就必须先按前端基础设施规则设计清楚，再进入实现。

## 为什么第 1 项先做

诊断已经有 `DiagnosticCategory` 和 `DiagnosticCode`，但如果 sema 仍靠 message 前缀或片段去推断它们，后续会同时伤到三件事：

- 文案调整会改变机器协议，CLI、JSON、LSP 无法稳定复用。
- diagnostics query 只能缓存“字符串”，不能缓存明确的语义事件。
- 本地化、IDE 展示和 future fix-it 都会被迫复制同一套脆弱规则。

因此 M2.5 的第一步不是新增更多诊断，而是固定“诊断是什么”与“诊断怎么说”之间的边界：kind/category/code 是结构化语义，message 只是展示文本。

## 已完成基础设施：Query Key

M2.5 第一批 query-key 基础设施已经完成。设计原则采用
Stable Semantic Query Key + Session Fast Handle + Red/Green Fingerprint QueryContext：

- query key 表示“要计算什么”，fingerprint 表示“输入或结果是否变化”。
- `TypeHandle`、`ExprId`、`StmtId`、vector index 和 source offset 只允许作为当前
  session 内快速 handle，不能作为持久 query key。
- `CanonicalTypeKey` 和 `GenericInstanceKey` 必须先落地，再推进 trait/resource、
  closure、iterator、module fragments 等高级特性。
- diagnostics 必须成为 query 产出的结构化事件，CLI、JSON 和未来 LSP 只做渲染。

完整设计和落地状态见 [M2.5 Query Key 设计](m2.5-query-key-design.md)。后续不再把
query key 当作待补基础项；新的 lossless、IDE 或高级语言能力必须复用这条默认主路径，
并在进入实现前声明自己的 stable identity、query invalidation、diagnostics protocol
和 module identity 影响面。

## 目标架构

### A. Stable-ID-driven Query

- query 以稳定定义身份、文件指纹、模块身份和局部 body identity 为键。
- parse、module graph、item signature、function body typecheck、generic instance、match check、diagnostics 都是独立 query。
- query 结果必须可判定失效，不能靠 display name、临时 NodeId 或输出字符串决定复用。

### B. Lossless Syntax + AST Lowering

- parser 先生成保留空白、注释和 token 的 lossless CST。
- 语义 AST 由 CST 降低得到，继续保留当前 typed AST / checked overlay 优势。
- syntax node 需要 stable identity，支持修改局部后只重建受影响范围。

### C. IDE-native Diagnostics

- CLI 文本输出、`--diagnostics=json`、未来 LSP 都消费同一诊断事件流。
- diagnostics query 的主键是语义事件，不是最终字符串。
- 顺序、去重、note/help 关联和 future fix-it 都必须是结构化字段。

### D. Language / Library Expressiveness Gate

M2.5 不把 regex 库问题误判成“重建 std”，但必须把它们转化为语言和前端架构的约束：

- 新能力必须有稳定语义身份，不能靠 display string、模块私有约定或 C ABI 名字参与类型/实例/key 判断。
- 任何资源能力都必须先定义 move/copy/drop 的 checked 语义、IR lowering 不变量和诊断边界，不能只靠 `defer destroy(...)` 约定。
- 任何 trait/protocol 能力都必须服务于静态分派、generic body checking、where clause 诊断和 query invalidation；不能先引入动态对象或隐式运行时查找。
- 模块可见性必须支持库内部拆分而不泄露 public ABI；这和 lossless/query 的 stable module identity 同步设计。

## 表达力补强 P0

这些项目来自 regex 库审计，是 M2.5 需要冻结设计或开始实现的最高优先级。它们和原 M2.5 的关系是“支撑后续系统库与 query 边界”，不是替代 query/lossless 主线。

| P0 项 | regex 暴露的问题 | 和原 M2.5 的对比/约束 | M2.5 交付边界 |
|:------|:------------------|:----------------------|:--------------|
| 泛型后端闭环 | `alloc_zeroed[T]` 仍要显式传 `sizeof[T]`，checked 可过但完整 lowering/backend 不稳 | 原 M2.5 已要求 stable generic instance key；这里补齐 generic body lowering，避免 checked/IR 语义分裂 | `sizeof[T]` / `alignof[T]` 在 generic 函数体可完整 IR/LLVM lowering；method-local generics 给出 parser/sema/ABI 设计；最小 const generics 写入设计但可分阶段实现 |
| package visibility / module fragments | `vm/engine.ax` 近 4000 行，拆文件会被迫把内部 helper 和字段 `pub` 化 | 原 M2.5 的 query key 需要稳定 module identity；这里补足“同一模块多文件”和 package-private visibility 的边界 | 设计 `pub(package regex)` / `pub(crate)` 或等价语义；设计 `module ... part ...` 或等价 module fragment；字段可被库内部访问但不进入 public API |
| 用户 trait / protocol 静态分派 | text/bytes、callback、input cursor、writer 都靠运行时 mode 或复制函数实现 | 原 M2.5 暂缓 trait 扩张；regex 证明 trait 是系统库去重的核心，不应继续无限后置 | 先做无 trait object 的静态 trait：methods、associated type、associated const、default method、`where T: Trait` body checking 和 instantiation diagnostics |
| 资源语义设计骨架 | `Regex`、`RegexSet`、`Captures`、`Workspace` 都靠手写 destroy 和 early-return cleanup | 原 M2.5 明确不塞完整 borrow checker；这里只冻结 move/copy/drop，不引入完整 lifetime/borrow checker | 设计 move-only struct、`Copy` capability、`Drop`/destructor 调用时机、partial move 禁止线、double-free 诊断和 IR drop lowering；实现可排到 query 主线后，但设计必须在 M2.5 固定 |
| 最小 owned containers / writer 形状 | 编译器数组扩容、workspace、replace/escape/database writer 都是样板代码 | 原 M2.5 不重建 std；这里只定义最小 no-std library contracts，避免旧 std 复活 | 给出 `Vec[T]`、`Box[T]`、`Result[T,E]`、`Option[T]`、`BufferWriter` 的语言依赖清单和 ABI/Drop 约束；实现依赖 P0 resource/generic 闭环 |

## 表达力补强 P1

P1 是高收益但不能压过 P0 的能力。它们进入 M2.5 文档，是为了防止后续继续用样板 API 扩张库表面。

| P1 项 | regex 暴露的问题 | 和原 M2.5 的对比/约束 | 准入条件 |
|:------|:------------------|:----------------------|:---------|
| closure / lambda | replacement callback 必须是顶层函数指针，不能捕获状态，text/bytes callback 分裂 | 原 M2.5 不做高层语言扩张；这里要求先从非捕获 lambda 开始，不引入 async/macro | 非捕获 lambda 可降为函数指针；捕获 lambda 设计成环境 struct + call method；`Fn` / `FnMut` / `FnOnce` 等 trait 依赖 P0 trait/resource |
| iterator protocol | `find_next`、`captures_next`、`split_next`、`stream_next` 全是手写状态同步 | 原 M2.5 已有 range-only `for`；这里只扩到 trait-backed iterator，不引入 generator 作为 P0 | 设计 `Iterator` associated `Item`、`next(&mut self) -> Option[Item]` 和 `for item in iter` lowering；generator/yield 后置 |
| `pub use` / re-export | `api.ax` 大量 type alias 和薄 wrapper | 不影响 query 主线，但必须和 module graph/export cache 同步 | 支持 `pub use module::{name}` 和 alias；export query 记录 re-export source，diagnostics/definition 可跳转 |
| named/default args | `replace_all`、`replace_first`、`replace_n`、append 版本组合爆炸 | 不能用 overload 破坏 lookup/key；只允许显式命名参数和编译期默认值 | 默认值进入 function signature stable key；调用点 lowering 后保持确定 ABI；禁止基于运行时 overload 选择 |
| safe slice / pointer boundary helper | 裸指针 indexing 让业务逻辑到处 `unsafe` | 原 M2.5 unsafe 边界已收紧；这里增加 safe slice-from-raw 的封装路径 | raw pointer 生成 slice 仍是 unsafe；slice indexing/copy/fill 可在 safe API 内统一 bounds/unchecked 策略 |
| derive / default impl | `RegexOptions`、结果结构和 enum status 有大量样板构造 | 只能用于已定义清楚的能力，不能偷偷生成资源行为 | 先支持 `Default` / `Copy` / `Eq` / `Debug` 设计；`Drop` 不 derive，保持手写 |

## 非目标

- 不在 M2.5 重新设计标准库。
- 不把完整 borrow checker、async 或 macro 提前塞进 query 重构。
- 不把完整 resource 系统提前实现为 M2.5 query 主线的阻塞项；M2.5 只冻结 move/copy/drop 语义设计和最小实现切面。
- 不引入 trait object、动态分派、完整 closure capture ABI、generator、macro 或 package manager 作为 M2.5 P0。
- 不为了“看起来增量”保留第二套并行前端。
- 不把 LSP protocol 适配层当成 query / syntax 架构本身。

## 验收

1. 诊断元数据不再由 message 文本派生；改文案不改 category/code。
2. 每个 query 都有明确输入、输出、stable key、依赖和失效边界。
3. 语法树能保留 trivia，并能在局部编辑后只重算受影响 parse / sema 区域。
4. CLI、JSON、LSP 复用同一 diagnostics 数据源，且输出顺序确定。
5. 现有 stress / coverage / regression 质量门继续保留，不以“架构迁移”为由放松。
6. 表达力 P0 的设计不破坏 query key、module graph、diagnostics protocol 或 checked/IR 分层；`sizeof[T]` generic lowering、package visibility/module fragments、静态 trait 和 move/copy/drop 至少有语义文档、negative/positive 测试计划和 IR/ABI 约束。
7. 表达力 P1 只有在 P0 约束可验证后进入实现；任何新增语法都必须说明它如何降低 regex 级系统库重复，而不是只改善表层写法。
