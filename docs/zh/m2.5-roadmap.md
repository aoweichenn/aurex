# M2.5 路线图

## 阶段定位

M2.1 已经把语义安全、主要性能爆点、诊断预算和当前 no-std 语言核心收口到可继续演进的基线。M2.5 的职责不是继续补旧标准库路线，也不是扩大语言表面；它只负责把前端从“顺序编译器”推进到“可查询、可复用、可被 IDE 消费”的工程骨架。

M2.5 的阶段边界现在固定为：

1. 冻结 query key 设计，把现有语义结果变成稳定、显式、可缓存的结构化数据。
2. 把 parse / module graph / signature / body / generic / diagnostics 接入 query 和依赖追踪。
3. 完成当前 lossless syntax、结构化诊断和 IDE-native snapshot 入口。
4. 保留 M3 所需的 stable identity 和 query 边界，但不在 M2.5 实现新的高级语言特性。

这样做的约束很明确：不能为了未来 query 化继续保留依赖文案、display string、临时 NodeId 或隐式上下文的语义元数据。

## 为什么第 1 项先做

诊断已经有 `DiagnosticCategory` 和 `DiagnosticCode`，但如果 sema 仍靠 message 前缀或片段去推断它们，后续会同时伤到三件事：

- 文案调整会改变机器协议，CLI、JSON、LSP 无法稳定复用。
- diagnostics query 只能缓存“字符串”，不能缓存明确的语义事件。
- 本地化、IDE 展示和 future fix-it 都会被迫复制同一套脆弱规则。

因此 M2.5 的第一步不是新增更多诊断，而是固定“诊断是什么”与“诊断怎么说”之间的边界：kind/category/code 是结构化语义，message 只是展示文本。

## 已完成基础设施：Query Key

M2.5 第一批 query-key 基础设施已经完成。设计原则采用 Stable Semantic Query Key + Session Fast Handle + Red/Green Fingerprint QueryContext：

- query key 表示“要计算什么”，fingerprint 表示“输入或结果是否变化”。
- `TypeHandle`、`ExprId`、`StmtId`、vector index 和 source offset 只允许作为当前 session 内快速 handle，不能作为持久 query key。
- `CanonicalTypeKey`、`GenericInstanceKey`、`ModuleKey` 和 `ModulePartKey` 是 M3 模块/泛型工作的前置基础，不再在 M2.5 继续扩展成 trait、resource、closure 或 iterator 语义。
- diagnostics 必须成为 query 产出的结构化事件，CLI、JSON 和未来 LSP 只做渲染。

完整设计和落地状态见 [M2.5 Query Key 设计](m2.5-query-key-design.md)。后续不再把 query key 当作待补基础项；新的 M3 模块和泛型能力必须复用这条默认主路径，并在进入实现前声明自己的 stable identity、query invalidation、diagnostics protocol 和 module identity 影响面。

## 目标架构

### A. Stable-ID-driven Query

- query 以稳定定义身份、文件指纹、模块身份和局部 body identity 为键。
- parse、module graph、item signature、function body typecheck、generic instance、match check、diagnostics 都是独立 query。
- query 结果必须可判定失效，不能靠 display name、临时 NodeId 或输出字符串决定复用。

### B. Lossless Syntax + AST Lowering

- parser 先生成保留空白、注释和 token 的 lossless CST。
- 语义 AST 可从 lossless CST facade 降低到现有 parser，继续保留当前 typed AST / checked overlay 优势。
- syntax node 已有 kind/range/token span/depth 组成的稳定身份；IDE snapshot 已用该身份做 offset 查询和编辑影响 node 选择，完整局部 subtree reparse 继续作为后续优化推进。

### C. IDE-native Diagnostics

- CLI 文本输出、`--diagnostics=json`、未来 LSP 都消费同一诊断事件流。
- diagnostics query 的主键是语义事件，不是最终字符串。
- 顺序、去重、note/help 关联和 future fix-it 都必须是结构化字段。

### D. M3 Handoff

M2.5 到 M3 的交接点不是“开始做所有高级特性”，而是：

- 模块系统从文件级拼接模型升级到逻辑模块和 module part 模型。
- 泛型从当前单态化路径升级到 query-backed template signature、instance signature 和 instance body。
- 泛型 ABI、generic instance identity 和 module identity 必须稳定，不能依赖 display string 或 session-only handle。

M3 的详细阶段见 [M3 路线图](m3-roadmap.md)。

## 后置能力

下面能力不再属于 M2.5 验收，也不作为 M3 模块/泛型阶段的阻塞项：

- 用户 trait / protocol、associated type、associated const 和 trait object。
- `Copy` / `Drop`、move-only struct、destructor lowering、borrow checker、lifetime 和 RAII。
- closure / lambda 捕获、`Fn` / `FnMut` / `FnOnce`、generator 和 async。
- 通用 iterator protocol、`for item in container`、derive/default impl 和复杂 named/default args。
- 标准库重建、package manager、版本解析和依赖求解。

这些能力可以保留研究文档和 query-key 预留 tag，但进入实现前必须作为独立专题重新设计，并复用 M3 之后稳定下来的模块和泛型边界。

## 非目标

- 不在 M2.5 重新设计标准库。
- 不把完整 borrow checking、resource/drop、trait、closure、iterator、async 或 macro 提前塞进 query 重构。
- 不把 M3 模块系统或 M3 泛型闭环伪装成 M2.5 余项。
- 不为了“看起来增量”保留第二套并行前端。
- 不把 LSP protocol 适配层当成 query / syntax 架构本身。

## 验收

1. 诊断元数据不再由 message 文本派生；改文案不改 category/code。
2. 每个 query 都有明确输入、输出、stable key、依赖和失效边界。
3. 语法树能保留 trivia，并能在局部编辑后只重算受影响 parse / sema 区域。
4. CLI、JSON、LSP 复用同一 diagnostics 数据源，且输出顺序确定。
5. 现有 stress / coverage / regression 质量门继续保留，不以“架构迁移”为由放松。
6. M3 入口文档明确：当前后续只推进模块系统和泛型闭环；RAII、trait、closure、iterator 等不再挂在 M2.5 验收尾巴上。
