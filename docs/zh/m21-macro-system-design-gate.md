# Aurex M21a Macro System Design Gate

日期：2026-06-11
阶段：M21a Macro System Design Gate
状态：设计 gate / query facts / validation baseline

## 目标

M21a 的目标不是直接实现完整宏系统，而是先把 Aurex 宏系统的边界固定成可验证事实，避免以后把宏做成难调试、
难增量、难做 IDE 支持、容易绕过语义检查的黑盒。

本阶段新增 `m21a_macro_design_gate_baseline()`，用 query 层结构化 facts 固定以下设计结论：

- 宏输入输出必须走 token tree / attribute surface，不做文本替换。
- 宏展开必须有 hygiene mark、origin id、declared generated names 和 expansion source map。
- 宏展开必须是 query-backed incremental expansion，而不是 parser 里的不可缓存副作用。
- 近期主线只打开 attached item codegen / derive codegen 的地基，不直接打开 typed expression macro。
- external procedural macro 必须等待 sandbox / manifest / permission / implementation fingerprint 设计完成。
- 本阶段不实现标准库、不实现 runtime helper、不执行外部过程宏、不生成用户可用宏展开代码。

## 当前仓库基础

现有仓库已经有 M20e 的内建 derive 属性：

```aurex
#[derive(Copy, Eq, Hash)]
struct Key {
    value: i32;
}
```

但它不是完整宏系统。当前 parser 只接受 `derive` item attribute，AST 只保存 `DeriveDecl`，sema 只把
`Copy`、`Eq`、`Hash` 降成 checked capability facts。它不会生成函数、impl、member、accessor，也没有
用户自定义 derive 或过程宏。

M21a 选择在此基础上推进，但不直接把 `DeriveDecl` 扩成真实实现。本阶段只新增 query design gate，用于固定
后续 M21b/M21c 的可实施路径。

`MacroDesignCapability` 固定七类能力：

| `query_backed_incremental_expansion` | 当前路线，下一步实现 |

打开高风险宏能力。

## 非目标

M21a 明确不实现：

- 标准库。
- runtime helper。
- 文本替换宏。
- 用户可执行 external procedural macro。
- typed expression macro。
- macro-generated user code lowering。
- 宏生成 `module`、`import`、`pub use` 或修改 operator precedence。
- 宏绕过 sema、borrow checking、visibility、trait solver 或 unsafe gate。

## 为什么不做文本宏

文本宏会带来括号错配、优先级错配、分号吞噬、副作用重复和名字捕获等问题。Aurex 已经有稳定 token、lossless
syntax、AST、query 和 checked facts；继续走文本替换会绕开这些地基，并使 diagnostics 和 IDE 无法可靠解释宏展开。

因此 M21a 把 `do_not_support_textual_macros` 写进每个 candidate 的 non-goal。后续任何把 textual macro 打开的变更
都会让 `is_valid_m21a_macro_design_gate()` 失败。

## Hygiene 模型

后续实现应采用 origin mark hygiene：

- 用户传入 fragment 使用 call-site mark。
- 宏定义处 helper 使用 definition-site mark 或显式绝对路径。
- 宏内部临时变量使用 fresh mark，不能被用户局部捕获。
- 宏导出的 item/member 名必须通过 declared names 暴露。
- lookup 不应扫描未声明的临时 generated names。

这套模型比“不卫生的过程宏默认像写在调用点”更适合 Aurex，因为 Aurex 已经有 visibility、borrow、trait、
generic instance 和 query cache 的严格边界。

## Expansion Source Map 与调试

宏系统必须从第一版就支持展开追踪：

```text
aurexc --dump-macro-expansions file.ax
aurexc --emit-expanded file.ax -o file.expanded.ax
aurexc --trace-macro Builder file.ax
```

诊断必须能显示 generated span 与原始 attachment span：

```text
error: unknown field `threds`
  --> <generated Builder for Config>:12:18
note: generated by macro `Builder`
  --> config.ax:3:1
note: while expanding #[builder] attached to `Config`
  --> config.ax:4:8
```

M21a 不实现这些 CLI，但用 `expansion_source_map_and_debug_trace` candidate 固定后续必须产出的 facts：

- `macro_expansion_origin_fact`
- `macro_expansion_trace_fact`
- `macro_generated_source_map_fact`

## Query 与增量

宏展开必须成为 query，而不是 parser 内部副作用。后续至少需要：

- `macro_manifest_query_fact`
- `macro_expand_item_query_fact`
- `macro_expansion_fingerprint_fact`

展开 fingerprint 至少应包含：

- macro schema version。
- macro definition stable id。
- macro implementation fingerprint。
- macro role。
- declared generated names policy。
- input token tree fingerprint。
- parent hygiene mark。
- target / feature config。

这样可以让未变化的宏展开复用，并让 lookup 在 attached macro 声明了 generated names 时避免展开无关宏。

## Attached Item Codegen 主线

M21a 选择 attached item codegen 作为第一条真实宏主线，原因是：

- M20e 的 `#[derive(...)]` 已经证明 item attribute 是当前用户痛点入口。
- 代码生成主要服务 item/member/impl 生成，必须在 declaration phase 前完成。
- attached macro 可以要求显式 role 和 declared names，便于 query/cache/IDE。
- 不依赖标准库即可先验证 compiler-only generated item 管线。

后续可能支持：

```aurex
#[derive(DefaultShape)]
struct Point {
    x: i32;
    y: i32;
}

#[builder]
struct Config {
    pub threads: i32 = 4;
    pub verbose: bool = false;
}
```

这些示例只是后续方向；M21a 本身不生成这些代码。

## External Procedural Macro 边界

外部过程宏必须后移，因为它需要：

- 进程隔离。
- 禁止默认网络访问。
- 禁止默认环境变量访问。
- 文件读取必须走 manifest declared inputs。
- 超时、内存上限和展开深度上限。
- implementation fingerprint。
- deterministic output policy。

M21a 将 `external_procedural_macro_sandbox` 标为 `blocked_by_dependency`，并要求
`requires_process_sandbox`。任何把 external process 标为不需要 sandbox 的变更都会被 gate validation 拒绝。
