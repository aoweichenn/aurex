# Aurex M3.1 泛型闭环执行计划

## 文档定位

本文是 M3.1 后续开发的执行入口。之后推进泛型时，默认先读本文和当前 work package，而不是每次重新梳理
M2、M2.5、M3.0 的全部历史。

## 当前基线

- 分支：`m3.1`
- 起点提交：`af4ed87 Start M3.1 generic ABI stabilization`
- M3.0 状态：module system closure 已完成，M3.1 是当前最高优先级。
- 已完成的 M3.1 首项工作：generic struct / enum / function 的 ABI suffix 已从 session-only
  `TypeHandle.value` 切到 `GenericInstanceKey` / stable fingerprint。
- 当前最大风险：method-local generics 已进入正式实现，后续风险集中在质量门禁、压力规模和后续 query authority
  继续下沉时不能破坏现有 ABI / lowering 闭环；generic query authority、generic body/lowering 和 builtin
  operand closure 已形成当前权威边界。

## 总目标

M3.1 的目标不是扩张语言面，而是把现有可用泛型推进为现代编译器式的稳定系统：

1. 泛型实例身份稳定：所有实例身份来自 `GenericInstanceKey` / canonical type identity。
2. 泛型 ABI 稳定：实例符号名不能依赖本次编译 session 的 handle 分配顺序。
3. 泛型 query 权威化：`GenericTemplateSignature`、`GenericInstanceSignature`、`GenericInstanceBody`
   成为检查、复用、增量失效和后续 IDE/LSP 的事实源。
4. 泛型 body/lowering 闭环：retained typed body、generic side table、IR lowering、LLVM lowering 和
   native execution 消费同一份 instance body 视图。
5. 泛型函数体内的 builtin type operand 完整通过 sema、IR 和 LLVM lowering：`sizeof[T]` /
   `alignof[T]`、`ptrat[*T]`、`ptrcast[*T]`、`bitcast[*T]` 等必须消费同一份 checked side table。
6. method-local generics 从 unsupported 进入实现，但必须在身份、ABI 和 query 边界稳定后进行。
7. 全量测试、覆盖率、generic stress、query pruning 和 native execution 不回退。

## 非目标

M3.1 明确不做这些内容：

- 用户 trait / protocol、associated type、associated const、trait object 或 dynamic dispatch。
- const generics。
- RAII、`Drop`、`Copy`、move-only struct、borrow checker、lifetime 或 automatic cleanup。
- closure capture、`Fn` / `FnMut` / `FnOnce`、generator、async 或 iterator protocol。
- 标准库重建、package manager、版本求解或依赖 resolver。
- 新的泛型语法体系；继续使用 `Name[T]`、`fn id[T](value: T)` 和 `where T: Sized + Eq`。

## 全局不变量

后续任何 work package 都必须遵守这些约束：

- `TypeHandle.value` 是 session-local fast handle，只能用于当前会话内部索引，不能进入稳定 ABI、stable id、
  incremental key、query key 或跨 session 可观察行为。
- display string、diagnostic text、checked dump text 和 C ABI name 是输出结果，不能反向作为语义身份。
- 泛型实例 identity 必须来自 `GenericInstanceKey`，其类型实参必须来自 `CanonicalTypeKey`，约束环境必须来自稳定
  `ParamEnvKey`。
- `GenericInstanceIdentity` 是 sema 侧连接 stable key、fingerprint text、stable definition id 和 ABI suffix 的
  统一桥梁；不允许各路径重复自造兼容但不相同的 identity。
- `GenericTemplateSignature`、`GenericInstanceSignature`、`GenericInstanceBody` 最终必须成为 query provider
  authority。eager sema 可以 materialize 或 consume query result，但不能长期作为唯一事实源。
- checked metadata 必须保存足够结构化身份，让 dump、lookup、incremental cache、IR lowering 和 tooling 不需要
  重新猜测泛型实例。
- IR lowering 和 LLVM lowering 只能消费 checked sema / generic instance body / side table 结果，不能回读 raw
  AST 假设去补泛型语义。
- method-local generic 参数必须与 owner generic 参数区分身份、scope、lookup、inference、ABI 和 query key。
- resource capability `Copy` / `Drop`、用户 trait 和 const generic 不得借 M3.1 work package 偷跑进入实现。

## 每次执行的最小上下文规则

后续每次处理一个 work package 时，默认只读：

1. 本文的总目标、全局不变量和对应 work package。
2. work package 标出的必读源码入口。
3. 直接 caller / callee 和对应测试。
4. 当前 diff 涉及的文档段落。

只有出现以下情况，才扩大阅读范围：

- 修改 query key、stable identity、ABI naming 或 incremental cache：必须跨 sema、query、checked metadata 和
  cache subject 读一圈。
- 修改 generic body 或 side table：必须同时读 sema retained body 和 IR lowering。
- 修改 method-local generics：必须同时读 lookup、call analysis、generic analyzer、checked metadata、query、
  diagnostics 和 integration tests。
- 修改 generic builtin type operand：必须同时读 sema type ABI layout / builtin checker、generic side
  table、IR value kind、LLVM lowering 和 native sample。

## WP-1B：Generic Instance Identity Propagation

### 目标

让 generic struct、enum、type alias、function、method 的 checked metadata 都携带统一实例身份，避免各阶段各自重建。

### 当前状态

已完成：

- `FunctionSignature` 保存 `GenericInstanceKey`，generic function / owner-generic method 在 retained side table
  和非 retained side table 路径都会把同一实例身份写入 checked function signature。
- `EnumCaseInfo` 保存 `GenericInstanceKey`；同时新增轻量 `GenericEnumInstanceInfo`，用于表达 generic enum
  实例的 query-facing 身份，不需要为普通 enum 引入完整 `EnumInfo`。
- generic type alias 实例不再只是 session-local resolved cache；新增 `GenericTypeAliasInstanceInfo`，保存
  `GenericInstanceKey`、resolved target type、stable id 和对 target type 敏感的 instance signature
  incremental key。语义上 type alias 仍然是透明别名，不产生独立运行时 ABI 类型。
- `CheckedModule` clone / copy / move 会保留 function / enum case / generic enum instance / generic type alias
  instance 的 identity。
- incremental cache 的 generic instance signature subject 从结构化 checked metadata 收集，并按
  `GenericInstanceKey` 去重；无效 key / incremental key 不再进入 subject 列表。
- 白盒测试覆盖 function signature identity、discard-side-table 路径、generic enum case identity、generic
  type alias instance identity、checked module copy/move 保真和 driver generic instance cache rows。

### 需要回答的问题

- `StructInfo::generic_instance_key` 目前已存在，enum/type alias/function/method 是否有同等级字段或可复用字段。
- `FunctionSignature` 是否需要保存 `GenericInstanceKey`，还是只保存 stable id / incremental key / generic args 不够。
- enum metadata 当前主要落在 `TypeInfo` 和 `EnumCaseInfo`，是否需要专门 `EnumInfo` 风格结构，或在现有结构上记录
  generic instance key。
- type alias 实例是否应该有 query-visible identity，还是只作为 resolved target type 的语义别名。
- checked dump / IDE definition / lowerer 是否应该直接展示或消费 generic instance key。

### 必读入口

- `include/aurex/sema/checked_module.hpp`
- `src/sema/checked_module.cpp`
- `src/sema/internal/sema_generic_analyzer.cpp`

### 允许修改范围

## WP-2：Generic Query Authority

### 目标

把现有 query 记录从“可以写出”升级为“事实源边界”：

- `GenericTemplateSignature` 权威化模板参数、约束、可见性、module/package/part identity。
- `GenericInstanceSignature` 权威化实例返回类型、参数类型、unsafe/variadic/method 信息、capability 环境和 signature fingerprint。
- `GenericInstanceBody` 权威化 retained typed body、side table layout、body fingerprint 和 lowering 所需输入。

### 必读入口

- `include/aurex/query/generic_template_signature_query.hpp`
- `include/aurex/query/generic_instance_signature_query.hpp`
- `include/aurex/query/generic_instance_body_query.hpp`
- `src/query/query_provider_set.cpp`
- `src/query/query_executor.cpp`
- `src/query/query_result.cpp`
- `src/driver/incremental_cache/subjects/semantic.cpp`
- `src/driver/incremental_cache/subjects/ordering.cpp`
- `src/sema/internal/sema_generic_analyzer.cpp`
- `tests/gtest/query/query_key_tests.cpp`

### 设计要求

- provider input/output 必须只包含稳定身份和结构化结果。
- query result fingerprint 必须对语义相关变化敏感，对声明顺序或 session handle 这类非语义变化不敏感。
- eager sema 可以作为 provider 的实现来源之一，但调用边界必须显式。
- provider-skip replay、query pruning 和 incremental cache fast path 必须能解释 generic instance signature/body。

### 完成标准

- query gtest 覆盖 provider input/output valid/invalid、record fingerprint、edge verifier、replay/pruning。
- sema 泛型实例化路径明确消费或写入 provider authority。
- incremental cache subjects 不再需要从 display/c_name 推断泛型事实。

### 当前状态

已完成：

- query 层新增 `GenericTemplateSignatureAuthority`、`GenericInstanceSignatureAuthority` 和
  `GenericInstanceBodyAuthority`，provider input 不再只携带裸 `IncrementalKey` / body fingerprint。
- `GenericTemplateSignatureAuthority` 显式记录 signature incremental key、`ModulePartKey`、namespace、
  visibility rank、template param count 和 constraint count；`GenericTemplateSignatureInfo` 也补齐
  `constraint_count`，让 checked metadata 能稳定喂给 provider。
- `GenericInstanceSignatureAuthority` 显式记录 instance kind、visibility、type/const argument 数量、
  param-env predicate 数量、value parameter 数量、generic argument 数量、return/receiver/unsafe/variadic/
  definition flags；provider result 由这些结构化字段生成，不再等同于裸 incremental key。
- 泛型 struct / enum 实例的 upstream signature fingerprint 已补强为解析后形状敏感：struct 混入字段 stable
  key、字段可见性和规范字段类型；enum 混入 underlying/payload layout、case stable key、discriminant 和
  payload 规范类型，避免同一个 `GenericInstanceKey` 下字段或 payload 变化被 cache 误判为未变。
- `GenericInstanceBodyAuthority` 显式记录 checked-body fingerprint、instance signature result、generic
  side-table span/count、sparse fallback count 和 retained/local-dense/sparse flags；body query result 与
  signature result、body fingerprint 和 side-table layout 同时相关。
- `src/driver/incremental_cache/subjects/semantic.cpp` 从 checked metadata 和 module records 构造 authority；
  `subjects/ordering.cpp` 使用同一组 query helper 复算 record result，避免 provider、provider-skip replay 和
  pruning 之间出现第二套 fingerprint 规则。
- query tests 覆盖 authority valid/invalid、template constraint 变化、instance method/receiver 变化、body
  side-table count 变化对 result fingerprint 的影响，以及 generic signature/body 的 dependency、fallback 和
  cycle 行为。
- sema whitebox tests 覆盖 generic struct field shape 和 generic enum payload shape 变化：同一个
  `GenericInstanceKey` 下 signature fingerprint 必须变化。
- driver/incremental-cache 测试覆盖 generic instance signature/body rows、query pruning reuse、malformed
  graph/identity 修复路径，证明 cache fast path 能解释新的 generic query record。

## WP-4：Generic Builtin Operand Closure

### 语义目标

在 generic function body 中，builtin 表达式必须沿用 sema 权威边界：

- `sizeof[T]` / `alignof[T]` 在 sema 阶段确认 `T` 满足 `Sized` 且是可求 ABI layout 的
  storage type。
- `ptrat[*T]`、`ptrcast[*T]`、`bitcast[*T]` 等携带 type operand 的 unsafe builtin
  必须在 generic instance body 分析后把替换后的 concrete type 记录到 retained generic side table。
- `ptraddr`、`sliceptr`、`slicelen`、`strptr`、`strblen`、`strvalid`、`strfromutf8`、`strraw`
  这类 value-only builtin 必须通过 retained expression type side table 被 IR lowering 消费。
- 对未满足条件的 generic parameter 或非法 type operand 给结构化诊断。
- IR/LLVM/native 输出和非泛型路径一致；IR lowering 不允许重新解析 raw AST 来弥补 sema 信息缺口。

### 当前状态

已完成：

- generic function / method instance body 的 `GenericAnalysisScope` 在 retained 和 non-retained side
  table 路径都启用 syntax type handle 缓存。
- `sizeof[T]` / `alignof[T]` lowering 在实例化后读取 sema 替换出的 concrete target type，例如
  `sizeof i32` / `alignof i32`。
- `ptrat[*const T]`、`ptrcast[*const T]`、`bitcast[*const T]` 在泛型函数体内通过同一份
  `syntax_type_handles` 闭环。
- `sliceptr`、`slicelen` 和 string UTF-8 / raw builtins 在泛型函数体内通过 retained expression
  side table 覆盖。
- `cast[T](value)` 仍受既有 scalar-cast 规则约束；当前没有 `Scalar` / `Cast` capability，因此任意
  `T` 作为顶层 cast target 不在 M3.1 扩张范围内。

### 必读入口

- `src/sema/internal/sema_builtin_expression_analyzer.cpp`
- `src/sema/internal/sema_type_services.cpp`
- `src/sema/internal/sema_generic_analyzer.cpp`
- `src/ir/lower_ast_expr.cpp`
- `src/ir/ir_dump.cpp`
- `src/backend/llvm/llvm_backend_value.cpp`
- `tests/samples/positive/core/builtins.ax`
- `tests/samples/positive/core/layout_primitive_matrix.ax`
- `tests/samples/negative/expressions/expression_diagnostics_matrix.ax`

### 已定决策

- generic param 不默认满足 `Sized`；`sizeof[T]` / `alignof[T]` 必须要求 `where T: Sized`。
- opaque、void、unsized-like 类型继续沿用 sema builtin checker 的结构化诊断。
- builtin type operand 只属于 body 语义，不进入 `GenericInstanceSignature`；body fingerprint / retained
  side table 是 lowering 权威。

### 完成标准

- positive sample 覆盖 generic function 内 `sizeof[T]` / `alignof[T]`、unsafe pointer builtin、
  slice builtin、string builtin 和普通 `cast`。
- negative sample 覆盖缺 `Sized` capability 和非法 generic `ptrat[T]` type operand。
- IR dump 断言覆盖 concrete layout target、pointer cast/address、slice 和 string builtin。
- native execution 覆盖至少一个完整泛型 builtin 实例。

## WP-5：Method-local Generics

### 目标

把当前 unsupported 的 method-local generic parameters 变成正式语言能力。

### 当前实现

已完成：

- parser 继续保持 `impl[T] Owner[T]` 的 owner 参数排在 method-local 参数之前；method item 的
  `generic_params` 顺序就是稳定实例参数顺序。
- sema 注册阶段不再拒绝 method-local generic parameters；owner 参数通过 `impl_type_pattern` 中出现的
  `GenericParamIdentity` 识别，剩余参数视为 method-local。
- `value.method[T](arg)` / `Type.method[T](arg)` 的显式 type args 只绑定 method-local 参数；owner 参数由
  receiver / associated owner type 推导。owner-only generic method 显式传 type args 会按 expected `0`
  报 arity 错误。
- 普通方法调用在存在 method-local 参数时从调用参数反推局部泛型；owner-only generic method 保留原 owner-type
  推导路径，避免把 receiver/参数 coercion 误当成泛型 unification 失败。
- generic method instance 使用 `DefNamespace::member` 的 `GenericInstanceKey`，identity args 以
  `owner_type + ordered generic args` 组成；semantic key 和 C symbol name 使用生成 key / ABI suffix，避免同一
  owner 上 `method[i32]` 与 `method[bool]` 碰撞。
- instantiated generic method 不再写入普通 method name lookup；调用解析始终回到 template bucket，并按本次
  type args 找到或复用对应实例，避免后续调用误命中上一组 method-local type args。
- IR lowering 支持 `generic_apply(field(...))` 形式的显式 generic method callee，并正确补 receiver 参数。
- diagnostics 已覆盖 method-local arity、无法推断、约束不满足和非泛型方法误传显式 type args。

### 设计必须先定

- owner generic 参数和 method-local generic 参数的 scope 顺序。
- explicit type args 语法是否沿用 `value.method[T](arg)` 和 `Type.method[T](...)`。
- method lookup 阶段如何同时处理 owner type、method name、method type args 和 inferred args。
- method-local generic 的 `GenericParamIdentity` 如何编码 owner def、method def、param index 和 source range。
- ABI suffix 如何组合 owner instance key 与 method-local args。
- `GenericInstanceKey` 的 namespace 用 `DefNamespace::member`，并且 type args 顺序必须稳定。
- diagnostics 如何区分 owner generic arg 不满足约束、method-local arg 不满足约束、无法推断 method-local arg。
