# Aurex M3.2 Query-backed Sema 设计与执行计划

## 文档定位

本文是 M3.2 的设计入口。M3.0 已完成模块系统收口，M3.1 已完成泛型闭环；M3.2 不急着扩张
trait、RAII、const generic 或标准库，而是把语义分析继续推进到现代编译器式 query-backed 架构。

M3.2 的关键词是：稳定身份、query authority、可复用 checked facts、明确失效边界、IDE/LSP 可消费。

## 当前基线

- 分支：`m3.2`
- 父基线：`m3` / `m3.1` 的 `59a2ddf Complete M3.1 generic closure audit`
- 已稳定的基础：
  - R5 driver/session/pipeline/stage metadata。
  - M3.0 `ModuleKey` / `ModulePartKey` / module graph/export 边界。
  - M3.1 `GenericInstanceKey` / `GenericInstanceIdentity` / generic signature/body authority。
  - `aurex_tooling::IdeSnapshot`、lossless syntax、structured diagnostics 和 query graph persistence。

## 总目标

M3.2 的目标不是新增语言语法，而是把 sema 从“单次 eager analyzer 产生 checked module”推进为“query-backed
semantic authority”：

1. `ItemSignature`、`GenericTemplateSignature`、`GenericInstanceSignature`、`BodySyntax`、`TypeCheckBody`、
   `GenericInstanceBody` 等结果必须有统一的 query authority 入口。
2. eager sema 可以 materialize query result，但不能继续作为唯一事实源。
3. checked metadata 必须分清 durable facts、session-local caches 和 lowering-only side tables。
4. incremental cache / query pruning / provider-skip replay 能解释 sema 结果复用，而不是只复用文件级结果。
5. IDE/tooling 能消费 query-backed semantic facts，后续 LSP 不需要绕过 parser/sema/query。
6. 当前 M3.1 泛型 identity、ABI、IR/LLVM/native 行为不能回退。

## 非目标

M3.2 明确不做这些内容：

- 用户 trait / protocol、associated type、associated const、trait object 或 dynamic dispatch。
- const generic。
- RAII、`Drop`、`Copy`、move-only struct、borrow checker、lifetime 或自动资源释放。
- closure、generator、async、iterator protocol。
- 标准库重建、package manager、依赖 resolver。
- 大规模 parser/IR/backend 重写。M3.2 只在 sema/query/tooling 边界上动刀。

## 现代编译器参考取舍

M3.2 采用这些现代编译器经验，但按 Aurex 当前规模裁剪：

- **rustc query model**：采用 stable key、provider、dependency edge、red-green reuse 的思想；暂不引入完整
  demand-driven HIR/MIR 分层，也不在本阶段重建 trait solver。
- **Salsa-style incremental**：采用输入 query、派生 query、结果 fingerprint 和依赖图；暂不引入并发 query
  executor，先保证单线程 determinism 和可验证 invalidation。
- **Clang Sema / ASTContext**：保留高性能 arena/interner 和 checked facts 的集中所有权；避免把所有行为都拆成
  过细虚接口。Aurex 需要清晰 service 边界，但不复制 Clang 的全局可变 ASTContext 风格。
- **LLVM pass/analysis manager**：沿用 analysis invalidation 的思路：每个 sema query provider 必须声明输入
  authority、输出事实和失效条件。

## 全局不变量

- stable query key 只能来自 `ModuleKey`、`ModulePartKey`、`StableDefId`、`StableMemberKey`、
  `GenericInstanceKey`、`BodyKey` 和 canonical type identity。
- `TypeHandle.value`、`IdentId.value`、vector index、display name、diagnostic text、checked dump、c_name 都不能成为
  跨 session query identity。
- `CheckedModule` 继续是当前编译产物的聚合视图，但 M3.2 后必须能指出每类 checked fact 来自哪个 query
  authority。
- sema provider 不得回读 raw AST 猜测已经被上游 query 明确化的事实；需要 AST 时必须通过 `ModulePartKey` /
  item/body source range 访问。
- IR lowering 仍只消费 checked sema / retained body / side table，不直接参与 sema query 计算。
- diagnostics 的 category/code/message 分层保持不变；query-backed diagnostics 必须输出同一结构化协议。
- 所有 work package 都必须保留 M3.1 generic stress、query pruning、coverage 和 native execution baseline。

## WP-1：Sema Query Authority Inventory

### 目标

先不改生产行为，做结构性盘点并补最小元数据：

- 列出 `CheckedModule` 中 durable facts、session-local caches、lowering-only side tables。
- 为每类 durable fact 指定 query owner：item signature、generic template signature、generic instance
  signature、body syntax、type-check body、generic instance body、diagnostics。
- 确认每个 provider 的 stable key、input authority、output fingerprint、dependency edges 和 invalidation 条件。

### 完成标准

- 中英文文档有 authority matrix。
- 现有 query record / checked metadata 的身份边界没有空白项。
- `git diff --check`、format、ctest 通过。若只改文档，至少 `git diff --check` 通过。

## WP-2：Item And Body Provider Boundary

### 目标

把非泛型 item 和 function body 的 query authority 补到 M3.1 泛型 query 的同等级别：

- `ItemSignatureAuthority` 显式记录 visibility、module part、definition kind、param/field/payload shape。
- `BodySyntaxAuthority` 显式记录 body source range、syntax fingerprint、owner def key。
- `TypeCheckBodyAuthority` 显式记录 signature result、diagnostics result、checked body fingerprint 和 side-table
  layout summary。

### 禁止事项

- 不用 display string 或 c_name 反推 body identity。
- 不让 lowerer 参与 type-check query 计算。
- 不把 diagnostics message 文本作为 result fingerprint 的唯一输入。

## WP-3：Checked Fact Materialization

### 目标

让 eager sema 继续跑完整编译，但把 durable checked facts 标注为由 query authority materialize：

- `CheckedModule` 保留聚合结果，新增或复用来源 metadata。
- incremental cache subject 直接从 authority result 收集，而不是重新拼一套近似 fingerprint。
- provider-skip replay 能解释 item/body/generic 三类语义结果。

## WP-4：Sema Service Boundary Split

### 目标

在不重写算法的前提下，把 `SemanticAnalyzerCore` 的高耦合职责拆成可查询服务：

- `LookupService`：module/import/re-export/type/value/method lookup。
- `TypeService`：canonical type、storage/layout eligibility、coercion。
- `GenericService`：template/instance identity、constraint/capability、instance body。
- `BodyCheckService`：函数体、表达式、statement、pattern checking。

拆分原则：服务接收显式 context 和 authority inputs，不隐式拥有整个 analyzer 状态。

## WP-5：Tooling Semantic Query Surface

### 目标

让 `IdeSnapshot` 消费 M3.2 的 query-backed facts：

- hover / definition / references 可指向 stable def/member/generic instance key。
- diagnostics owner stage 和 query dependency edge 可一起返回。
- 为后续 LSP adapter 预留 semantic fact pagination / filtering，不在本阶段实现完整 LSP。

## WP-6：Incremental Reuse And Quality Gates

每个代码改动 work package 默认运行：

```sh
tools/format_check.py
git diff --check
cmake --build --preset full-llvm -j 8
ctest --test-dir build/full-llvm --output-on-failure
tools/check_coverage.sh
tools/query_pruning_gate.py --skip-build
make perf-stress-threshold
```

涉及 query graph 或 invalidation 时追加：

```sh
tools/query_graph_fuzz.py --skip-build --rounds 4 --functions 8 --function-step 8
```

涉及泛型 provider 或 side table 时追加：

```sh
AUREX_STRESS_BUILD_DIR=build/full-llvm tools/generic_stress.py --skip-build --counts 200,500,1000,2000
```

## 后续执行协议

后续用户说“继续 M3.2 WP-1”时，默认流程：

1. 读取本文的目标、不变量和对应 WP。
2. 核对 `git status --short --branch`，确认在 `m3.2`。
3. 只读 WP 标出的源码入口和直接调用链。
4. 实现或补文档。
5. 运行对应质量门。
6. 更新本文状态。
7. 提交并推送。
