# Aurex M3.3 Tooling Session 与 Incremental Sema 计划

## 文档定位

M3.3 接在 M3.2 Query-backed Sema 收口之后。M3.2 已经把语义事实 query-backed 化，并通过
`aurex_tooling::IdeSnapshot` 暴露给工具层；M3.3 的任务是把这个 snapshot 推进成长期运行的 tooling /
LSP 可消费架构。

M3.3 不是语言特性阶段。它要解决的是：从“对单个内存 buffer 构建一次 IDE snapshot”，推进到“从带版本的
workspace state 服务编辑器请求，并复用 query-backed semantic facts”。

## 当前基线

- 分支：`m3.3`
- 父基线：`m3` / `m3.2` 的
  `7f35b52 Complete M3.2 sema service and tooling query surface`
- 已稳定基础：
  - R5 driver/session/pipeline/stage metadata。
  - M3.0 module/package/part identity。
  - M3.1 generic identity、ABI suffix 和 generic query authority。
  - M3.2 item/body/generic sema query authority，以及 `IdeSnapshot::query`
    semantic facts。

## 外部设计参考

M3.3 采用这些成熟 IDE/compiler 经验，但按 Aurex 当前规模裁剪：

- [Language Server Protocol 3.17](https://microsoft.github.io/language-server-protocol/specifications/lsp/3.17/specification/)：
  采用标准 initialize/shutdown lifecycle、text-document sync、request/notification、cancellation、
  diagnostics、hover、definition、references 和 semantic-token capability 形状；但不让协议 DTO 变成编译器内部结构。
- [clangd design](https://clangd.llvm.org/design/)：把打开文件和未保存改动作为一等状态，持续产出
  diagnostics，从每个文件的 parsed/checked view 响应编辑器请求，跨文件能力需要 index。Aurex 先做单线程、
  进程内版本，不立即复制 clangd 的 `TUScheduler` 和后台索引规模。
- [rust-analyzer architecture](https://rust-analyzer.github.io/book/contributing/architecture.html)：
  client 提供的文件和 project structure 是 input state，semantic state 是派生结果；syntax 独立于 LSP 和语义状态；
  IDE-facing API 用可序列化形状的 value types；LSP serialization 只留在 adapter 层。

## 总目标

1. 在 `IdeSnapshot` 上方增加长期 tooling session 边界。
2. 引入最小 LSP adapter shell，只消费 tooling API 输出，不读取 parser / sema / query / driver internals。
3. 让 open-buffer edits 带版本、可复现、可诊断。
4. 复用 M3.2 semantic query records 和 dependency edges，解释一次编辑失效了什么。
5. 为跨文件 IDE 能力打基础，但第一阶段不做完整后台索引。
6. 所有 compile、query、coverage、stress 门禁保持 green。

## 非目标

- 不做 VS Code extension、编辑器插件安装或平台发布。
- 第一阶段不做完整 completion、rename、code action、formatting、inlay hints 或 semantic tokens。
- 第一阶段不做多线程 scheduler。
- 不做 remote index、LSIF index 或持久 workspace database。
- 不做用户 trait、resource semantics、RAII、closure、const generic、package manager 或标准库重建。
- 不绕过 `IdeSnapshot`、`PipelineStage` 或 query-backed sema facts。

## 核心架构

```text
LSP JSON-RPC transport
  |
  +-- LspServer / LspDispatcher
        |
        +-- ToolingSession
              |
              +-- VersionedDocumentStore
              +-- ToolingProjectConfig
              +-- SnapshotCache
              +-- SemanticFactIndex
              +-- IncrementalReusePlanner
                    |
                    +-- aurex_tooling::build_ide_snapshot(...)
                    +-- M3.2 QueryContext records / dependency edges
```

核心规则：LSP 只是 adapter。编译器侧边界是 `ToolingSession`，不是 LSP protocol struct。
`ToolingSession` 持有 open document state、project config、snapshot cache 和 query reuse summary，并返回
IDE-native value types；LSP adapter 只负责把这些值转换成协议 JSON。

## 全局不变量

- 一次 tooling request 必须读取同一个 immutable snapshot version。
- document change 必须先生成新的 versioned document state，再触发 diagnostics 或 semantic query。
- LSP URI、filesystem path、package identity、module identity 和 `SourceRole` 必须先归一化，再构造 query key。
- LSP DTO 不是 semantic identity。稳定身份仍只来自 `PackageKey`、`FileKey`、`ModuleKey`、
  `ModulePartKey`、`DefKey`、`MemberKey`、`BodyKey`、`GenericInstanceKey` 和 source range。
- diagnostics 必须继续使用 `IdeDiagnostic.owner_stages` 和已有 structured diagnostic category/code contract。
- syntax tree 只存 syntax 数据，不存语义状态。
- body-local edit 在 query dependency 证明可复用时，不应失效无关 item signature 或无关 function body。
- 第一阶段保持 deterministic single-threaded；cancellation 用 generation check 表达，不做抢占式线程取消。

## 当前实现状态（2026-05-29）

- 新增 `aurex_tooling::ToolingSession`，在 `IdeSnapshot` 上方持有版本化 open-document state、
  package/source-role config、URI/path 归一化、stale client version 拒绝和 snapshot cache。session API
  保持协议无关，不暴露 LSP JSON 或协议 DTO。

- 新增 `aurex_tooling::LspServer`，实现 deterministic JSON-RPC framing、initialize/shutdown/exit
  lifecycle、full-document text sync、diagnostics publishing、hover、definition、references 和 documentSymbol。
  LSP 层只调用 `ToolingSession`，不读取 parser、sema、query 或 driver internals。
- IDE projection 已补齐协议无关 diagnostics、hover、definition、references 和 document symbols。document
  symbols 优先消费 M3.2 的 checked `semantic_facts`，在没有 checked facts 时退回 top-level AST item
  projection。

  抽成协议无关的增量复用规划器。
- 新增 `src/tooling/workspace_index.cpp`，为 open snapshots 建立按 `DefKey`、`MemberKey`、`BodyKey` 和
  `GenericInstanceKey` 的小型 dynamic index，并在 document change / close 时清理旧条目。
- 新增 `tests/gtest/tooling/session_lsp_tooling_tests.cpp` 覆盖 session lifecycle、缓存复用、URI/path
  normalization、error paths、JSON-RPC framing、LSP lifecycle/text sync、diagnostics、hover、definition、

本批次的 WP-4、WP-5 和 WP-6 已收口；后续 tooling 工作仍应避免 completion、rename、formatting 或
semantic tokens，直到新的工作包明确引入它们。

## WP-1：Tooling Session And VFS Boundary

### 范围

创建协议无关 session 层：

- `ToolingDocumentId`：归一化 URI/path identity，加 package/source role。
- `ToolingDocumentVersion`：客户端版本存在时使用客户端递增版本，否则使用 session-local generation。
- `ToolingDocumentStore`：先支持 open/close/update full text；增量 text edit application 后续再优化。
- `ToolingProjectConfig`：package identity、import paths、root path 和默认 source role。
- `ToolingSession`：按 document version 构建或复用 `IdeSnapshot`。
- `ToolingSnapshotHandle`：hover/definition/references 读取的 immutable view。

### 验收

- 单测覆盖 open/change/close、stale version 拒绝、path/URI 归一化、package identity 传递和 snapshot reuse。
- 编译器/tooling session API 中不出现 LSP JSON。
- 直接调用 `build_ide_snapshot(...)` 的旧 API 行为保持不变。

## WP-2：LSP Protocol Shell

### 范围

增加最小可用协议 adapter：

- JSON-RPC header/content framing。
- `initialize`、`initialized`、`shutdown`、`exit`。
- `textDocument/didOpen`、`textDocument/didChange`、`textDocument/didClose`。
- `textDocument/hover`、`textDocument/definition`、`textDocument/references`。
- `textDocument/publishDiagnostics`。
- 静态 capability response。
- cancellation 按 request id 和 generation check 识别。

### 明确限制

- 先做 full-document sync。增量 text-edit application 等 document store 有单测后再做。
- 暂不做 completion、rename、code action、formatting 或 semantic tokens。
- 不加 editor-specific extension fields。

## WP-3：IDE Feature Projection

### 范围

把协议无关 IDE value 层补到足够转 LSP：

- Hover：stable label、detail、source range、owner stages、semantic fact key。
- Definition：`DefKey`、`MemberKey`、`GenericInstanceKey`、range、part index。
- References：先支持 local references，再支持同 open-file checked references。
- Document symbols：top-level item list 和 checked symbol kinds。
- Diagnostics：structured owner stages 和 module-part context。

该阶段 public API struct 要保持可序列化形状，但不能依赖 JSON library 或 LSP 类型。

## WP-4：Incremental Reuse Planner

### 范围

用 M3.2 query records 解释复用：

- 把 `IdeEditImpact` 转成候选 invalidation root。
- 比较 old/new semantic `QueryRecord` 和 dependency edges。
- 报告哪些 facts unchanged、recomputed 或 invalidated。
- 保留 body-local edit 行为：函数体编辑优先重算 `function_body_syntax` / `type_check_body`，不重算全局
  signature。
- 在测试和可选 debug API 中暴露 reuse summary。

第一阶段仍可调用 `build_ide_snapshot(...)`；planner 先解释复用，不要求立刻替换成完整 demand-driven executor。

## WP-5：Workspace Semantic Index

### 范围

为 open files 和 package-local facts 增加小型 dynamic index：

- 按 `DefKey` 索引 definitions。
- 按 `MemberKey` 索引 members。
- 按 `BodyKey` 索引 function bodies。
- 按 `GenericInstanceKey` 索引 generic instances。
- 合并同文件 `IdeSnapshot` facts 与 sibling module parts 的 open-buffer facts。

这不是后台索引。它只是让 LSP definition/references 后续能扩展到单 snapshot 之外，而不用扫描无关状态。

## WP-6：Quality Gates

代码改动包默认运行：

```sh
tools/format_check.py
git diff --check
cmake --build --preset full-llvm -j 8
ctest --test-dir build/full-llvm --output-on-failure
tools/check_coverage.sh
tools/query_pruning_gate.py --skip-build
tools/query_graph_fuzz.py --skip-build --rounds 4 --functions 8 --function-step 8
make perf-stress-threshold
```

如果改到 generic/type-check reuse，还要运行：

```sh
AUREX_STRESS_BUILD_DIR=build/full-llvm tools/generic_stress.py --skip-build --counts 200,500,1000,2000
```

LSP adapter 相关包还必须有 deterministic JSON-RPC fixture tests，覆盖 message framing、lifecycle、
document sync、diagnostics、hover、definition 和 references。
