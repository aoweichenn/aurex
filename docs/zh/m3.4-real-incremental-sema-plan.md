# Aurex M3.4 Real Incremental Sema Execution 计划

## 文档定位

M3.4 接在 M3.3 Tooling Session And Incremental Sema 收口并合并回 `m3` 之后。M3.3 已经建立长期
tooling boundary、LSP adapter、reuse planner 和 open-file workspace index。M3.4 的任务是把这些
reuse explanation 变成真实 incremental semantic execution。

M3.4 不是语言特性阶段。它不新增 trait、resource semantics、closure、package management、completion、
rename、semantic tokens 或 formatting。它要解决的是：编辑后编译器真正复用稳定语义事实，而不是全量重建
snapshot 后只解释“哪些 query 本可以复用”。

## 当前基线

    LSP adapter 和 workspace semantic index。

## 设计参考

M3.4 把成熟编译器经验裁剪到 Aurex 当前规模：

- rustc 风格 query reuse：input 与 derived fact 分离，result fingerprint 决定 red/green reuse，
  dependency edge 解释为什么某个 fact 必须重算。
- rust-analyzer / Salsa 风格 IDE incrementality：open files 和 project structure 是显式输入；
  semantic facts 从 immutable snapshot 派生；消费者只看稳定 value types。
- clangd 风格编辑器响应模型：一次请求读取一个一致 snapshot，拒绝 stale generation，
  diagnostics/hover/definition 操作缓存后的 parsed/checked state。
- LLVM pass-manager invalidation 纪律：一个 computation 明确说明 preserved / invalidated facts，
  而不是让所有 consumer 自己猜。

Aurex 不照搬任何一个实现。当前选择是保持单线程、确定性执行模型，先扩展既有 query/tooling interface，
再考虑 scheduler 或 background-index 复杂度。

## 总目标

1. snapshot build 能接收 previous snapshot / query context。
2. reuse decision 从解释性结果变成可执行输入。
3. 局部编辑后保留未变化 semantic facts。
4. 对 body edit、signature edit、module surface edit 和 malformed reuse input 给出明确 invalidation root。
5. workspace index 尽量按 affected fact identity 更新。
6. CLI、diagnostics JSON、profile JSON 和 emit-mode 行为保持兼容。
7. tests、coverage、query pruning、fuzz 和 stress gates 保持 green。

## 非目标

- 不新增语言语法。
- 不做 completion、rename、formatting、semantic tokens、code actions 或 inlay hints。
- 不做多线程 scheduler。
- 不做 persistent workspace database；这是 M3.6 的范围。
- 不做 package manager 或 version solver。
- 不做用户 trait、associated type、const generic、resource semantics、RAII、closure、async 或 iterator protocol。
- 不另起第二套 semantic analyzer，不绕开 R5 driver/session/query/tooling 架构。

## 核心架构

```text
ToolingSession
  |
  +-- VersionedDocumentStore
  +-- SnapshotCache
  +-- IncrementalSnapshotInput
  |     |
  |     +-- previous IdeSnapshot summary
  |     +-- previous query records
  |     +-- previous dependency edges
  |     +-- previous semantic facts
  |     +-- edit impact / invalidation roots
  |
  +-- build_ide_snapshot_incremental(...)
        |
        +-- reusable query fact set
        +-- recomputed query providers
        +-- updated IdeSnapshot
        +-- updated SemanticFactIndex
```

公开 compiler/tooling 边界仍然是 `ToolingSession` 和 `IdeSnapshot`。M3.4 可以新增 incremental input /
result summary value types，但 LSP JSON 和 protocol DTO 不能进入 sema、query 或 driver API。

## 全局不变量

- 一次 tooling request 读取同一个 immutable snapshot version。
- previous snapshot context 是优化输入，不是正确性前提；reuse input 缺失或 malformed 时必须回退重算，并记录
  diagnostics/debug facts，不能 `assert` 崩溃。
- 稳定身份只来自 `PackageKey`、`FileKey`、`ModuleKey`、`ModulePartKey`、`DefKey`、`MemberKey`、
  `BodyKey`、`GenericInstanceKey` 和 source range。
- body-local edit 可以只失效被编辑 body 的 `FunctionBodySyntax` 和 `TypeCheckBody`，不失效无关 item signature。
- signature edit 必须失效依赖该 signature result 的 body checks、generic facts 或下游 module facts。
- module-surface edit 必须失效 module graph/export facts 和相关 visibility/import facts，但不应重算无关 bodies。
- reused facts 必须携带和 recomputed facts 一致的 owner stage、source range、part index 和 semantic identity metadata。
- workspace index entry 按 document identity 和 fact identity 带版本；发布新 entry 前必须清理 stale entry。
- M3.4 保持 deterministic single-threaded。

## WP-1：Incremental Snapshot Build Input

状态：当前验收边界已完成。

### 范围

新增 protocol-neutral value types 描述上一轮语义状态：

- `IncrementalSnapshotInput`：document id/version、previous snapshot summary、previous query records、
  dependency edges、semantic facts、workspace facts 和 edit impact。
- `IncrementalSnapshotResult`：reused/recomputed/invalidated counts、fallback reason 和 updated
  query/fact summaries。
- `ToolingSession` 集成：document version 变化且 previous snapshot 存在时，把 previous context 传进
  snapshot build path。

### 验收

- 既有 `build_ide_snapshot(...)` 行为不变。已完成。
- 新增 incremental entry points 保持 protocol-neutral，不暴露 LSP DTO。已完成。
- 缺少 previous context 时回退到当前 full build 行为。已完成。
- malformed previous context 不触发 assertion。已完成。
- 单测覆盖 no-previous、matching-previous、stale-version、mismatched context 和 malformed input。已完成。

### 实现记录

- 新增 `ToolingIncrementalSnapshotInput`、`ToolingIncrementalSnapshotResult` 和
  `ToolingIncrementalSnapshotStatus`。
- `ToolingSnapshotHandle` 现在携带当前 snapshot 的 incremental build result。
- `ToolingSession::change_document(...)` 在上一版已经 materialized 时保留 previous cached snapshot，
  随后 `snapshot(...)` 把这个 context 接入下一次 build summary。
- `tooling_incremental_snapshot_result(...)` 分类 clean build、cached snapshot、accepted previous context、
  mismatched previous context、stale previous context 和 malformed previous context。
- 既有 one-shot `build_ide_snapshot(...)` / `build_ide_snapshot_into(...)` caller 行为不变。

## WP-2：Query Record Reuse Execution

### 范围

在 snapshot build 中消费 M3.2/M3.3 query records 和 dependency edges：

- 把 facts 分类为 reusable、recomputed、invalidated 或 malformed。
- identity 和 result fingerprint 匹配时，把 item signature、body syntax、type-check body、
  generic template、generic instance signature 和 generic instance body 作为 reusable facts 输入新 snapshot。
- dependencies 或 fingerprints 变化时，recomputation 仍然是权威路径。
- 为测试和 profile 记录 deterministic reuse execution summary。

### 验收

- body-local edit 复用无关 signatures 和无关 bodies。
- signature edit 失效依赖它的 body type checks。
- module export edit 失效相关 visible/export facts。
- reuse 与 recomputation 对外产生相同 diagnostics/hover/definition 结果。

## WP-3：Semantic Fact Stability

### 范围

让 semantic fact identity 足够稳定，能承接重复编辑循环：

- 对未变化 facts 保持 `DefKey`、`MemberKey`、`BodyKey`、`GenericInstanceKey`、source range、
  part index、checked flag 和 owner-stage metadata。
- 对 tooling consumer 来说，reused facts 和 recomputed facts 不应可见地区分。
- 增加 body-local、signature-local、module-surface 和 generic-instance edit fixtures。
- 当 source range 发生移动但 identity 仍稳定时，fallback 行为必须确定。

### 验收

- Tooling document symbols、hover、definition、references 和 diagnostics 读取一致的 reused facts。
- stable facts 在重复局部编辑中保持稳定。
- changed facts 报告正确 invalidation root。
- generic instance reuse 不依赖 display string 或 session-local type handle。

## WP-4：Workspace Index Incremental Update

### 范围

增量更新 `SemanticFactIndex`：

- 按 document/version 和 fact identity 移除 stale entries。
- 从更新后的 snapshot 插入 changed facts。
- 保留 sibling module parts 和 open files 的未受影响 entries。
- 为测试和 debug summary 报告 index update counts。

### 验收

- close document 会移除其 entries。
- reopen / change document 只替换该 document 的受影响 facts。
- 无关 open files 未变化时，cross-file references / definitions 继续可用。
- 不返回旧 document version 的 stale fact。

## WP-5：Performance And Robustness Gates

### 范围

为 incremental path 增加质量门：

- body-local edit stress。
- signature edit stress。
- split module-part edit stress。
- generic instance edit stress。
- malformed reuse input tests。
- query pruning 和 dependency graph checks。
- 新增文件和受影响旧文件覆盖率。

### 默认验证

```sh
ctest --test-dir cmake-build-release --output-on-failure -j4
./tools/check_coverage.sh -j4
```

query graph 行为变化时还要运行：

```sh
tools/query_pruning_gate.py --skip-build
tools/query_graph_fuzz.py --skip-build --rounds 4 --functions 8 --function-step 8
```

generic reuse 变化时还要运行：

```sh
AUREX_STRESS_BUILD_DIR=build/full-llvm tools/generic_stress.py --skip-build --counts 200,500,1000,2000
```
