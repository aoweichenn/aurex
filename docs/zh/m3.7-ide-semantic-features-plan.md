# Aurex M3.7 IDE Semantic Features 计划与收口记录

## 阶段定位

M3.7 建立在 M3.3 `ToolingSession`、M3.4 executable query reuse、M3.5 stable syntax identity 和
M3.6 project/workspace graph 之上。该阶段不新增语言语义，不改 CLI 编译结果；目标是把已有 query-backed
semantic facts 投影成现代 IDE 需要的协议无关能力，再由 LSP adapter 做薄投影。

核心原则：

- `aurex_tooling` 是 IDE 语义能力的内部 API 边界，LSP DTO 不进入 parser、sema、query 或 session。
- completion、rename、semantic tokens、inlay hints、code actions 和 workspace symbols 都从同一份
  `IdeSnapshot` / `ToolingWorkspaceSemanticIndex` 读取事实。
- stale generation 必须在 LSP 请求边界检查，避免旧 snapshot 结果覆盖新文档版本。
- quick fix 只从结构化 diagnostics 信息生成，不按普通错误文本猜测任意修复。

## 已完成工作包

### WP-1 Completion

- 新增 `IdeCompletionItem` / `ToolingCompletionItem`。
- completion context 区分 item、expression、member 和 module path。
- 当前文件补全合并 checked globals、parameters、locals、fields、enum cases、methods 和关键词。
- session 层额外合并 open workspace semantic facts，并用 stable definition/member/generic keys 去重。
- 局部符号只在其 scope range 内、且声明点之后可见。

### WP-2 Rename

- 新增 `ToolingRenamePlan`、`ToolingRenameEdit` 和 `ToolingRenameConflict`。
- rename 目标来自 `definition_at_offset` 的 symbol identity，而不是只按文本匹配。
- edit plan 复用 references 结果，包含 definition range 和当前 workspace index 可证明的跨文件 definition facts。
- 新名字会检查 Aurex identifier 形状、reserved keyword 和当前位置可见符号冲突。
- 结果是协议无关 workspace edit plan；LSP 只把它序列化为 `WorkspaceEdit`。

### WP-3 Semantic Tokens And Inlay Hints

- 新增 `IdeSemanticToken` / `ToolingSemanticToken`。
- semantic token 先按 syntax token kind 分类，再用 checked symbol facts refine identifier：
  function、method、parameter、variable、type、enum、enum member、property、keyword、number、string、
  comment、operator、punctuation 等。
- token modifiers 覆盖 declaration、definition、readonly。
- 新增 `IdeInlayHint` / `ToolingInlayHint`，当前先为缺少显式类型标注的 checked local 生成 type hint。

### WP-4 Code Actions

- `ToolingDiagnostic` 现在保留 diagnostic children。
- 新增 `ToolingCodeAction` / `ToolingCodeActionEdit`。
- 当前 quick fix 支持 lookup suggestion：`did you mean \`name\`?` 的 help diagnostic 会生成替换该 range 的
  `quickfix`。
- 该实现只消费 structured severity/range/code/message，不扫描普通 error 文本生成未证明的修复。

### WP-5 Workspace Symbols And Cross-file Surface

- 新增 `ToolingWorkspaceSymbol`。
- `workspace_symbols(query)` 会先 materialize 所有 open document snapshot，确保 workspace index 当前。
- 返回 item signature / generic template signature / member facts 的稳定符号列表，并支持 query 过滤。
- completion、references、definition 和 rename 共享 workspace semantic index 的 stable keys。

### WP-6 LSP Projection

LSP adapter 新增并声明：

- `textDocument/completion`
- `textDocument/rename`
- `textDocument/semanticTokens/full`
- `textDocument/codeAction`
- `workspace/symbol`
- `textDocument/inlayHint`

实现保持同步单线程边界，但所有文档请求都会记录请求开始时的 `ToolingDocumentVersion`，计算结束后通过
`is_generation_current(...)` 做 stale guard。

## 当前限制

- completion 的 visible module exports 目前来自 open workspace semantic facts；还不是 closed-world package index。
- rename 可以跨 open workspace 的 definition facts 规划编辑，但尚不扫描未打开文件，也不拥有完整引用数据库。
- member completion 当前基于已索引 member facts，不做完整 receiver type-directed completion。
- quick fix 当前只实现 lookup suggestion；module/source-part 修复、import 插入和类型转换修复仍留给后续阶段。
- inlay hints 当前只覆盖 local type hints；parameter hints、generic argument hints 和 implicit conversion hints 未进入本阶段。
- LSP server 仍是轻量 JSON adapter，不实现后台 scheduler、取消令牌或并发请求执行。

这些限制是阶段性边界，不是 API 形状限制；新增能力必须继续沿 `IdeSnapshot -> ToolingSession -> LSP adapter`
投影链路实现。

## 下一阶段入口

M3.8 应把 query architecture 下沉到 lowering、IR 和 backend：

- function/generic instance lowering query authority。
- type layout、enum layout、ABI symbol 和 lower-generic-IR query facts。
- IR analysis preservation 与 query invalidation 的连接。
- LLVM emission unit 与 target-independent IR unit 的显式边界。

M3.8 不应重新打开 M3.7 的 LSP DTO 边界；IDE 侧继续只消费 tooling value types。
