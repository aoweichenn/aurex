# Aurex M3.9 M3 Release Baseline 与 Authority Audit

## 状态

M3.9 是 M3 的最终 hardening 和 release baseline 阶段。它不新增语言特性，而是把 M3.0 到 M3.8 已完成的模块、泛型、query-backed sema、tooling session、incremental syntax、project graph、IDE semantic features、lowering/IR/backend reuse 统一收口成后续 M4 之前的工程基线。

M3 已完成的核心主线：

- M3.0：语言级模块系统、module parts、package visibility、selective re-export 和 module query 边界。
- M3.1：泛型身份、泛型 query authority、generic body/lowering、generic builtins 和 method-local generics。
- M3.2：query-backed sema authority 和 IDE/tooling 可消费语义事实。
- M3.3：协议无关 tooling session、LSP adapter 边界、reuse planner 和 workspace semantic index。
- M3.4：真实 incremental sema execution。
- M3.5：range edit、lossless syntax stable key、syntax reuse stats 和 AST projection。
- M3.6：project graph、workspace model 和 persistent query database。
- M3.7：completion、rename、semantic tokens、inlay hints、code actions、workspace symbols 和 LSP projection。
- M3.8：query-backed lowering、IR unit fingerprint、pass invalidation summary 和 backend emission unit boundary。

## Authority Boundary Audit

M3.9 固定下列 authority 边界。后续阶段可以扩展边界内的能力，但不能绕过边界读取下游私有状态或重新建立并行编译通路。

| 层级 | 权威输入 | 权威输出 | 禁止绕过 |
|:-----|:---------|:---------|:---------|
| Source / Lex | `SourceManager`、source file identity、open buffer text | tokens、lossless trivia-preserving source facts | 语义层直接读取文件系统或自行 lex |
| Parse / Syntax | token stream、parser recovery policy | AST、lossless tree、stable syntax keys、source ranges | sema/tooling 从 token stream 猜 AST 结构 |
| Module / Project | `ProjectModel`、`WorkspaceModel`、`ModuleRecord`、`ModulePartKey` | module graph、module exports、module package exports、project graph query facts | import lookup 绕过 loader/project graph |
| Sema | AST、module records、query authority input | `CheckedModule` durable facts、query records、semantic facts、diagnostics | tooling/LSP 直接调用 sema internals 或读取 session-local caches |
| Generic | `GenericInstanceKey`、canonical type identity、generic authority records | stable id、ABI suffix、generic signature/body facts、retained body view | 使用 display string、C name 或 session-only `TypeHandle.value` 作为跨 session identity |
| Query / Incremental Cache | stable keys、authority records、dependency edges、project inputs | query records、query edges、reuse plan、provider-skip replay | cache 只按 file timestamp 或 coarse source hash 判定 sema/lowering reuse |
| Tooling | `ToolingSession`、`IdeSnapshot`、workspace semantic index | diagnostics、hover、definition、references、completion、rename、tokens、hints/actions | LSP DTO 泄漏进 compiler core 或 LSP handler 旁路 tooling value types |
| Lowering / IR | AST + checked facts + retained side tables | verified Aurex IR、IR pass summary、IR unit fingerprints | backend 回读 AST/sema 私有表，或 lower IR rows 在真实 lowering 前写入 |
| Backend | optimized Aurex IR、target config、LLVM emitter | LLVM IR/native artifacts、emission unit profile facts | backend text 反向成为 frontend/query authority |

## Public API Baseline

M3.9 保留并冻结以下公开 API 作为后续阶段的入口：

- `aurex::driver::CompilerInvocation`、`Compiler`、`CompilationPipeline` 相关 public facade。
- `aurex::driver::pipeline_stage_record(...)`、`PipelineStageMetadata` 和 profile phase classification。
- `aurex::query::*Key`、query authority/result helpers、`QueryContext`、`QueryExecutor`、`QueryProviderSet`。
- `aurex::tooling::ToolingSession`、`IdeSnapshot`、protocol-neutral IDE value types 和 `LspServer` adapter。
- `aurex::project::ProjectModel` / `WorkspaceModel`。
- `aurex::ir::ModulePassManager`、`ModuleAnalysisManager`、IR verifier gate、IR fingerprint APIs。

后续 trait/resource/closure/iterator/package-manager 设计必须先说明它接入这些 API 的位置，而不能创建新的 driver、module loader、sema 或 backend 旁路。

## Final Gate Set

M3 release baseline 的最终验收门：

```sh
cmake --build cmake-build-release -j4
ctest --test-dir cmake-build-release --output-on-failure -j4
./tools/check_coverage.sh -j4
tools/query_pruning_gate.py --skip-build
tools/query_graph_fuzz.py --skip-build --rounds 4 --functions 8 --function-step 8
AUREX_STRESS_BUILD_DIR=cmake-build-release tools/generic_stress.py --skip-build --counts 200,500,1000,2000
tools/format_check.py $(git diff --name-only -- '*.cpp' '*.hpp') $(git ls-files --others --exclude-standard -- '*.cpp' '*.hpp')
git diff --check
```

其中 coverage 仍保持 source lines / functions / regions 全部不低于 95%。如果后续新增生产代码导致 regions 掉线，必须补正常仓库测试，不允许下调阈值或改用临时测试。

## Residual Risks

M3.9 明确记录而不在 M3 内解决的事项：

- physical green-tree graft、完整 subtree reparse 和后台多线程 scheduler 仍未进入主路径。
- workspace database 仍是当前 project/query cache 边界，不是完整 package manager 或 dependency resolver。
- 用户 trait、associated type、const generic、resource capability、RAII、closure、async/generator/iterator 和标准库重建仍是 M3 后新设计范围。
- backend emission unit 已可观测，但还没有完整多目标 codegen-unit scheduler。
- coverage regions 当前是硬阈值，后续改动需要优先维护测试余量。

## 收口结论

M3 已经把 Aurex 从 M2.5 的 frontend foundation 推进到现代编译器主线：模块/泛型身份稳定，sema/query/tooling/lowering/backend 边界可观测，IDE 和 incremental cache 都建立在同一套 authority facts 上。下一阶段不应继续在 M3 分支扩语言面，而应基于本 baseline 单独设计 M4 的 trait/resource/package/language-expressiveness 工作。
